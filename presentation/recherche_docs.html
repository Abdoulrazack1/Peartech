00 — Avant de commencer

Debounce, requête PearTechDB et modale de résultats

Quand l'utilisateur tape dans la barre de recherche, recherche.js attend qu'il s'arrête de frapper, interroge la base de données, et affiche les résultats correspondants sous le champ.

Analogie : C'est comme une bibliothécaire. Vous commencez à lui dire un titre — elle attend que vous ayez fini de parler avant d'aller chercher. Si vous changez d'avis en milieu de phrase, elle ignore l'ancienne demande et repart sur la nouvelle. Quand elle trouve des livres, elle les pose sur le comptoir. Si vous regardez ailleurs, elle les rerange.
Vocabulaire spécifique à ce fichier
Debounce
Technique qui retarde une action jusqu'à ce que l'utilisateur se soit arrêté. Sans debounce, chaque lettre frappée lancerait une recherche — soit des dizaines de requêtes inutiles pour un seul mot.
setTimeout(fn, ms)
"Dans X millisecondes, exécute cette fonction." 300ms = 0,3 secondes. Si on appelle clearTimeout avant que le délai soit écoulé, la fonction n'est jamais exécutée.
clearTimeout(id)
Annule un timer créé par setTimeout avant qu'il se déclenche. C'est le mécanisme central du debounce.
classList.add / remove
Ajoute ou retire une classe CSS sur un élément HTML. Ici, ajouter la classe active sur #search-results le rend visible (défini dans le CSS). La retirer le cache.
e.target
Dans un événement, e.target est l'élément exact qui a déclenché l'événement. Si on clique sur un texte dans un résultat, e.target = ce texte précis (pas le conteneur parent).
element.contains()
Retourne true si l'élément en paramètre est l'élément lui-même ou l'un de ses descendants dans le HTML. Utile pour savoir si un clic est "à l'intérieur" d'une zone.
Object.values()
Extrait toutes les valeurs d'un objet dans un tableau, sans avoir besoin de connaître les noms des propriétés. Object.values({a:1, b:2}) [1, 2].
window.location.href
L'URL actuelle de la page. En lui assignant une nouvelle valeur, on navigue vers cette URL — c'est l'équivalent de cliquer sur un lien.

02 — Le debounce

Attendre la fin de la frappe

Sans debounce, taper "iPhone 15" déclencherait 9 recherches : une pour "i", une pour "iP", une pour "iPh"… C'est du gaspillage. Le debounce n'en lance qu'une seule — 300ms après la dernière lettre.

Analogie : Imaginez un assistant qui attend que vous ayez fini de parler avant d'aller chercher. À chaque mot que vous prononcez, il remet le minuteur à zéro. Ce n'est qu'au bout de 300ms de silence qu'il part chercher. Si vous continuez à parler, il ne bouge pas.

Pourquoi Object.values(specs).slice(0, 2) ?

Les specs d'un produit sont un objet : {processor: "M2", ram: "8 Go", storage: "256 Go"}.

Object.values() extrait juste les valeurs : ["M2", "8 Go", "256 Go"]. On n'a pas besoin des noms de propriétés, juste des valeurs à afficher.

.slice(0, 2) prend les 2 premiers éléments : ["M2", "8 Go"]. On ne veut pas tout afficher dans une liste de résultats — juste un aperçu compact.

.join(' - ') les colle en une chaîne : "M2 - 8 Go".

Pourquoi re-attacher les événements à chaque affichage ?

Le problème du innerHTML : Quand on écrit searchResults.innerHTML = html, le navigateur détruit tous les anciens éléments et les recrée. Un écouteur de clic est placé sur le document entier. Il est déclenché par chaque clic sur la page. Pour chaque clic, on vérifie si la cible est dans la zone de recherche ou non.

Comment fonctionne .contains() ?

parent.contains(enfant) retourne true si enfant est parent lui-même, ou n'importe quel descendant de parent dans l'arbre HTML.

Exemple : Si on clique sur le texte "MacBook Air" à l'intérieur d'un résultat, e.target = le <div> du texte. searchResults.contains(ce div) retourne true car ce div est bien à l'intérieur de #search-results. Les résultats ne se ferment donc pas.

07 — Récapitulatif

Le film complet d'une recherche

Étape 1
Chargement et mise en place
L'IIFE s'exécute. Au DOMContentLoaded, les deux éléments HTML sont récupérés. searchTimeout est déclaré vide. Les 3 écouteurs sont installés. Le module est prêt.
Étape 2
L'utilisateur commence à taper
L'événement input se déclenche à chaque lettre. clearTimeout annule le timer précédent. setTimeout(fn, 300) programme une nouvelle recherche dans 300ms. Si l'utilisateur continue à taper avant 300ms, le cycle recommence.
Étape 3
300ms de silence recherche lancée

Affichage des résultats

Clic direct ou touche Entrée window.location.href = 'page_produit.html?id=42'. Le navigateur quitte la page et charge la fiche produit.

Fermeture sans sélection
Clic ailleurs sur la page active retirée, résultats cachés. Touche Échap même chose + focus retiré du champ. Champ vidé même chose dès la première frappe qui remet à vide.
00 — Vue d'ensemble

recherche.js — Recherche en temps réel avec debounce ¶

recherche.js implémente une recherche instantanée : l'utilisateur tape, les résultats apparaissent sous la barre sans rechargement de page. Le module repose sur 3 concepts fondamentaux : le debounce (ne pas chercher à chaque frappe), la délégation (confier la logique de recherche à PearTechDB), et la gestion des listeners dynamiques (résultats régénérés à chaque recherche).

⚠️
recherche.js ne contient aucun algorithme de recherche. Toute la logique de matching (chercher dans le nom, les specs, les tags) est dans PearTechDB.search() défini dans data.js. Si les résultats sont mauvais, c'est dans data.js que la correction doit se faire.

Ce que fait le fichier — 6 responsabilités

1. Écouter les frappes dans #search-input avec debounce 300ms.
2. Vérifier la disponibilité de PearTechDB avant de chercher.
3. Déléguer la logique de recherche à PearTechDB.search().
4. Générer et injecter le HTML des résultats.
5. Attacher les listeners de navigation sur les résultats injectés.
6. Fermer les résultats au clic hors zone et à la touche Échap.

Recherche en temps réel vs recherche classique

TEMPS RÉEL (actuel)

Résultats immédiats

Affiche après chaque frappe (avec debounce). Aucune action de l'utilisateur.

FORMULAIRE CLASSIQUE

Submit requis

L'utilisateur tape, appuie sur Entrée ou clique. Rechargement de page ou requête AJAX.

HYBRIDE

Les deux

Suggestions en temps réel + submit pour la page résultats complète. Pattern Google/Amazon.

💡
recherche.js implémente le pattern temps réel sans rechargement : les résultats s'affichent dans un dropdown sous le champ, et cliquer sur un résultat navigue vers la fiche produit. Il n'y a pas de page de résultats dédiée — tout se passe dans l'overlay.
01 — Récupération et protection

Les deux éléments centraux et le double guard ¶

javascript
const searchInput   = document.getElementById('search-input');   // Champ de saisie
const searchResults = document.getElementById('search-results'); // Zone d'affichage

if (!searchInput || !searchResults) {
    console.warn('Éléments de recherche introuvables');
    return; // Arrête : pas de recherche sur cette page
}

let searchTimeout; // Timer du debounce — doit survivre entre les frappes

getElementById vs querySelector

getElementById('search-input') utilise l'index interne du DOM (une hashmap par id), plus direct que querySelector('#search-input') qui parse un sélecteur CSS. Pour un id unique connu, getElementById est la méthode idiomatique.

Le double guard

La condition !searchInput || !searchResults est évaluée en court-circuit : si !searchInput est true, la deuxième partie n'est pas évaluée. Un seul return protège tout le reste du callback.

💡
recherche.js est chargé sur toutes les pages. Sur les pages sans barre de recherche (FAQ, confirmation…), ces éléments n'existent pas. Le guard clause empêche des erreurs sur ces pages.

let searchTimeout — Scope critique

searchTimeout est déclarée dans le scope du module (pas dans le callback input). Elle doit survivre entre deux frappes pour que clearTimeout(searchTimeout) puisse annuler le timer précédent.

⚠️
Si searchTimeout était dans le callback : elle serait réinitialisée à undefined à chaque frappe → clearTimeout(undefined) ne fait rien → le debounce ne fonctionne pas → une recherche par frappe → 10 frappes = 10 appels simultanés à PearTechDB.search().

Pourquoi ces deux éléments et pas d'autres

#search-input est le déclencheur : on écoute ses frappes. #search-results est la zone d'affichage : on y injecte le HTML et on contrôle sa visibilité via la classe active. Ce sont les deux points de contact entre le module JS et le HTML. Tout le reste (construction HTML, navigation, fermeture) est géré en mémoire ou via des sélecteurs dynamiques après injection.

Pourquoi let searchTimeout hors de l'événement input

⚠️
let searchTimeout est déclaré dans le scope DOMContentLoaded — ni dans le listener input ni dans performSearch(). Si on le déclarait dans le listener, chaque frappe créerait une nouvelle variable locale — clearTimeout(searchTimeout) n'annulerait jamais le timer précédent. La variable doit survivre entre les appels du listener.

Double guard — pourquoi deux vérifications séparées

javascript
const searchInput   = document.getElementById('search-input');
const searchResults = document.getElementById('search-results');

// Guard combiné — les deux doivent exister
if (!searchInput || !searchResults) {
    console.warn('Éléments de recherche manquants');
    return; // Arrête tout — le module ne s'initialise pas
}
searchInput absent
Pas de champ de recherche sur cette page
Normal sur certaines pages (confirmation, paiement)
return silencieux — pas d'erreur
searchResults absent
Champ présent mais pas la zone de résultats
Bug HTML — les deux doivent aller ensemble
Le warn() signale ce cas anormal
🔍
Le || dans le guard est un OU logique : si l'un OU l'autre manque, on sort. Les deux éléments sont co-dépendants : sans input, on ne peut pas écouter les frappes ; sans results, on ne peut pas afficher. L'un sans l'autre est toujours un bug ou une absence intentionnelle.
02 — Le debounce

Ne pas chercher à chaque frappe ¶

Le debounce est l'un des patterns JS les plus importants en UI : au lieu d'exécuter à chaque événement, on attend que les événements se calment.

Le problème sans debounce

Sans debounce, chercher iphone déclencherait 6 recherches : i, ip, iph, ipho, iphon, iphone. Chaque recherche boucle sur tous les produits de PearTechDB et régénère le HTML. 600 opérations en moins d'une seconde pour un résultat dont seul le dernier a de la valeur.

javascript
searchInput.addEventListener('input', function() {
    const query = this.value.trim(); // Texte sans espaces début/fin

    clearTimeout(searchTimeout); // Annule le timer de la frappe précédente

    if (query.length === 0) {     // Champ vide → cacher les résultats
        searchResults.classList.remove('active');
        return;
    }

    // Lance la recherche 300ms après la DERNIÈRE frappe
    searchTimeout = setTimeout(function() {
        performSearch(query);
    }, 300);
});

'input' vs 'keyup' vs 'keydown'

'input' se déclenche à chaque modification : frappe, coller (Ctrl+V), couper, auto-complétion. 'keyup' ne capte que le clavier (manque le coller). 'keydown' se déclenche avant la mise à jour de la valeur. 'input' est l'événement correct.

Pourquoi 300ms et pas 100ms ou 500ms ?

300ms est le seuil de perception du délai pour les retours d'interface : en dessous, l'utilisateur ne perçoit pas la latence. Au-dessus de 400-500ms, il commence à trouver l'interface lente. 300ms est le compromis standard pour les recherches à la frappe — suffisant pour annuler les frappes intermédiaires d'un mot normal, assez court pour paraître instantané.

03 — performSearch() et displaySearchResults()

Le pont vers PearTechDB et la construction du HTML ¶

performSearch — Vérification et délégation

javascript
function performSearch(query) {
    if (!window.PearTechDB) { // DB pas encore chargée
        searchResults.innerHTML = '<div class="search-no-results">Chargement...</div>';
        searchResults.classList.add('active');
        return;
    }
    const results = PearTechDB.search(query); // Délègue à data.js
    displaySearchResults(results);
}

La vérification !window.PearTechDB est faite à chaque recherche (pas uniquement à l'init) car data.js peut charger en parallèle. Au lieu de crasher, on affiche Chargement... — c'est la dégradation gracieuse.

displaySearchResults — Construction HTML

javascript
function displaySearchResults(results) {
    if (results.length === 0) {
        searchResults.innerHTML = '<div class="search-no-results">Aucun résultat</div>';
        searchResults.classList.add('active');
        return;
    }

    let html = ''; // Accumulation en mémoire (O(n), pas O(n²))
    results.forEach(product => {
        const specsPreview = Object.values(product.specs).slice(0, 2).join(' - ');
        html += '<div class="search-result-item" data-id="' + product.id + '">';
        html += '<div class="search-result-name">' + product.name + '</div>';
        html += '<div class="search-result-specs">' + specsPreview + '</div></div>';
    });

    searchResults.innerHTML = html;  // 1 seule injection DOM
    searchResults.classList.add('active');
    attachSearchResultEvents();       // OBLIGATOIRE après innerHTML
}

Accumulation + 1 injection vs innerHTML +=

Le pattern let html = ''; forEach → html += ...; element.innerHTML = html est plus efficace que element.innerHTML += ... à chaque itération. Chaque += innerHTML force le navigateur à re-parser tout le HTML existant — O(n²). L'accumulation puis 1 injection est O(n).

Object.values(product.specs).slice(0, 2).join(' - ')

product.specs est un objet { stockage: '128 Go', ram: '6 Go' }. Object.values() → tableau de valeurs. .slice(0, 2) → 2 premières. .join(' - ') → '128 Go - 6 Go'. Aperçu rapide des specs dans le résultat de recherche.

⚠️
Pourquoi attachSearchResultEvents() est dans displaySearchResults() et pas ailleurs : les .search-result-item n'existent QU'APRÈS searchResults.innerHTML = html. De plus, à chaque nouvelle recherche le innerHTML est réécrit — les anciens listeners sont détruits et doivent être rebranchés.

Fermeture au clic hors zone

javascript
document.addEventListener('click', function(e) {
    if (!searchInput.contains(e.target) && !searchResults.contains(e.target)) {

element.contains(node) retourne true si node est le même élément ou un descendant. La propagation des événements (bubbling) fait remonter chaque clic jusqu'à document — on capture ainsi tous les clics de la page.

Navigation clavier

searchInput.addEventListener('keydown', function(e) {

Le code complet de recherche.js résumé

javascript
(function() {
    'use strict';
    document.addEventListener('DOMContentLoaded', function() {

        // 1. Éléments + double guard
        const searchInput   = document.getElementById('search-input');
        const searchResults = document.getElementById('search-results');
        if (!searchInput || !searchResults) return;

        let searchTimeout; // scope module — survit entre les frappes

        // 2. Debounce 300ms
        searchInput.addEventListener('input', function() {
            const query = this.value.trim();
            clearTimeout(searchTimeout);
            if (!query) { searchResults.classList.remove('active'); return; }
            searchTimeout = setTimeout(() => performSearch(query), 300);
        });

        // 3. Délégation vers PearTechDB
        function performSearch(query) {
            if (!window.PearTechDB) { /* message chargement */ return; }
            displaySearchResults(PearTechDB.search(query));
        }

        // 4. Rendu HTML + injection unique
        function displaySearchResults(results) {
            if (!results.length) { /* message vide */ return; }
            let html = '';
            results.forEach(p => { html += '<div data-id="' + p.id + '">...</div>'; });
            searchResults.innerHTML = html;
            searchResults.classList.add('active');
            attachSearchResultEvents(); // APRÈS innerHTML
        }

        // 5. Listeners post-injection
        function attachSearchResultEvents() {
            document.querySelectorAll('.search-result-item').forEach(item => {
                item.addEventListener('click', function() {
                    window.location.href = 'page_produit.html?id=' + this.dataset.id;
                });
            });
        }

        // 6. Fermeture + clavier
        document.addEventListener('click', e => {
            if (!searchInput.contains(e.target) && !searchResults.contains(e.target))
                searchResults.classList.remove('active');
        });
        searchInput.addEventListener('keydown', e => {
            if (e.key === 'Enter') searchResults.querySelector('.search-result-item')?.click();
            if (e.key === 'Escape') { searchResults.classList.remove('active'); searchInput.blur(); }
        });
    });
})();

06 — Ordre du code

Pourquoi chaque bloc est placé là où il est ¶

recherche.js a la cascade la plus stricte du projet : input → performSearch → displaySearchResults → attachSearchResultEvents. Chaque fonction en appelle la suivante. L'ordre du code = l'ordre d'exécution réel.

1
Ressources centralesgetElementById('search-input') + getElementById('search-results')
Utilisés dans toutes les fonctions — une seule récupération en tête de scope. searchInput est utilisé dans le listener input, le listener keydown, le listener document.click et dans performSearch(). searchResults dans displaySearchResults(), performSearch() et les listeners. Économie : 1 requête DOM au lieu de 8+.
Utilisés dans toutes les étapes 3 à 9.
↓ double guard — un seul return protège tout
2
Double protectionif (!searchInput || !searchResults) return
Un seul return protège contre l'absence des deux éléments. L'opérateur || court-circuite : si searchInput est null, searchResults n'est même pas évalué. Le module entier est conditionnel à l'existence des deux — sans eux, rien n'a de sens.
Sans guard : si searchInput est null → searchInput.addEventListener → TypeError immédiat. Si searchResults est null → l'erreur surgit dans displaySearchResults() bien plus loin → plus difficile à déboguer (l'erreur ne pointe pas vers la vraie cause).
↓ la variable debounce doit survivre entre les appels du listener
3
État debouncelet searchTimeout
Déclarée dans le scope du module pour persister entre les appels du listener. Le listener input est appelé à chaque frappe. clearTimeout(searchTimeout) annule le timer de la frappe précédente — cela ne fonctionne que si searchTimeout est dans un scope qui survit entre les appels. Si elle était dans le callback, elle serait réinitialisée à chaque frappe → clearTimeout(undefined) → aucun effet de debounce.
Si déclarée dans le callback : let searchTimeout; clearTimeout(searchTimeout) → clearTimeout(undefined) → pas d'annulation → une recherche par frappe → 10 frappes = 10 appels à performSearch() simultanément → lag visible + charge inutile.
↓ entrée principale — déclenche toute la cascade
4
Entrée utilisateursearchInput.addEventListener('input') — debounce 300ms
Premier listener — c'est l'entrée principale du module. Tout commence ici : l'utilisateur tape → 300ms d'inactivité → performSearch(query). Le debounce de 300ms (vs 200ms pour la FAQ) est délibéré : la recherche implique un parcours de PearTechDB (potentiellement des centaines de produits) — plus coûteux que le simple filtrage de texte de la FAQ.
Appelle performSearch() — hoistée depuis l'étape 5.
↓ performSearch délègue à PearTechDB puis à displaySearchResults
5
Moteur de recherchefunction performSearch(query)
Pont entre l'input utilisateur et PearTechDB. Ne contient aucun algorithme de recherche — délègue entièrement à PearTechDB.search(query). La vérification de PearTechDB est dans cette fonction (pas dans le listener) car elle sera revérifiée à chaque recherche — PearTechDB peut charger après le premier appel.
Si PearTechDB n'est pas chargé : affiche 'Chargement en cours…' au lieu de crasher. Cette vérification est dans performSearch() (et non dans le listener) car elle doit être faite à chaque recherche, pas une seule fois à l'initialisation.
↓ injecter le HTML AVANT d'attacher les listeners dessus
6
Affichage résultatsfunction displaySearchResults(results)
Appelée par performSearch() — doit appeler attachSearchResultEvents() après l'injection HTML. Elle injecte via searchResults.innerHTML = html, puis appelle attachSearchResultEvents(). L'ordre interne est une contrainte DOM absolue : les .search-result-item n'existent qu'après l'injection.
Si attachSearchResultEvents() était appelée avant innerHTML : querySelectorAll('.search-result-item') → 0 éléments → aucun listener → les résultats s'affichent mais les clics ne font rien. Bug non bloquant mais fonctionnellement cassé.
↓ les items existent dans le DOM → on peut les écouter
7
Listeners résultatsfunction attachSearchResultEvents()
La contrainte la plus forte du fichier — après l'injection HTML, jamais avant. Appelée à la fin de displaySearchResults() car les .search-result-item sont régénérés à chaque nouvelle recherche. Les listeners sont également régénérés. Une délégation d'événements sur searchResults aurait évité ce pattern (un seul listener parent), mais l'approche actuelle est plus explicite et plus facile à lire.
Chaque fois que displaySearchResults() régénère le HTML (innerHTML =), tous les anciens listeners sur les enfants sont détruits. attachSearchResultEvents() les recrée immédiatement après.
↓ fermeture au clic hors zone — comportement secondaire
8
Fermeture externedocument.addEventListener('click') — ferme si clic hors champ ou résultats
Comportement secondaire indépendant — placé après la cascade principale. Surveille tous les clics sur le document. Si le clic n'est ni sur searchInput ni sur searchResults, les résultats sont masqués. Indépendant du flux recherche → affichage → listeners.
↓ navigation clavier — accessibilité en complément
9
Navigation claviersearchInput.addEventListener('keydown') — Entrée + Échap
En avant-dernier — amélioration d'accessibilité au-dessus de la fonctionnalité principale. Entrée → navigue vers le premier résultat en simulant .click() (déclenche le listener de l'étape 7). Échap → masque les résultats. Ces comportements ne sont utiles que si des résultats s'affichent — ils viennent après tout le système d'affichage.
↓ confirmation finale
10
Log de confirmationconsole.log('Recherche initialisée avec', productCount, 'produits')
En tout dernier — signal que tout s'est bien initialisé. L'accès à PearTechDB.products.length ici vérifie implicitement que la base est disponible. Le log 'X produits' est un outil de diagnostic passif : si on voit '0 produits', c'est que PearTechDB n'est pas encore chargé au moment de l'initialisation — indice pour vérifier l'ordre des <script> dans le HTML.

La cascade input → performSearch → displaySearchResults → attachResultEvents est une dépendance stricte : chaque étape prérequiert la précédente. attachSearchResultEvents() est appelée à chaque rendu car innerHTML = détruit les anciens listeners.

Concepts et termes techniques utilisés dans ce fichier ¶

Définitions des termes importants rencontrés dans cette documentation.

Document Object Model
Représentation en mémoire de la page HTML sous forme d'arbre. Chaque balise devient un nœud. JavaScript modifie cet arbre via getElementById(), querySelector(), innerHTML. Modifier le DOM = modifier ce que l'utilisateur voit en temps réel.
Immediately Invoked Function Expression
Fonction qui s'exécute immédiatement à sa déclaration : (function() { ... })(). Crée un scope isolé — tout ce qui est déclaré dedans reste privé et ne pollue pas window. Tous les fichiers JS du projet utilisent ce pattern.
DOMContentLoaded
Événement déclenché quand le HTML est parsé et le DOM prêt — avant images et CSS. Moment idéal pour initialiser les modules JS. Distinct de window.load qui attend tous les assets réseau.
debounce
Regroupe plusieurs appels rapides en un seul différé. Frappe 5 lettres en 200ms → 1 seule recherche lancée 300ms après la dernière frappe. Implémenté avec clearTimeout(timer) + timer = setTimeout(fn, 300).
Template literal
Chaîne avec backtick `...` qui permet d'insérer des expressions avec ${variable} et des retours à la ligne. Remplace la concaténation de strings. Utilisé massivement pour générer le HTML des cards dynamiquement.
dataset
Interface qui expose les attributs data-* d'un élément. el.dataset.productId lit data-product-id. Permet de stocker des données métier directement dans le HTML sans variables globales. La conversion camelCase est automatique : data-product-id → dataset.productId.

Recherche temps réel : debounce, accessibilité et edge cases ¶

💡
Analogie — le debounce expliqué autrement : Sans debounce, taper "clavier" lance 7 recherches : "c", "cl", "cla"… C'est comme sonner à la porte de quelqu'un 7 fois en 2 secondes. Avec debounce, tu attends qu'il ait arrêté de sonner depuis 300ms avant d'ouvrir.
💡
Analogie — contains() pour la fermeture : Vérifier si un clic est "dans" la zone de recherche avec .contains(), c'est comme vérifier si quelqu'un est dans ta cuisine en regardant si cette personne est toi-même ou dans ta famille directe. Si c'est un étranger, tu fermes la porte.
🚨 PIÈGE — Recherche sensible à la casse
Toujours normaliser avant de comparer : query.toLowerCase() et product.name.toLowerCase(). L'utilisateur qui tape "macbook" doit trouver "MacBook Pro". Idem pour les accents : normaliser avec .normalize('NFD').replace(/[̀-ͯ]/g,'').
🚨 PIÈGE — Recherche qui plante sur caractères spéciaux
Si la recherche utilise une RegExp construite depuis l'input utilisateur, des caractères comme ., *, ( vont casser la regex. Toujours échapper : query.replace(/[.*+?^${}()|[\]\]/g, '\$&') ou utiliser includes() au lieu de test().
PRO TIP
Mets en surbrillance les caractères correspondants dans les résultats : remplace le texte trouvé par <mark>texte</mark>. L'utilisateur comprend immédiatement pourquoi ce résultat est affiché.
PRO TIP
Ajoute role="combobox", aria-autocomplete="list" sur l'input et role="listbox" sur les résultats. Les lecteurs d'écran annoncent "3 suggestions" quand les résultats apparaissent.

// recherche.js - Recherche en temps réel
// Affiche les résultats sous la barre de recherche
// pendant la frappe, avec debounce et navigation clavier
// ============================================

(function() {
    'use strict'; // Mode strict activé

    document.addEventListener('DOMContentLoaded', function() {

        // ── Récupération des éléments de recherche ────────────────

        const searchInput   = document.getElementById('search-input');   // Champ de saisie
        const searchResults = document.getElementById('search-results'); // Zone d'affichage des résultats

        if (!searchInput || !searchResults) { // Si l'un des deux est absent de la page
            console.warn('Éléments de recherche introuvables');
            return; // Arrête : pas de recherche sur cette page
        }

        let searchTimeout; // Référence au setTimeout pour le debounce (annulation possible)

        // ── Événement de frappe avec debounce ─────────────────────
        // Le debounce évite de lancer une recherche à chaque lettre :
        // on attend 300ms après la dernière frappe

        searchInput.addEventListener('input', function() {
            const query = this.value.trim(); // Texte saisi sans espaces en début/fin

            clearTimeout(searchTimeout); // Annule le timer de la frappe précédente

            if (query.length === 0) { // Champ vide : on cache les résultats
                searchResults.classList.remove('active'); // Cache la zone de résultats
                return;
            }

            // Lance la recherche 300ms après la dernière frappe
            searchTimeout = setTimeout(function() {
                performSearch(query); // Lance la recherche avec le terme actuel
            }, 300);
        });

        // ── Fonction de recherche ──────────────────────────────────

        function performSearch(query) {
            if (!window.PearTechDB) { // Base de données pas encore chargée
                console.warn('PearTechDB non chargé, recherche impossible');
                searchResults.innerHTML = '<div class="search-no-results">Chargement en cours...</div>';
                searchResults.classList.add('active'); // Affiche quand même le message
                return;
            }
            const results = PearTechDB.search(query); // Délègue la logique de recherche à data.js
            displaySearchResults(results); // Affiche les résultats retournés
        }

        // ── Affichage des résultats ────────────────────────────────

        function displaySearchResults(results) {
            if (results.length === 0) { // Aucun produit trouvé
                searchResults.innerHTML = '<div class="search-no-results">Aucun résultat trouvé</div>';
                searchResults.classList.add('active');
                return;
            }

            let html = ''; // Construit le HTML des résultats
            results.forEach(product => {
                // Extrait les 2 premières spécifications pour l'aperçu
                const specsPreview = Object.values(product.specs).slice(0, 2).join(' - ');
                html += `
                    <div class="search-result-item" data-id="${product.id}">
                        <div class="search-result-name">${product.name}</div>
                        <div class="search-result-specs">${specsPreview}</div>
                    </div>
                `;
            });

            searchResults.innerHTML = html;          // Injecte les résultats
            searchResults.classList.add('active');   // Affiche la zone de résultats
            attachSearchResultEvents();              // Attache les clics sur les résultats
        }

        // ── Événements sur les résultats de recherche ─────────────

        function attachSearchResultEvents() {
            const resultItems = document.querySelectorAll('.search-result-item');
            resultItems.forEach(item => {
                item.addEventListener('click', function() {
                    const productId = this.dataset.id; // Récupère l'ID du produit cliqué
                    window.location.href = `page_produit.html?id=${productId}`; // Navigue vers la fiche
                });
            });
        }

        // ── Fermeture en cliquant ailleurs ────────────────────────

        document.addEventListener('click', function(e) {
            // Ferme si le clic n'est ni dans le champ ni dans les résultats
            if (!searchInput.contains(e.target) && !searchResults.contains(e.target)) {
                searchResults.classList.remove('active'); // Cache les résultats
            }
        });

        // ── Navigation clavier ────────────────────────────────────

        searchInput.addEventListener('keydown', function(e) {
            if (e.key === 'Enter') {
                // Entrée : navigue vers le premier résultat
                const firstResult = searchResults.querySelector('.search-result-item');
                if (firstResult) {
                    firstResult.click(); // Déclenche la navigation via le click handler
                }
            } else if (e.key === 'Escape') {
                searchResults.classList.remove('active'); // Cache les résultats
                searchInput.blur(); // Enlève le focus du champ
            }
        });

        // Log du nombre de produits indexés pour la recherche
        const productCount = window.PearTechDB ? PearTechDB.products.length : 0;
        console.log('Recherche initialisée avec', productCount, 'produits');
    });
})(); // Fin de l'IIFE

② La normalisation — recherche robuste

function normalize(str) {
  return str
    .toLowerCase()
    // 'MacBook' → 'macbook'

    .normalize('NFD')
    // Décompose les caractères accentués en lettre de base + accent.
    // 'é' devient 'e' + '◌́' (deux code points séparés)

    .replace(/[\u0300-\u036f]/g, '');
    // Supprime tous les code points d'accents (plage Unicode U+0300–U+036F).
    // Résultat : 'é' → 'e', 'ç' → 'c', 'ü' → 'u'
}
// Test : normalize('Écran 4K') → 'ecran 4k'
// L'utilisateur peut taper 'ecran' et trouver 'Écran 4K'. ✓

function performSearch(query) {
  const q = normalize(query);

  if (q.length < 2) {
    hideResults();
    return; // Pas de recherche pour 0 ou 1 caractère
  }

  const results = PearTechDB.products.filter(p => {
    return normalize(p.name).includes(q)
        || normalize(p.category).includes(q);
    // On normalise AUSSI les données produits (pas seulement la query).
    // 'Écran' en base → normalize → 'ecran'
    // query 'ecran'   → normalize → 'ecran'
    // 'ecran'.includes('ecran') → true ✓
  });

  renderResults(results, q);
  // On passe q (normalisé) pour surligner les occurrences dans les résultats.
}

③ Surlignage des résultats — highlight

function highlightMatch(text, query) {
  // Échappe les caractères spéciaux pour que la query soit littérale dans la regex.
  // Sans ça : query = 'a.b' → regex /a.b/i → . = n'importe quel caractère → faux positifs.
  const escaped = query.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
  // '\\$&' : $& = le match lui-même, \\ = l'échapper. Ex: 'a.' → 'a\.'

  const regex = new RegExp(escaped, 'gi');
  // Flags : g = global (toutes les occurrences), i = insensible à la casse

  return text.replace(regex, match => `<mark>${match}</mark>`);
  // Chaque occurrence est entourée de <mark> → surlignage CSS natif.
  // On garde la casse originale (match, pas query) : 'MacBook' reste 'MacBook'.
}
