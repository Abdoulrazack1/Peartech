Comprendre
cart.js

Ce fichier gère exclusivement la page panier (page_panier.html). Il lit les articles sauvegardés, les affiche, permet de modifier les quantités, applique des codes promo, calcule les totaux, et redirige vers le paiement en transmettant la promo via sessionStorage.

00 — Distinction fondamentale

panier.js vs cart.js — deux fichiers, deux rôles

panier.js vs cart.js — qui fait quoi
panier.js — le moteur (toutes les pages)
✅ Écrit dans localStorage
✅ Expose window.PearTechCart
✅ Écoute les clics "Ajouter" sur tout le site
✅ Met à jour le badge du header
✅ Émet productAddedToCart
🚫 N'affiche pas de récapitulatif
cart.js — l'interface (page_panier.html uniquement)
✅ Lit localStorage pour afficher les articles
✅ Modifie les quantités via PearTechCart
✅ Valide les codes promo
✅ Calcule les totaux
✅ Écrit sessionStorage['peartech-promo']
✅ Redirige vers page_paiement.html
Analogie : panier.js = la caisse enregistreuse invisible qui gère le stock · cart.js = le vendeur qui te montre le récapitulatif et prend ta commande
Les deux fichiers "panier"
panier.js — TOUTES les pages
✓ Écoute les clics "Ajouter au panier"
✓ Écrit dans localStorage
✓ Met à jour le badge de navigation
✓ Expose window.PearTechCart
✗ Ne sait pas afficher le contenu du panier
cart.js — page_panier.html SEULEMENT
✓ Lit localStorage et affiche les articles
✓ Modifie quantités avec +/-
✓ Applique des codes promo
✓ Calcule sous-total / livraison / total
✓ Transmet la promo vers le paiement
💡Analogie : panier.js est le caissier qui enregistre chaque ajout dans le caddie. cart.js est l'écran de caisse qui affiche le contenu complet et calcule le total.
📖 Termes nouveaux dans cart.js
sessionStorage
Comme localStorage mais effacé à la fermeture de l'onglet. Utilisé pour transporter la promo de la page panier vers la page paiement. La promo ne doit pas persister indéfiniment.
.splice(index, 1)
Supprime 1 élément à la position index en modifiant le tableau original sur place. cartItems.splice(2, 1) supprime l'article à l'index 2.
[...tableau].sort()
Le spread [...] crée une copie avant de trier. Sans ça, .sort() modifierait PearTechDB.products définitivement — cassant toutes les fonctions qui en dépendent.
renderCart()
Recrée entièrement le HTML du panier à chaque appel. Quand une quantité change, tout est regénéré. Les écouteurs doivent être ré-attachés à chaque render.
01 — Chargement

Lire le panier et normaliser les images

La première chose que fait cart.js est de lire le panier depuis localStorage et de corriger un problème de rétrocompatibilité : selon la page d'origine, l'image peut être sous le nom image (string) ou images (tableau ou string).

Normalisation du champ image — 3 formats d'entrée → 1 format unifié
item.image = "phone.jpg"
// champ image déjà présent
→
item.image = "phone.jpg" ✓
item.images = ["a.jpg","b.jpg"]
// tableau, pas de .image
→
item.image = "a.jpg" ✓
// item.images[0]
item.images = "a.jpg"
// string dans images (legacy)
→
item.image = "a.jpg" ✓
// string directement
Résultat : tous les items ont item.image (string). renderCart() peut faire <img src="${item.image}"> sans condition.
javascript
let cartItems = (JSON.parse(localStorage.getItem(CART_STORAGE_KEY)) || [])
    .map(item => {
        if (!item.image) {
            if (Array.isArray(item.images) && item.images.length > 0)
                item.image = item.images[0];          // images est un tableau → prend [0]
            else if (typeof item.images === 'string')
                item.image = item.images;                // images est une string → copie
        }
        return item;
    });
3 cas de normalisation de l'image
Cas normal
{image: "/img/iph.jpg"}
→ rien à faire
Tableau
{images: ["/img/…"]}
→ image = images[0]
String legacy
{images: "/img/…"}
→ image = images

renderCart() — reconstruire le DOM à chaque changement

⚠️
Pourquoi ré-attacher les événements à chaque render ? cartContainer.innerHTML = html détruit et recrée tous les éléments DOM. Les anciens boutons +/- et leurs écouteurs disparaissent. calculateTotals() — la formule

calculateTotals() — exemple avec des vrais chiffres
Sous-total brut (reduce qty × price) 1 249,00 €
Code promo PEARTECH10 (−10%) − 124,90 €
Livraison standard + 9,99 €
TOTAL 1 134,09 €
subtotal = items.reduce((sum, item) => sum + item.price * item.quantity, 0)
discount = promoActive ? subtotal * promoDiscount : 0
total = subtotal - discount + shippingCost
javascript
function calculateTotals() {
    const subtotal = cartItems.reduce((acc, item) => acc + (item.price * item.quantity), 0);
    const shipping = 5.90;
    let   discount = currentPromo ? subtotal * currentPromo.discount : 0;
    // discount s'applique SUR le sous-total, pas sur sous-total+livraison
    const total    = subtotal + shipping - discount;

    subtotalSpan.textContent = formatPrice(subtotal);
    shippingSpan.textContent = formatPrice(shipping);
    discountSpan.textContent = `-${formatPrice(discount)}`;
    totalSpan.textContent    = formatPrice(total);
}
// formatPrice(1299) → "1299.00" → "1299,00" → "1 299,00 €"
// toFixed(2).replace('.', ',') + ' €'

05 — Paiement

Transmettre la promo via sessionStorage

Passage de données vers paiement.js via sessionStorage
1
cart.js
Utilisateur valide un code promo → validatePromoCode(code) → objet {code, discount}
↓ localStorage ne suffit pas (paiement.js a besoin de savoir si une promo est active pour cette session)
2
cart.js
Clic "Commander" → sessionStorage.setItem('peartech-promo', JSON.stringify({code, discount}))
↓ navigation vers page_paiement.html
3
paiement.js
Lit sessionStorage.getItem('peartech-promo') → parse → applique la réduction dans renderSummary()
↓ sessionStorage disparaît à la fermeture de l'onglet — la promo ne persiste pas indéfiniment
Pourquoi pas localStorage ? Car le code promo est lié à cette session d'achat précise. Une promo utilisée en janvier ne doit pas réapparaître en mars.
localStorage vs sessionStorage
localStorage
Persiste après fermeture du navigateur. → Panier peartech-cart doit survivre entre sessions.
sessionStorage
Effacé à la fermeture de l'onglet. → Promo peartech-promo ne doit durer que cette session.
Flux promo panier → paiement
cart.js — currentPromo = {code:"PEARTECH10", discount:0.10}
↓
sessionStorage.setItem('peartech-promo', JSON.stringify(currentPromo))
sessionStorage — '{"code":"PEARTECH10","discount":0.1}'
↓
window.location.href = 'page_paiement.html'
paiement.js lit sessionStorage et applique la réduction
06 — Recommandations

4 produits aléatoires — spread avant sort

4 produits aléatoires — spread avant sort, pourquoi ?
❌ Sans spread — mutation du tableau original
products.sort(() => Math.random() - .5)
← trie products[] en place
2e appel → ordre différent
← allProducts est corrompu
✅ Avec spread — copie, puis tri
[...products]
.sort(() => Math.random() - .5)
.slice(0, 4)
← allProducts intact
Le spread [...arr] crée une copie superficielle (shallow copy) du tableau — seul le tableau est copié, pas les objets dedans. Parfait pour trier sans effets de bord.
javascript
const shuffled = [...allProducts].sort(() => 0.5 - Math.random());
// [...]  = copie du tableau — OBLIGATOIRE
// Sans ça, .sort() mélange PearTechDB.products en place
// → getProductsByCategory() renvoie des produits dans un ordre cassé
// () => 0.5 - Math.random() : positif ou négatif aléatoire → mélange
const recommendations = shuffled.slice(0, 4);

// Appel conditionnel : PearTechDB disponible immédiatement ou dans 500ms
if (window.PearTechDB) { loadRecommendations(); }
else { setTimeout(loadRecommendations, 500); }
