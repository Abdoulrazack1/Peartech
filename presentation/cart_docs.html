02 — Affichage

renderCart() — reconstruire le DOM à chaque changement ¶

La fonction renderCart() est le cœur de cart.js. Elle reconstruit intégralement le HTML du panier à chaque modification (ajout, suppression, changement de quantité). C'est une approche impérative de re-rendu total — simple et fiable.

Le code clé : innerHTML + ré-attachement des listeners

javascript
function renderCart() {
    if (cartItems.length === 0) {
        cartContainer.innerHTML = '<p class="empty-cart">Votre panier est vide.</p>';
        return;
    }

    let html = ''; // Accumulation O(n) — 1 seule injection
    cartItems.forEach((item, index) => {
        const optionsHtml = item.options && item.options.length
            ? '<div class="item-options">' + item.options.map(...).join(', ') + '</div>'
            : '';
        html += '<div class="cart-item" data-index="' + index + '">...</div>';
    });

    cartContainer.innerHTML = html; // ← innerHTML réécrit TOUT

    // OBLIGATOIRE : ré-attacher les listeners après innerHTML
    document.querySelectorAll('.plus').forEach(btn => {
        btn.addEventListener('click', function() {
            cartItems[this.dataset.index].quantity++;
            renderCart();         // ← Récursif : re-render après chaque action
            calculateTotals();
            saveCart();
        });
    });
    // ... idem pour .minus, .remove-item, .save-for-later
}

Pourquoi innerHTML détruit les anciens listeners

Quand on écrit cartContainer.innerHTML = html, le navigateur détruit tous les éléments DOM existants et en crée de nouveaux. Les addEventListener attachés aux anciens éléments disparaissent avec eux — ils ne sont pas transférés aux nouveaux. C'est pourquoi le bloc de querySelectorAll('.plus').forEach(...) se trouve à l'intérieur de renderCart() et non une seule fois à l'initialisation.

⚠️
C'est le piège le plus courant avec innerHTML : croire que les listeners persistent. Ce n'est pas le cas. Toujours ré-attacher les listeners après une injection innerHTML. L'alternative architecturale serait la délégation d'événements (cartContainer.addEventListener('click', e => {...})) qui ne nécessite pas de ré-attachement.

data-index — Le pont entre DOM et tableau

Chaque .cart-item a un attribut data-index="0", data-index="1"… Ces index correspondent exactement aux positions dans le tableau cartItems[]. Quand l'utilisateur clique sur un bouton + ou -, on lit this.dataset.index pour savoir quel article modifier dans le tableau. C'est le mécanisme de synchronisation entre le DOM visuel et la donnée en mémoire.

renderCart() est récursif — intentionnellement

Quand l'utilisateur clique sur +, le listener appelle renderCart() qui reconstruit tout le DOM, ce qui crée de nouveaux boutons + avec de nouveaux listeners qui pourront à leur tour appeler renderCart(). Cette récursivité indirecte est le comportement voulu : chaque état du panier produit un DOM frais et cohérent.

💡
La séquence renderCart() → calculateTotals() → saveCart() est systématiquement appelée dans cet ordre après chaque modification. renderCart() met à jour l'affichage, calculateTotals() recalcule les prix, saveCart() persiste dans localStorage. L'ordre compte : on affiche d'abord, on calcule ensuite, on sauvegarde en dernier.

04 — Codes promo

Validation côté client ¶

Le module de codes promo implémente une validation entièrement côté client. Les codes valides sont définis dans un objet validPromoCodes directement dans le fichier JS.

La structure des codes promo

javascript
const validPromoCodes = {
    'PEARTECH10': { discount: 0.10, label: '-10%' },  // 10% de réduction
    'PEARTECH20': { discount: 0.20, label: '-20%' },  // 20% de réduction
    'BIENVENUE':  { discount: 0.05, label: '-5%'  }   // 5% pour nouveaux clients
};

Un objet { [code]: { discount, label } } permet une lookup en O(1) — lire validPromoCodes['PEARTECH10'] est instantané quelle que soit la taille de l'objet, contrairement à un Array.find() qui serait O(n).

La logique de validation

javascript
applyPromoBtn.addEventListener('click', function() {
    const code = promoInput.value.trim().toUpperCase(); // Normalisation
    if (!code) {
        promoMessage.textContent = 'Veuillez entrer un code.';
        promoMessage.className = 'promo-message error';
        return;
    }
    if (validPromoCodes[code]) { // Vérification dans l'objet
        const promo = validPromoCodes[code];
        currentPromo = { code, discount: promo.discount }; // Stocke la promo active
        promoMessage.className = 'promo-message success';
    } else {
        currentPromo = null; // Annule toute promo précédente
        promoMessage.className = 'promo-message error';
    }
    calculateTotals(); // Recalcule avec la réduction
});

.toUpperCase().trim() — Normalisation défensive

L'utilisateur peut saisir peartech10, PEARTECH10 ou Peartech10 — tous doivent être acceptés. .trim() supprime les espaces en début/fin. .toUpperCase() standardise la casse. Après normalisation, la comparaison avec les clés de l'objet est fiable.

currentPromo — Variable d'état partagée

currentPromo est déclarée dans le scope du module (pas dans le listener). Elle stocke le code promo actif et est lue par calculateTotals() pour calculer la réduction et par checkoutBtn pour transmettre la promo à la page de paiement. C'est un état partagé entre plusieurs fonctions.

⚠️
La validation côté client est pratique mais non sécurisée : n'importe qui peut ouvrir la console et taper validPromoCodes pour voir tous les codes. En production réelle, la validation se ferait côté serveur via une API — le client envoie le code, le serveur retourne la réduction sans exposer les autres codes.

05 — Paiement

Transmettre la promo via sessionStorage ¶

Le bouton de paiement effectue deux opérations : transmettre le code promo à la page de paiement via sessionStorage, puis naviguer vers page_paiement.html.

Le code complet

javascript
checkoutBtn.addEventListener('click', function() {
    if (cartItems.length === 0) {
        alert('Votre panier est vide.'); // Empêche la navigation si panier vide
        return;
    }
    if (currentPromo) {
        try {
            // Sérialise la promo en JSON pour sessionStorage
            sessionStorage.setItem('peartech-promo', JSON.stringify(currentPromo));
        } catch(e) {
            console.warn('sessionStorage indisponible :', e);
        }
    } else {
        try { sessionStorage.removeItem('peartech-promo'); } catch(e) {}
    }
    window.location.href = 'page_paiement.html'; // Navigation
});

sessionStorage vs localStorage — Pourquoi sessionStorage ici ?

localStorage persiste jusqu'à effacement explicite. sessionStorage existe uniquement pour la durée de l'onglet — il est détruit à la fermeture. Pour transmettre la promo de cart.js à paiement.js, on veut une donnée éphémère : elle n'a de sens que pendant cette transaction. Si l'utilisateur rouvre le site plus tard, l'ancienne promo ne doit pas s'appliquer automatiquement.

try/catch autour de sessionStorage

Bien que rare, sessionStorage peut être indisponible : en navigation privée sur certains navigateurs, en mode kiosque, ou quand le stockage est plein. Le try/catch garantit que l'impossibilité de sauvegarder la promo ne bloque pas la navigation vers le paiement — la commande passe quand même, sans réduction.

JSON.stringify() — Sérialiser un objet pour le stockage

sessionStorage ne stocke que des chaînes de caractères. currentPromo est un objet { code: "PEARTECH10", discount: 0.10 }. JSON.stringify() le convertit en '{"code":"PEARTECH10","discount":0.1}'. Côté paiement.js, JSON.parse() reconvertit la chaîne en objet.

✓
La suppression de la promo dans le else (sessionStorage.removeItem('peartech-promo')) est importante : si l'utilisateur avait appliqué une promo, l'avait retirée, puis cliquait sur Payer — une ancienne promo résiduelle dans sessionStorage serait appliquée. Le nettoyage explicite dans le else évite ce cas.

Pourquoi [...products].sort() et pas products.sort() directement
products.sort() — mutation du tableau source
1er appel → [A, C, B, D] (aléatoire)
products[] = [A, C, B, D] corrompu
2e appel → [C, A, D, B] (depuis état corrompu)
tri non reproductible

1er appel → copie [A,B,C,D] → tri → [C,A,B,D]
products[] = [A, B, C, D] intact
2e appel → copie fraîche → nouveau tri
source toujours propre

💡

Analogie — JSON.parse/stringify : localStorage ne comprend que du texte. JSON.stringify traduit ton tableau d'objets en texte (comme écrire une recette), JSON.parse traduit le texte en objets utilisables (comme lire la recette et cuisiner).
🚨 PIÈGE — JSON.parse sans try/catch
Si localStorage contient une valeur corrompue (bug précédent, extension navigateur), JSON.parse lève une SyntaxError qui plante tout le module. Toujours entourer de try/catch et retourner un tableau vide en cas d'erreur : try { return JSON.parse(…) } catch { return [] }.
🚨 PIÈGE — Comparer des objets avec === pour trouver un article
Deux objets identiques ne sont pas égaux avec === en JavaScript : {id:1} !== {id:1}. Pour trouver un article dans le panier, compare toujours les IDs : cart.find(item => item.id === productId), pas cart.find(item => item === product).
PRO TIP

// Utilisation d'une IIFE (Immediately Invoked Function Expression) pour isoler le code

(function() {
    'use strict'; // Active le mode strict pour une meilleure sécurité et optimisation

    // Clé utilisée pour stocker le panier dans localStorage
    const CART_STORAGE_KEY = 'peartech-cart';

    // Charge le panier depuis localStorage : 
    // - JSON.parse convertit la chaîne stockée en objet JavaScript
    // - || [] fournit un tableau vide si rien n'est stocké
    // - .map() parcourt chaque article pour normaliser le champ 'image'
    let cartItems = (JSON.parse(localStorage.getItem(CART_STORAGE_KEY)) || []).map(item => {
        // Si l'article n'a pas de propriété 'image' directe
        if (!item.image) {
            // Si un tableau 'images' existe et contient des éléments, on prend la première image
            if (Array.isArray(item.images) && item.images.length > 0) {
                item.image = item.images[0];
            } else if (typeof item.images === 'string') {
                // Si 'images' est une chaîne, on l'utilise directement comme image
                item.image = item.images;
            }

     * Sauvegarde le panier dans localStorage et met à jour le badge du panier
     */
    function saveCart() {
        // Sauvegarde le panier (chaîne JSON) dans localStorage
        localStorage.setItem(CART_STORAGE_KEY, JSON.stringify(cartItems));

        // Si l'API centralisée PearTechCart existe, on l'utilise pour mettre à jour le badge
        if (window.PearTechCart) {
            // On suppose que PearTechCart a une méthode updateCart pour remplacer le panier
            window.PearTechCart.updateCart(cartItems);
        } else {
            // Fallback : mise à jour manuelle du badge
            // Calcule le nombre total d'articles (somme des quantités)
            const total = cartItems.reduce((acc, item) => acc + item.quantity, 0);
            // Sauvegarde le compteur dans localStorage (au cas où)
            localStorage.setItem('peartech-cart-count', total);
            // Récupère l'élément badge par son ID

                // Cache le badge si total = 0, sinon l'affiche avec display:flex
                badge.style.display = total === 0 ? 'none' : 'flex';
            }
        }
    }

    /**
     * Calcule les totaux (sous-total, frais, réduction, total) et met à jour l'affichage
     */
    function calculateTotals() {
        // Sous-total : somme des prix * quantité pour chaque article
        const subtotal = cartItems.reduce((acc, item) => acc + (item.price * item.quantity), 0);
        const shipping = 5.90; // Frais de port fixes
        let discount = 0; // Réduction initiale

            // Si un code promo est appliqué, la réduction = sous-total * taux

        const total = subtotal + shipping - discount; // Total final

        subtotalSpan.textContent = formatPrice(subtotal);
        shippingSpan.textContent = formatPrice(shipping);
        discountSpan.textContent = `-${formatPrice(discount)}`; // Affiche avec un signe moins
        totalSpan.textContent    = formatPrice(total);

     * Formate un nombre en prix français (ex: 1234.56 -> "1 234,56 €")
     * @param {number} price - Le prix à formater
     * @returns {string} Le prix formaté

function getTotal() {
  const cart = getCart();

  // reduce() parcourt le tableau et accumule une valeur
  // acc   = accumulateur (commence à 0)
  // item  = élément courant
  return cart.reduce(function(acc, item) {
    return acc + (item.price * item.quantity);
  }, 0);

  // Exemple : [{price:10, quantity:2}, {price:5, quantity:1}]
  // Tour 1 : 0 + (10×2) = 20
  // Tour 2 : 20 + (5×1) = 25  → résultat : 25
}

— Ordre du code

Pourquoi chaque bloc est placé là où il est ¶

cart.js sans DOMContentLoaded (script en bas du body) suit : données → constantes → DOM → utilitaires → affichage → comportements → démarrage. La récursion dans renderCart() (ses listeners la rappellent) est intentionnelle.

1
Données centralesCART_STORAGE_KEY + let cartItems = JSON.parse(…).map(…)
En tout premier — cartItems est la donnée centrale de tout le module. Chargée une seule fois depuis localStorage et immédiatement normalisée (le .map() s'assure que chaque article a un champ image utilisable). Toutes les fonctions opèrent sur ce tableau en mémoire. Le re-lire dans renderCart() provoquerait une désynchronisation : les modifications en mémoire (quantité augmentée) non encore sauvegardées seraient perdues à chaque rendu.
Si chargé dans renderCart() : clic '+' → cartItems[0].quantity++ → renderCart() → cartItems = JSON.parse(localStorage…) → la quantité incrémentée est écrasée par la valeur en storage → le compteur ne s'incrémente jamais.
↓ constantes et état avant les fonctions qui les lisent
2
Constantes promo + étatconst validPromoCodes + let currentPromo = null
Avant calculateTotals() et le listener promo qui les utilisent. validPromoCodes est statique (données pures). currentPromo est l'état mutable de la promo active — partagé entre calculateTotals() (qui l'utilise pour calculer la réduction) et le listener 'Appliquer' (qui le met à jour). Les deux doivent accéder à la même variable — scope commun obligatoire.
currentPromo : lue dans calculateTotals(), écrite dans le listener applyPromoBtn. validPromoCodes : consultée dans le listener applyPromoBtn.
↓ éléments DOM une seule fois — pas de DOMContentLoaded car script en bas du body
3
Éléments DOM × 9cartContainer + subtotalSpan + … + recommendationsGrid
Un seul accès DOM par élément — pas de DOMContentLoaded car le script est en bas du <body>. Le script est chargé après tout le HTML — les éléments existent déjà quand le script s'exécute. Récupérer les 9 éléments ici (et non dans chaque fonction) économise 9 × N getElementById où N peut atteindre 20+ rendus de renderCart() par session.
Tous utilisés dans renderCart(), calculateTotals(), et les listeners promo/checkout.
↓ saveCart — brique atomique de persistance
4
Persistancefunction saveCart() — localStorage + badge header
Avant calculateTotals() et renderCart() qui la finissent par appeler. Brique atomique : persiste cartItems et met à jour le badge du header. Définie tôt car elle est la plus simple des trois fonctions principales. Elle est aussi appelée à l'initialisation (étape 10) pour synchroniser le badge dès le chargement.
↓ calculer avant afficher — logique avant DOM
5
Calculsfunction calculateTotals() — sous-total, frais, réduction, total
Indépendante de renderCart() — calculer n'est pas afficher. calculateTotals() ne génère pas de HTML, elle met uniquement à jour les spans de totaux. Elle est appelée après chaque renderCart() dans les callbacks, mais peut aussi être appelée seule (changement de promo). Cette indépendance justifie une définition séparée.
Lit : cartItems, currentPromo. Écrit dans : subtotalSpan, shippingSpan, discountSpan, totalSpan.

Helpers pursfunction formatPrice(price) + function handleImageError(img)

Leurs placer avant renderCart() : outils simples avant leur utilisateur complexe.
↓ renderCart — la plus complexe, après tous ses outils
7
Affichage + listeners inlinefunction renderCart() — génère HTML + attache .plus/.minus/.remove-item
La fonction la plus complexe — placée après tous ses outils et dépendances. Génère le HTML de toutes les cards puis attache immédiatement les listeners sur .plus, .minus, .remove-item. Ces listeners doivent être dans renderCart() (pas en dehors) car innerHTML = détruit tous les anciens listeners — ils doivent être rebranchés à chaque rendu. C'est une récursion contrôlée : les listeners appellent à nouveau renderCart().
Si les listeners étaient attachés une seule fois en dehors de renderCart() : après le premier cartContainer.innerHTML = html, les éléments précédents (.plus, .minus) sont détruits et recréés → les anciens listeners sont orphelins → les boutons +/- ne font plus rien. Bug silencieux visible uniquement après la première action.
↓ comportements ponctuels — attachés une seule fois sur des éléments fixes
8
Promo + CheckoutapplyPromoBtn.addEventListener + checkoutBtn.addEventListener
Attachés une seule fois sur des éléments fixes (pas détruits par renderCart). Le bouton promo et le bouton checkout sont des éléments HTML fixes — ils ne font pas partie de cartContainer qui est réécrit à chaque rendu. Leurs listeners survivent aux re-rendus. L'ordre promo → checkout reflète le flux utilisateur : appliquer un code avant de valider.
↓ fonctionnalité optionnelle — fonctionnerait sans
9
Recommandationsfunction loadRecommendations() + appel conditionnel if/else setTimeout
Fonctionnalité optionnelle — la page fonctionne sans elle. Si PearTechDB n'est pas encore disponible, un setTimeout(loadRecommendations, 500) réessaie après un délai. Si PearTechDB ne charge jamais, le grid de recommandations reste vide — aucun impact sur le panier ou les totaux.
↓ démarrage effectif — trois appels dans l'ordre obligatoire
10
DémarragerenderCart() + calculateTotals() + saveCart() — initialisation
Trois appels dans cet ordre précis — non interchangeables. (1) renderCart() : génère le HTML et attache les listeners. (2) calculateTotals() : calcule les totaux à partir du HTML rendu. (3) saveCart() : met à jour le badge du header. L'ordre 1→2→3 est logique : afficher, calculer, synchroniser. Inverser 1 et 2 fonctionnerait mais serait moins lisible.
Les appels d'initialisation sont toujours les dernières lignes — ils déclenchent le système une fois tout défini.
Règle générale de ce fichier

La décision clé : les listeners +/- sont dans renderCart() car innerHTML = les détruit à chaque rendu. Ils doivent être rebranchés à chaque génération. Cette contrainte impose de bien séparer les éléments fixes (promo, checkout) des éléments dynamiques (cards d'articles).

Concepts et termes techniques utilisés dans ce fichier ¶

Définitions des termes importants rencontrés dans cette documentation.

Document Object Model
Représentation en mémoire de la page HTML sous forme d'arbre. Chaque balise devient un nœud. JavaScript modifie cet arbre via getElementById(), querySelector(), innerHTML. Modifier le DOM = modifier ce que l'utilisateur voit en temps réel.
Immediately Invoked Function Expression
Fonction qui s'exécute immédiatement à sa déclaration : (function() { ... })(). Crée un scope isolé — tout ce qui est déclaré dedans reste privé et ne pollue pas window. Tous les fichiers JS du projet utilisent ce pattern.
DOMContentLoaded
Événement déclenché quand le HTML est parsé et le DOM prêt — avant images et CSS. Moment idéal pour initialiser les modules JS. Distinct de window.load qui attend tous les assets réseau.
localStorage
Stockage clé-valeur persistant dans le navigateur — survit à la fermeture. Capacité ~5 Mo. Clé/valeur uniquement en string : les objets passent par JSON.stringify() avant stockage et JSON.parse() à la lecture.
JSON — JavaScript Object Notation
Format texte pour sérialiser des objets. JSON.stringify(obj) → string stockable. JSON.parse(str) → objet utilisable. Obligatoire pour stocker/lire des objets dans localStorage ou sessionStorage.
Template literal
Chaîne avec backtick `...` qui permet d'insérer des expressions avec ${variable} et des retours à la ligne. Remplace la concaténation de strings. Utilisé massivement pour générer le HTML des cards dynamiquement.
Opérateur spread [...]
Étale les éléments d'un itérable. [...arr] crée une copie superficielle. [...a, ...b] fusionne. Utilisé pour copier un tableau avant de le trier sans modifier l'original.
Délégation d'événements
Un seul listener sur un parent au lieu d'un listener par enfant. L'événement remonte (bubble) de l'enfant vers le parent. e.target.closest('.btn') identifie l'élément source. Avantage : fonctionne pour les éléments ajoutés dynamiquement après l'initialisation.
dataset
Interface qui expose les attributs data-* d'un élément. el.dataset.productId lit data-product-id. Permet de stocker des données métier directement dans le HTML sans variables globales. La conversion camelCase est automatique : data-product-id → dataset.productId.
