00 — Avant de commencer

Affichage et masquage des cartes par data-category

Quand l'utilisateur clique sur un onglet ("Tous", "Apple", "Android"…), filtre.js montre les cartes correspondantes et cache toutes les autres. C'est tout.

Analogie : Imaginez une vitrine de magasin avec des produits. Certains sont étiquetés "Apple", d'autres "Android". Quand vous allumez le projecteur "Apple", seuls les produits Apple sont éclairés — les autres sont dans l'ombre. Cliquer sur un onglet, c'est choisir quel projecteur allumer.
Vocabulaire spécifique à ce fichier
classList
La liste des classes CSS d'un élément HTML. On peut l'ajouter, la retirer ou vérifier sa présence. Ex : element.classList.add('active') ajoute la classe "active" qui peut changer le style visuel de l'élément.
dataset
Les attributs data-* d'un élément HTML. Si un bouton a data-filter="apple", on peut lire cette valeur avec bouton.dataset.filter renvoie "apple".
display: none
Propriété CSS qui rend un élément complètement invisible ET retire son espace dans la page (comme s'il n'existait pas). Différent de visibility: hidden qui cache mais garde l'espace.
display: flex
Propriété CSS qui rend un élément visible en mode "flex" (mise en page flexible). Les cartes produit utilisent flex pour aligner leur contenu interne — il faut donc remettre "flex" et non "block" pour les réafficher correctement.
MutationObserver
Un outil JavaScript qui surveille les changements dans le DOM (la page). Il peut détecter quand des éléments sont ajoutés, supprimés ou modifiés, et exécuter une action en réponse.
this
Dans un écouteur d'événement avec function, this désigne l'élément qui a déclenché l'événement. Si l'onglet "Apple" est cliqué, this = l'onglet "Apple".

02 — La fonction de filtrage

filterProducts( filter )

Cette fonction reçoit un mot (ex: "apple") et parcourt toutes les cartes produit. Pour chacune, elle regarde sa catégorie et décide de la montrer ou la cacher.

La condition avec || (OU logique)

Opérateur || (OU) : La condition est vraie si au moins un des deux côtés est vrai. A || B est vrai si A est vrai, si B est vrai, ou si les deux sont vrais.

Court-circuit : JavaScript est paresseux. Si A est déjà vrai, il n'évalue même pas B. Donc si filter === 'tous', il ne vérifie pas category === filter. C'est plus efficace.

Pourquoi display: 'flex' et pas 'block' ?

Les .product-card sont stylées en CSS avec display: flex. Flex permet d'aligner l'image, le texte et le bouton à l'intérieur de la carte. Si on remettait 'block' au lieu de 'flex', le contenu interne de la carte serait désorienté — tout s'empilerait verticalement sans mise en forme. Il faut remettre exactement ce que le CSS attend.

Les cartes cachées (display:none) sont toujours présentes dans le HTML — elles ne sont pas supprimées. Elles reprennent leur place instantanément si on reclique "Tous". C'est bien plus rapide que de les supprimer et les recréer.
javascript 
function filterProducts(filter) {
// Récupère toutes les cartes présentes dans le DOM à cet instant const productCards = document.querySelectorAll('.product-card');
productCards.forEach(card => {
// Lit l'attribut data-category="apple" de la carte
// card.dataset.category  "apple" const category = card.dataset.category;
if (filter === 'tous' || category === filter) {
card.style.display = 'flex';  // Réaffiche (flex = style original) card.style.opacity = '1';    // Opacité totale (100%) } else {
card.style.display = 'none';  // Cache — retire du flux visuel }
});
}
Le lien entre filtre.js et product_grid.js : c'est product_grid.js qui écrit data-category="apple" sur chaque carte quand il la génère. filtre.js se contente de lire ce que product_grid.js a écrit. Les deux fichiers communiquent via le HTML, sans se parler directement.

Pourquoi retire-t-on "active" de TOUS avant d'en ajouter un ?

Le piège : Si on ajoutait seulement active au nouvel onglet sans retirer l'ancien, on se retrouverait avec deux onglets actifs en même temps visuellement. En retirant d'abord de tous, on repart toujours d'une ardoise vierge avant d'en activer un seul.

Sans l'étape "ardoise vierge" : si on faisait juste android.add('active'), on aurait à la fois "Tous .active" et "Android .active" deux onglets surlignés en même temps. L'interface serait incohérente.

Pourquoi function et pas une arrow function ?

Arrow function (() =>) et function se comportent différemment avec this.

Dans une function classique à l'intérieur d'un addEventListener, this = l'élément qui a reçu le clic (l'onglet). C'est ce qu'on veut.

Dans une arrow function () =>, this hérite du contexte parent et ne désigne pas l'onglet cliqué. On ne pourrait donc pas utiliser this.classList.add('active') ni this.dataset.filter.
javascript 
tabs.forEach(tab => {

// Ici "this" = l'onglet sur lequel on a cliqué
// Grâce à "function" (pas une arrow function) // Étape 1 : efface le marquage actif sur tous les onglets tabs.forEach(t => t.classList.remove('active'));
// Étape 2 : marque l'onglet cliqué this.classList.add('active');
// Étape 3 : lit la valeur du filtre et applique const filter = this.dataset.filter;  // ex: "apple" filterProducts(filter);

});

04 — MutationObserver

Surveiller les nouvelles cartes

Il y a un problème de timing. filtre.js démarre au même moment que product_grid.js. Mais product_grid.js n'injecte les cartes qu'après avoir chargé la base de données (ce qui prend un peu de temps).

Le problème sans observer

Scénario problématique

1. filtre.js démarre aucune carte dans la page.
2. L'utilisateur clique "Apple" filterProducts('apple') 0 carte trouvée, rien ne se passe.
3. product_grid.js injecte les cartes elles s'affichent toutes, même les Android, en ignorant le filtre actif.

La solution avec l'observer

Comment le MutationObserver résout ça

Il surveille en permanence #products-grid. Dès que product_grid.js y injecte des cartes, l'observer se déclenche et appelle immédiatement filterProducts avec le filtre actif. Résultat : le filtre est respecté même si les cartes arrivent en retard.

Flux du MutationObserver
Cherche l'élément #products-grid dans la page
La grille existe ?
NON
Observer non créé — fin silencieuse
OUI
Crée le MutationObserver avec une fonction callback
Observer commence à surveiller #products-grid
plus tard, product_grid.js injecte des cartes…
Callback se déclenche automatiquement
Lit l'onglet actif récupère son filtre
filterProducts(filtre_actuel) — re-filtre les nouvelles cartes

Chronologie complète de la coopération

Frise temporelle — filtre.js et product_grid.js en parallèle

Mais le filtre "Apple" est visuellement actif. Quand product_grid.js injecte les cartes au temps X, l'observer se déclenche et applique immédiatement "apple" sur les nouvelles cartes.

L'observer réagit immédiatement
Le navigateur notifie l'observer que des enfants ont été ajoutés dans #products-grid. L'observer lit l'onglet actif (ex: "Apple"), et appelle filterProducts('apple'). Seules les cartes Apple restent visibles — exactement comme l'utilisateur avait demandé.
javascript 
const productsGrid = document.getElementById('products-grid');
if (productsGrid) {

// Cette fonction est appelée automatiquement par le navigateur
// dès que des enfants sont ajoutés/supprimés dans productsGrid const activeTab = document.querySelector('.tab.active');
// .tab.active = élément qui a DEUX classes : "tab" et "active"

On aurait pu utiliser { subtree: true, attributes: true } pour surveiller tout — mais cela déclencherait l'observer des dizaines de fois à chaque petit changement. childList: true seul est suffisant et minimal.

filtre.js — Filtrage des produits par onglets ¶

filtre.js est le module le plus court du projet (55 lignes) mais son rôle est fondamental : il permet de filtrer les cartes produits par catégorie (Tous, Apple, Android, Accessoires…) via les onglets de la page d'accueil. Il ne crée rien, ne fetch rien — il cache et montre des éléments déjà présents dans le DOM.

Ce que fait ce fichier

1. Écouter les clics sur les onglets (.tab) et activer visuellement le bon onglet.
2. Appeler filterProducts(filter) pour cacher/montrer les cards selon leur data-category.
3. Surveiller via MutationObserver si product_grid.js injecte de nouvelles cards pour re-appliquer le filtre actif automatiquement.

Ce qu'il ne fait PAS

⚠️
filtre.js ne crée aucun élément HTML et ne fait aucun appel réseau. Il agit uniquement sur la visibilité des éléments déjà présents. Si les cards ne s'affichent pas, le problème vient de product_grid.js — pas de filtre.js.

Dépendances

Reçoit de : product_grid.js et catalogue.js — qui injectent les .product-card avec attribut data-category
Ne dépend pas de : PearTechDB, panier.js, ni aucun autre module — il est totalement autonome.

01 — Isolation et timing

IIFE + DOMContentLoaded — Pourquoi et comment ¶

Chaque fichier JS du projet est enveloppé dans deux couches : une IIFE et un listener DOMContentLoaded. Ces deux constructions répondent à des problèmes différents.

L'IIFE — Immediately Invoked Function Expression

javascript
(function() {
    'use strict'; // Mode strict activé
    // ... tout le code est ici ...
})(); // La fonction s'appelle immédiatement

L'IIFE crée un nouveau scope JS isolé. Toutes les variables déclarées à l'intérieur (const tabs, function filterProducts…) restent privées — elles n'existent pas dans window et ne peuvent pas entrer en conflit avec d'autres scripts.

⚠️
Sans IIFE, const tabs serait accessible depuis la console ou tout autre script. Si deux scripts déclaraient const tabs, le second lèverait une SyntaxError: Identifier already declared.
💡
'use strict' active le mode strict : interdit les variables non déclarées (x = 5 sans let/const → erreur), interdit this non défini, et permet aux moteurs JS d'optimiser le code. Les erreurs silencieuses deviennent des exceptions claires.

DOMContentLoaded — Attendre le HTML

javascript
document.addEventListener('DOMContentLoaded', function() {
    // Le HTML est intégralement parsé ici
    // Tous les éléments existent dans le DOM
    const tabs = document.querySelectorAll('.tab'); // fonctionne
});

DOMContentLoaded se déclenche quand le navigateur a fini de parser le HTML — mais avant que les images et CSS soient chargés. C'est le moment idéal : les éléments existent, on n'attend pas les ressources lourdes.

Ordre des événements navigateur
Script s'exécute — querySelectorAll('.tab') → 0 éléments (trop tôt)
↓
DOMContentLoaded — HTML 100% parsé
↓
correct
querySelectorAll('.tab') → retourne les onglets
↓
window.load — images+CSS chargés (trop tard)
🔍
Différence DOMContentLoaded vs window.load : DOMContentLoaded se déclenche quand le HTML est parsé (~50-200ms). window.load se déclenche quand TOUT est chargé — images, CSS, polices (2-5 secondes). Attendre load pour manipuler le DOM serait une perte de temps.

02 — Récupération des onglets

querySelectorAll('.tab') et le guard clause fail-fast ¶

Le module récupère les onglets et se protège immédiatement. Ce pattern — récupérer puis protéger — est utilisé dans tous les fichiers du projet.

javascript
const tabs = document.querySelectorAll('.tab'); // NodeList de tous les onglets
if (tabs.length === 0) return; // Pas d'onglets → on arrête tout

document.querySelectorAll() retourne une NodeList — similaire à un tableau, contenant tous les éléments correspondant au sélecteur CSS. Contrairement à querySelector() (retourne le premier ou null), querySelectorAll() retourne toujours une NodeList, même vide.

querySelector vs querySelectorAll
querySelector('.tab') → premier élément ou null
↓
1 résultat max
querySelectorAll('.tab') → NodeList de 0, 1 ou N éléments

La NodeList — pas un vrai tableau

Une NodeList supporte .forEach() et .length, mais pas .map(), .filter(), ou .find(). C'est pourquoi dans d'autres fichiers du projet on utilise Array.from(nodeList) pour la convertir. Ici .forEach() suffit.

💡
La NodeList de querySelectorAll() est statique — elle représente l'état du DOM au moment de l'appel. Si des onglets étaient ajoutés dynamiquement après, ils ne seraient pas dedans. C'est pourquoi le MutationObserver re-lit les onglets à chaque changement.

Le guard clause — fail fast

La ligne if (tabs.length === 0) return; est un guard clause : si la condition n'est pas remplie, on sort immédiatement. filtre.js est chargé sur toutes les pages du site. Sur la page FAQ, contact, profil — il n'y a pas d'onglets .tab. Sans ce return, le code s'exécuterait inutilement.

✓
Le return dans DOMContentLoaded ne quitte pas le script — il quitte uniquement ce callback. Le script reste chargé en mémoire, ce qui est correct car d'autres modules peuvent en avoir besoin sur la même page.

03 — La fonction centrale

filterProducts() — Logique de filtrage ligne par ligne ¶

C'est la seule vraie logique du fichier. Elle prend un filtre (chaîne de caractères) et met à jour la visibilité de chaque carte produit.

javascript
function filterProducts(filter) {
    // Récupère TOUTES les cards à chaque appel
    // Pourquoi ? product_grid.js peut en avoir ajouté depuis le dernier appel
    const productCards = document.querySelectorAll('.product-card');

    productCards.forEach(card => {
        const category = card.dataset.category; // ex: 'apple', 'android'

        if (filter === 'tous' || category === filter) {
            card.style.display = 'flex'; // IMPORTANT : flex, pas block !
            card.style.opacity = '1';   // Force la visibilité
        } else {
            card.style.display = 'none'; // Cache les cards qui ne correspondent pas
        }
    });
}

Pourquoi display:flex et pas display:block ?

Les cartes produits utilisent display:flex pour aligner leur contenu interne. Réafficher une card avec display:block casserait son layout. Règle : quand on montre un élément caché par display:none, on doit lui redonner son type d'affichage d'origine.

⚠️
Utiliser card.style.display = '' (chaîne vide) pour effacer le style inline est élégant mais risqué : certains reset CSS peuvent annuler le display:flex défini en CSS. L'assignation explicite 'flex' est plus fiable.

L'opérateur || et le court-circuit

La condition filter === 'tous' || category === filter utilise le court-circuit : si filter === 'tous' est true, la deuxième partie (category === filter) n'est jamais évaluée. Plus efficace qu'un if/else if séparé.

card.dataset.category — Lire les attributs data-*

L'attribut HTML data-category="apple" est accessible en JS via card.dataset.category. L'API dataset convertit automatiquement les attributs data-* en propriétés camelCase : data-product-id → dataset.productId. Si l'attribut est absent, dataset.category retourne undefined — la comparaison est false, la card serait cachée.

💡
Dans le HTML : <div class="product-card" data-category="apple">. En JS : card.dataset.category → "apple". C'est le mécanisme de communication entre HTML statique et JS dynamique — sans requête réseau, sans PearTechDB.

04 — Événements et surveillance

Listeners d'onglets, état initial et MutationObserver ¶

javascript
tabs.forEach(tab => {
    tab.addEventListener('click', function() {
        tabs.forEach(t => t.classList.remove('active')); // Désactive tout
        this.classList.add('active');                    // Active l'onglet cliqué
        const filter = this.dataset.filter;             // ex: 'apple', 'tous'
        filterProducts(filter);                         // Applique le filtre
        console.log('Filtre appliqué :', filter);
    });
});

L'ordre : d'abord tout désactiver, puis activer un

On retire 'active' de tous les onglets, PUIS on l'ajoute à l'onglet cliqué. Si on l'ajoutait d'abord puis retirait, l'onglet cliqué serait retiré dans la boucle forEach — il faudrait un test pour l'exclure. L'ordre actuel est simple et sans cas particulier.

Le mot-clé 'this' dans un listener

Dans un addEventListener, this fait référence à l'élément qui a reçu l'événement — ici l'onglet cliqué. C'est pourquoi le callback utilise function() et non une arrow function : les arrow functions héritent du this du contexte englobant, ce qui serait incorrect.

this dans function() vs arrow function
function() { this → l'onglet cliqué ✓ }
↓
() => { this → window ou undefined ✗ }

data-filter vs data-category — Deux attributs distincts

L'onglet a data-filter="apple". La card produit a data-category="apple". Les valeurs doivent être strictement identiques (même casse). "Apple" === "apple" est false en JS — une incohérence de casse casserait silencieusement le filtrage.

filterProducts('tous') — L'appel d'initialisation

javascript
filterProducts('tous'); // Affiche toutes les cards au chargement initial

Cet appel assure la cohérence dès le chargement. Il s'exécute après l'attachement des listeners — si une card avait un problème d'affichage initial, cet appel le corrigerait.

Le MutationObserver — Surveiller les ajouts futurs

javascript
const productsGrid = document.getElementById('products-grid');
if (productsGrid) {
    const gridObserver = new MutationObserver(function() {
        const activeTab = document.querySelector('.tab.active');
        const currentFilter = activeTab ? activeTab.dataset.filter : 'tous';
        filterProducts(currentFilter); // Re-applique le filtre sur les nouvelles cards
    });
    gridObserver.observe(productsGrid, { childList: true });
}

product_grid.js injecte les cartes produits après DOMContentLoaded. Sans le MutationObserver, si l'utilisateur avait sélectionné 'Apple' avant que les cards soient injectées, toutes les cards ignoreraient le filtre actif.

Sans MutationObserver — le bug
Utilisateur clique 'Apple' → filterProducts('apple') — 0 cards visibles
↓
product_grid.js injecte 30 cards — TOUTES visibles, filtre ignoré
↓
bug
Avec MutationObserver → filterProducts('apple') → corrige automatiquement
💡
L'option { childList: true } surveille uniquement l'ajout ou la suppression d'enfants directs. Suffisant car product_grid.js injecte les cards directement dans la grille via innerHTML.
🔍
L'opérateur ternaire activeTab ? activeTab.dataset.filter : 'tous' : si activeTab est null (aucun onglet actif), on utilise 'tous' par défaut. Sans cette sécurité, activeTab.dataset.filter sur null lèverait un TypeError.

Le code complet de filtre.js — 55 lignes commentées

javascript
(function() {
    'use strict';
    document.addEventListener('DOMContentLoaded', function() {

        // 1. Récupération
        const tabs = document.querySelectorAll('.tab');
        if (tabs.length === 0) return; // guard

        // 2. Listeners sur chaque onglet
        tabs.forEach(tab => {
            tab.addEventListener('click', function() {
                tabs.forEach(t => t.classList.remove('active'));
                this.classList.add('active');
                filterProducts(this.dataset.filter);
            });
        });

        // 3. Fonction centrale
        function filterProducts(filter) {
            document.querySelectorAll('.product-card').forEach(card => {
                const show = filter === 'tous' || card.dataset.category === filter;
                card.style.display = show ? 'flex' : 'none';
                if (show) card.style.opacity = '1';
            });
        }

        // 4. Appel initial
        filterProducts('tous');

        // 5. Surveillance des cards injectées dynamiquement
        const grid = document.getElementById('products-grid');
        if (grid) {
            new MutationObserver(() => {
                const active = document.querySelector('.tab.active');
                filterProducts(active ? active.dataset.filter : 'tous');
            }).observe(grid, { childList: true });
        }
    });
})();
✓
filtre.js est intentionnellement court. Sa force est sa minimalité : 1 fonction centrale, 1 observer, 0 dépendance externe. Il peut être compris en entier en 2 minutes et debuggé en 30 secondes.

04 — Ordre du code

Pourquoi chaque bloc est placé là où il est ¶

filtre.js est court mais précis : vérifier → logique → écouter → état initial → surveiller le futur. Le MutationObserver est systématiquement en dernier dans tout le projet.

1
Condition d'existencedocument.querySelectorAll('.tab') + if (tabs.length === 0) return
Le module n'a de sens que si des onglets existent. filtre.js est chargé sur toutes les pages. Sur les pages sans onglets, la NodeList est vide. L'avantage du return immédiat sur un if (tabs.length) { … tout le code … } : le corps principal n'est pas indenté, la lecture est plus fluide, et l'intention est claire — 'si pas d'onglets, on sort'.
Sans guard : tabs.forEach(…) s'exécute 0 fois (pas d'erreur), mais filterProducts('tous') est appelée quand même → querySelectorAll('.product-card') sur une page sans cards → boucle vide → CPU gaspillé. Pas d'erreur visible, mais du code inutile s'exécute.
↓ la logique avant les événements qui la déclenchent
2
Logique de filtragefunction filterProducts(filter) { … }
Définie avant les listeners qui l'appellent — le 'quoi' avant le 'quand'. filterProducts() est le cœur fonctionnel : elle cache/montre les .product-card en comparant leur dataset.category au filtre actif. La placer avant les listeners suit la convention : on comprend d'abord ce que fait le système, puis on voit quand il se déclenche. Elle est hoistée, donc techniquement l'ordre n'a pas d'importance.
Lit : tabs (pour l'onglet actif), .product-card (éléments à filtrer). Appelée par : listeners des onglets (étape 3), filterProducts('tous') (étape 4), MutationObserver (étape 5).
↓ connecter l'UI à la logique
3
Listeners ongletstabs.forEach(tab => tab.addEventListener('click', …))
Après filterProducts() — ce bloc connecte les onglets à la logique de filtrage. Chaque clic : (1) désactive tous les onglets (forEach removeClass active), (2) active celui cliqué (this.classList.add active), (3) appelle filterProducts(filter) avec la valeur du data-filter du bouton. L'ordre interne est impératif : d'abord réinitialiser tous les états, puis en activer un seul.
↓ état initial après configuration des interactions
4
État initialfilterProducts('tous') — affichage initial de toutes les cards
Après les listeners — configure l'état de départ après avoir configuré les changements d'état. Cet appel affiche toutes les cards au premier chargement (onglet 'Tous' actif par défaut). Pourrait techniquement être avant les listeners (pas de dépendance directe), mais l'ordre choisi suit la convention : configurer les règles du jeu, puis définir l'état de départ.
L'état initial est défini après les handlers qui le modifieront — on configure d'abord comment le système réagit, puis on le place dans son état de départ.
↓ surveiller les futures injections de cards (MutationObserver toujours en dernier)
5
Surveillance futurenew MutationObserver(…).observe(productsGrid, { childList: true })
En tout dernier — surveille les ajouts futurs, pas l'état présent. Sans ce MutationObserver, les cards injectées par product_grid.js ou catalogue.js après le chargement initial ignoreraient le filtre actif — elles s'afficheraient toutes même si l'onglet 'Apple' était sélectionné. Il re-applique le filtre actif à chaque injection de nouvelles cards.
Appelle filterProducts(currentFilter) où currentFilter est lu depuis l'onglet actuellement actif via querySelector('.tab.active')?.dataset.filter.

Concepts et termes techniques utilisés dans ce fichier ¶

Définitions des termes importants rencontrés dans cette documentation.

Document Object Model
Représentation en mémoire de la page HTML sous forme d'arbre. Chaque balise devient un nœud. JavaScript modifie cet arbre via getElementById(), querySelector(), innerHTML. Modifier le DOM = modifier ce que l'utilisateur voit en temps réel.
Immediately Invoked Function Expression
Fonction qui s'exécute immédiatement à sa déclaration : (function() { ... })(). Crée un scope isolé — tout ce qui est déclaré dedans reste privé et ne pollue pas window. Tous les fichiers JS du projet utilisent ce pattern.
DOMContentLoaded
Événement déclenché quand le HTML est parsé et le DOM prêt — avant images et CSS. Moment idéal pour initialiser les modules JS. Distinct de window.load qui attend tous les assets réseau.
dataset
Interface qui expose les attributs data-* d'un élément. el.dataset.productId lit data-product-id. Permet de stocker des données métier directement dans le HTML sans variables globales. La conversion camelCase est automatique : data-product-id → dataset.productId.
MutationObserver
API qui observe les modifications du DOM en temps réel et appelle un callback quand des nœuds sont ajoutés ou supprimés. Plus efficace que poller le DOM avec setInterval. Utilisé pour détecter les cards produit injectées dynamiquement.
classList
Interface de manipulation des classes CSS. add(), remove(), toggle(), contains(). Plus fiable que modifier className comme une string. toggle('active', condition) ajoute ou retire selon un booléen.

Filtres multi-critères : état, combinaison et URL ¶

💡
Analogie — les filtres comme un tamis multicouche : Chaque filtre actif est un tamis. Les produits passent en cascade — d'abord le tamis "catégorie", puis "prix", puis "marque". Seuls les produits qui passent tous les tamis arrivent à l'affichage. Ajouter un filtre, c'est ajouter un tamis dans la chaîne.
💡
Analogie — l'objet filtres actifs : L'objet activeFilters est comme un formulaire de commande. Chaque case cochée ajoute une ligne. Quand tu soumets, le système ne regarde que les lignes remplies. Si aucune ligne, tous les produits passent.
🚨 PIÈGE — Appliquer les filtres séquentiellement en modifiant le DOM
Ne jamais cacher/montrer les éléments DOM un par un pour filtrer. Toujours filtrer le tableau de données JS (PearTechDB.products), puis régénérer tout le HTML en une seule injection. Le DOM est une sortie, pas un filtre.
🚨 PIÈGE — Réinitialiser tous les filtres quand un seul change
Chaque modification de filtre doit mettre à jour seulement la clé concernée dans activeFilters et relancer l'affichage. Pas tout réinitialiser. L'utilisateur a défini les autres filtres pour une raison — les préserver.
PRO TIP
Sérialise les filtres actifs dans l'URL (?cat=audio&max=200) avec URLSearchParams. L'utilisateur peut partager un lien avec ses filtres appliqués, et le retour arrière restaure l'état.
PRO TIP
Affiche un compteur de résultats mis à jour en temps réel : "12 produits". Si 0 résultat, affiche un message et une suggestion de réinitialiser les filtres. Ne laisse jamais l'utilisateur face à une page vide sans explication.

activeFilters

Objet unique centralisant tous les filtres actifs. La seule source de vérité de l'état de filtrage.

every() pour AND

filters.every(f => f(product)) — un produit passe si tous les filtres retournent true.

Données → DOM

Filtrer JS d'abord, régénérer le HTML ensuite. Jamais l'inverse.

URL state

Filtres persistés en URL. Partageables, historisables, restaurables au rechargement.

        if (tabs.length === 0) return; // Pas d'onglets sur cette page : on arrête

        // ── Fonction de filtrage des cards ────────────────────────

        function filterProducts(filter) {
            const productCards = document.querySelectorAll('.product-card'); // Toutes les cards de la grille
            productCards.forEach(card => {
                const category = card.dataset.category; // Catégorie de la card (ex: "apple", "android")
                if (filter === 'tous' || category === filter) {
                    // L'onglet "tous" accepte toutes les catégories
                    card.style.display = 'flex'; // Réaffiche la card (flex car les cards utilisent flex)
                    card.style.opacity = '1';    // Assure la visibilité complète
                } else {
                    card.style.display = 'none'; // Cache les cards qui ne correspondent pas au filtre
                }
            });
        }

        // ── Événements sur les onglets ────────────────────────────

        tabs.forEach(tab => {
            tab.addEventListener('click', function() {
                tabs.forEach(t => t.classList.remove('active')); // Désactive tous les onglets
                this.classList.add('active');                    // Active l'onglet cliqué
                const filter = this.dataset.filter;             // Récupère le filtre ("tous", "apple"...)
                filterProducts(filter);                         // Applique le filtre
                console.log('Filtre appliqué :', filter);
            });
        });

        filterProducts('tous'); // Affiche toutes les cards au chargement initial

        // ── MutationObserver : re-applique le filtre actif si la grille est rechargée ──
        // Nécessaire car product_grid.js peut injecter des cards après le chargement

        const productsGrid = document.getElementById('products-grid');
        if (productsGrid) {
            const gridObserver = new MutationObserver(function() {
                const activeTab = document.querySelector('.tab.active'); // Onglet actuellement actif
                const currentFilter = activeTab ? activeTab.dataset.filter : 'tous'; // Filtre courant
                filterProducts(currentFilter); // Re-applique le filtre sur les nouvelles cards
            });
            gridObserver.observe(productsGrid, { childList: true }); // Surveille les enfants directs de la grille
        }
    });
})();

③ Application des filtres — every() pour le AND logique

javascript
function applyFilters(products) {
  // On construit un tableau de fonctions-filtres actives.
  const filters = [];

  if (activeFilters.category) {
    filters.push(p => p.category === activeFilters.category);
  }
  if (activeFilters.minPrice > 0) {
    filters.push(p => p.price >= activeFilters.minPrice);
  }
  if (activeFilters.maxPrice < Infinity) {
    filters.push(p => p.price <= activeFilters.maxPrice);
  }
  if (activeFilters.inStock) {
    filters.push(p => p.stock > 0);
  }
  // Seulement les filtres ACTIFS sont dans le tableau.
  // Si aucun filtre : filters = [] → every sur [] → true → tous les produits passent.

  return products.filter(p =>
    filters.every(fn => fn(p))
  );
  // every(fn => fn(p)) : un produit passe si CHAQUE filtre retourne true.
  // C'est le AND logique : catégorie ET prix ET stock ET …
  //
  // Exemple : filters = [catFn, priceFn]
  // p = {category:'audio', price:150, stock:5}
  //   catFn(p)   → true  (c'est de l'audio)
  //   priceFn(p) → true  (150 <= maxPrice)
  //   every → true → produit inclus ✓
}

④ Sérialisation des filtres dans l'URL

javascript
function updateURL() {
  const params = new URLSearchParams();
  // URLSearchParams gère l'encodage des valeurs automatiquement.
  // Pas besoin de encodeURIComponent manuellement.

  if (activeFilters.category) params.set('cat', activeFilters.category);
  if (activeFilters.maxPrice < Infinity) params.set('max', activeFilters.maxPrice);
  if (activeFilters.inStock) params.set('stock', '1');
  if (activeFilters.search) params.set('q', activeFilters.search);

  const query = params.toString();
  // Ex : 'cat=audio&max=500&stock=1'

  history.pushState({}, '', query ? `?${query}` : location.pathname);
  // pushState : modifie l'URL sans recharger la page.
  // L'utilisateur peut copier l'URL, la partager, ou naviguer en arrière.
  // location.pathname (ex: '/catalogue.html') efface les paramètres si aucun filtre.
}

function restoreFromURL() {
  const params = new URLSearchParams(location.search);
  // location.search = '?cat=audio&max=500' (la partie après ?)
  // URLSearchParams le parse automatiquement.

  if (params.has('cat'))   activeFilters.category = params.get('cat');
  if (params.has('max'))   activeFilters.maxPrice  = +params.get('max');
  if (params.has('stock')) activeFilters.inStock   = true;
  // + devant params.get : convertit string → number ('500' → 500)

  renderCatalogue(); // Applique les filtres restaurés
  syncUIWithFilters(); // Coche les cases et curseurs correspondants
}
