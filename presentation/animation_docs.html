Comprendre
animation.js

Ce fichier gère 4 effets visuels présents sur toutes les pages : l'apparition progressive des cartes au scroll (scroll reveal), l'ombre du header, l'agrandissement de la barre de recherche au clic, et le défilement fluide vers les ancres. Il utilise deux APIs modernes du navigateur : IntersectionObserver et MutationObserver.

animation.js · 4 effets · 2 observers · toutes les pages · ~130 lignes

📖 Termes nouveaux dans ce fichier
IntersectionObserver
API navigateur qui surveille si un élément HTML est visible dans la fenêtre. Plus efficace qu'écouter l'événement scroll et calculer les positions manuellement. Le callback est déclenché uniquement quand l'élément entre ou sort du viewport.
MutationObserver
API navigateur qui surveille les modifications du DOM. Ici il détecte quand de nouvelles cartes produits sont ajoutées dynamiquement par JavaScript (product_grid.js, catalogue.js) pour leur appliquer l'animation d'entrée.
threshold: 0.1
Seuil de déclenchement de l'IntersectionObserver. 0.1 = l'animation se déclenche quand 10% de l'élément est visible. 0 = dès qu'un pixel est visible. 1 = seulement quand tout l'élément est visible.
rootMargin
Marge virtuelle autour de la zone observée. '0px 0px -50px 0px' = la zone visible est réduite de 50px en bas. Concrètement, l'animation se déclenche 50px avant que l'élément atteigne le bord visible de l'écran.
getBoundingClientRect()
Méthode qui retourne la position et les dimensions d'un élément par rapport à la fenêtre. rect.top = distance entre le haut de l'élément et le haut de la fenêtre. Utilisé pour savoir si un élément est déjà visible au chargement.
dataset.scrollObserved
Attribut HTML data-scroll-observed utilisé comme flag. Une fois qu'une carte a été prise en charge par observeCard(), ce flag empêche de la traiter deux fois si le MutationObserver la détecte à nouveau.
childList: true, subtree: true
Options du MutationObserver. childList = surveille les ajouts/suppressions d'enfants directs. subtree = surveille aussi tous les descendants. Nécessaire pour détecter les cartes peu importe leur niveau d'imbrication.
01 — Scroll Reveal

IntersectionObserver — animer l'entrée dans le viewport

IntersectionObserver — comment il détecte l'entrée dans le viewport
viewport (ce que l'utilisateur voit)
✅ .reveal.visible — déjà animé
✅ .reveal.visible — déjà animé
↓ scroll vers le bas ↓
⚡ .reveal — entre dans le viewport → classList.add('visible')
↓ pas encore visible ↓
🔲 .reveal — en attente (opacity:0, translateY)
threshold: 0.1 → déclenche quand 10% de l'élément est visible · observer.unobserve(el) → ne re-déclenche pas
🔭
Analogie : L'IntersectionObserver est comme un vigile posté à l'entrée d'une salle. Au lieu de regarder en permanence si quelqu'un arrive (ce que ferait addEventListener('scroll', ...)), il attend sagement et est notifié automatiquement quand un élément passe la porte (entre dans le viewport).
javascript
const observer = new IntersectionObserver(function(entries) {
    entries.forEach(entry => {
        if (entry.isIntersecting) {         // L'élément vient d'entrer dans le viewport
            entry.target.style.opacity    = '1';
            entry.target.style.transform  = 'translateY(0)'; // Annule le décalage vertical initial
        }
    });
}, {
    threshold:  0.1,                       // 10% visible → déclenche
    rootMargin: '0px 0px -50px 0px'        // Zone visible réduite de 50px en bas
});
Comportement de rootMargin: '0px 0px -50px 0px'
Sans rootMargin (threshold:0.1)
L'animation se déclenche quand la carte atteint le tout bas de l'écran. L'utilisateur ne voit l'animation qu'une fraction de seconde avant que la carte soit complètement visible — peu naturel.
Avec rootMargin: -50px
L'animation se déclenche 50px avant que la carte atteigne le bas visible. La carte a déjà commencé à apparaître quand l'utilisateur l'aperçoit — effet plus doux et naturel.

observeCard() — la logique de décision

javascript
function observeCard(card) {
    if (card.dataset.scrollObserved) return; // Déjà traitée → ignore (idempotence)
    card.dataset.scrollObserved = '1';       // Marque comme traitée

    const rect      = card.getBoundingClientRect();
    const isVisible = rect.top < window.innerHeight && rect.bottom > 0;
    // isVisible = la carte est actuellement visible dans la fenêtre

    if (isVisible) {
        // Déjà visible au chargement : afficher directement sans animation
        card.style.opacity   = '1';
        card.style.transform = 'translateY(0)';
    } else {
        // Hors écran : préparer l'état initial + observer pour l'animation d'entrée
        card.style.opacity    = '0';
        card.style.transform  = 'translateY(20px)'; // 20px plus bas que sa position finale
        card.style.transition = 'opacity 0.5s ease, transform 0.5s ease'; // Transition fluide
        observer.observe(card); // L'observer commence à surveiller cet élément
    }
}
États d'une carte au fil du scroll
État 0 — Carte hors écran, non encore observée
↓
observeCard() appelée
État 1 — opacity:0 · translateY(20px) · transition 0.5s
Invisible et légèrement décalée vers le bas
↓
L'utilisateur scrolle · 10% de la carte devient visible
IntersectionObserver déclenche isIntersecting = true
↓
opacity: '1' · translateY(0) — transition CSS anime le passage
État final — Carte visible et à sa position normale

observer.observe({childList: true, subtree: true}) → détecte ajouts/suppressions dans tout le DOM

Un problème : les cartes produits sont générées après le chargement de la page par product_grid.js ou catalogue.js. Au moment où animation.js s'exécute, ces cartes n'existent pas encore. Le MutationObserver résout ce problème.

javascript
const mutationObs = new MutationObserver(function(mutations) {
    mutations.forEach(function(mutation) {
        mutation.addedNodes.forEach(function(node) {
            if (node.nodeType !== 1) return; // Ignore les nœuds texte (type 3) et commentaires (type 8)

            // Cas 1 : le nœud ajouté EST directement une card
            if (node.classList && node.classList.contains('product-card')) {
                observeCard(node);
            }
            // Cas 2 : le nœud est un conteneur (ex: innerHTML réécrit)
            // → cherche toutes les .product-card à l'intérieur
            node.querySelectorAll && node.querySelectorAll('.product-card').forEach(observeCard);
        });
    });
});

const productsGrid = document.getElementById('products-grid');
if (productsGrid) {
    mutationObs.observe(productsGrid, { childList: true, subtree: true });
}
Pourquoi deux cas (node direct vs conteneur) ?
Cas 1 — Node direct
Quand un script fait productsGrid.appendChild(card). Le nœud ajouté est directement la card. node.classList.contains('product-card') est vrai.
Cas 2 — Conteneur
Quand un script fait productsGrid.innerHTML = '...'. Tout le HTML est remplacé d'un coup. Le MutationObserver détecte 1 gros nœud → il faut chercher les cards en profondeur avec querySelectorAll.
💡catalogue.js et product_grid.js utilisent innerHTML = html (cas 2). Sans le querySelectorAll, les cartes seraient générées mais aucune n'aurait l'animation d'entrée.
03 — Header Shadow

Ombre sous le header au scroll

Header Shadow — seuil de déclenchement scrollY
scrollY = 0
Header sans ombre — classList.remove('scrolled')
scrollY = 49px
Toujours sans ombre
scrollY = 50px
Seuil atteint → classList.add('scrolled') → box-shadow CSS active
scrollY = 200px
Ombre persistante — pas de re-déclenchement au-delà du seuil
scrollY < 50
Retour en haut → classList.remove('scrolled') → ombre disparaît
window.scrollY > 50 ? header.classList.add('scrolled') : header.classList.remove('scrolled') — évalué à chaque événement scroll
javascript
window.addEventListener('scroll', function() {
    if (window.pageYOffset > 50) {
        header.style.boxShadow = '0 4px 12px rgba(0,0,0,0.3)'; // Ombre portée
    } else {
        header.style.boxShadow = 'none'; // Pas d'ombre en haut de page
    }
});
Pourquoi 50px et pas 0 ?
Avec un seuil à 0px, l'ombre apparaît dès qu'on bouge la molette d'un pixel. Cela crée un flash visuel désagréable. Avec 50px, la transition est imperceptible car l'utilisateur a déjà quitté le haut de page et ne voit plus le changement brusque.

Agrandissement de la barre au focus — avant / après
ÉTAT NORMAL (blur)
🔍
width: 200px
ÉTAT FOCUS (actif)
🔍
width: 300px
focus → classList.add('expanded') + style.width = '300px' · blur → classList.remove('expanded') + style.width = '200px'
javascript
searchInput.addEventListener('focus', function() {
    searchBar.style.transform = 'scale(1.02)'; // +2% de taille
});

searchInput.addEventListener('blur', function() {
    setTimeout(function() {
        searchBar.style.transform = 'scale(1)'; // Retour à la taille normale
    }, 200); // Délai 200ms : évite un flash si l'utilisateur clique sur un résultat
});
💡
Pourquoi le délai de 200ms sur blur ? Si l'utilisateur tape dans la barre puis clique directement sur un résultat de recherche, blur se déclenche avant le clic sur le résultat. Sans délai, la barre retrouve sa taille et "saute" visuellement au moment où l'utilisateur clique. Les 200ms laissent le temps au clic de se traiter d'abord.
05 — Smooth Scroll

Défilement fluide vers les ancres

Smooth Scroll — comment l'ancre est calculée
Clic <a href="#section-faq">
→
e.preventDefault()
href.slice(1) → "section-faq"
→
getElementById("section-faq")
target trouvé
→
scrollIntoView({ behavior: 'smooth' })
target null (href="#inexistant") → querySelector retourne null → return immédiat, pas d'erreur
javascript
document.querySelectorAll('a[href^="#"]').forEach(anchor => {
    anchor.addEventListener('click', function(e) {
        e.preventDefault(); // Annule le comportement par défaut (saut brutal)
        const target = document.querySelector(this.getAttribute('href'));
        if (target) {
            target.scrollIntoView({ behavior: 'smooth', block: 'start' });
        }
    });
});
// Sélecteur CSS 'a[href^="#"]' : tous les liens dont href COMMENCE PAR "#"
// Cela cible href="#accueil" mais pas href="https://..." ni href="page.html"
ℹ️
En 2024, scroll-behavior: smooth dans le CSS global suffit souvent à remplacer ce code. Mais animation.js garantit que ça fonctionne même sur les anciens navigateurs et donne un contrôle plus fin (ex: ajouter un délai ou un callback).

Dépendances
N'importe quel fichier JS — animation.js est entièrement autonome. Il ne lit pas PearTechDB, ne touche pas localStorage, n'émet aucun événement personnalisé. Il observe le DOM passivement.
Performance
IntersectionObserver est bien plus performant qu'écouter scroll + calculer des positions à chaque frame. Le navigateur gère la détection en dehors du thread principal.
