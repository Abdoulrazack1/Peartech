Comprendre
main.js

Le fichier d'entrée global du projet. Il ne gère aucune UI directement — son rôle est d'écouter les événements custom des autres modules, d'exposer des utilitaires globaux via window.PearTech (thème, notifications), et de logguer les performances au démarrage.

main.js · ~80 lignes · CONFIG + 3 event listeners + window.PearTech · toutes les pages
00 — Vue d'ensemble

Le chef d'orchestre — sans jouer d'instrument

Analogie : main.js est comme un chef d'orchestre. Il n'joue aucune note lui-même, mais il écoute tous les musiciens (les autres modules), maintient le tempo (logs), et met à disposition des outils communs (window.PearTech) que tout le monde peut utiliser.
Les 4 responsabilités de main.js
1 — CONFIG
Objet centralisé : siteName, version, debug. Un seul endroit à modifier pour changer le comportement global.
2 — Écoute d'événements custom
filterChanged · productAddedToCart · productSelected. Centralise les logs de tous les modules.
3 — window.PearTech
API globale : notify(), getTheme(), setTheme(). Accessible depuis tous les scripts.
4 — Performance
performance.now() après window.load — mesure le temps de chargement complet de la page.

Intérêt : si la clé 'peartech-cart' doit changer, on la modifie une seule fois dans CONFIG plutôt que dans chaque fichier.
javascript
const CONFIG = {
siteName: 'PearTech', // Nom affiché dans les logs version:  '1.0.0',   // Version sémantique (major.minor.patch) debug:    true //  Mettre false en production pour désactiver tous les logs
};
// Chaque console.log est conditionné par CONFIG.debug :
// if (CONFIG.debug) { console.log(...) }
//  En production : changer debug: true  false  silence total
Pourquoi un objet CONFIG ? Sans lui, si on veut désactiver tous les logs en production, il faudrait chercher et supprimer chaque console.log un par un dans le fichier. Avec CONFIG, un seul changement (debug: false) suffit pour tout éteindre d'un coup.

Le log de démarrage stylisé

javascript
console.log('%c' + CONFIG.siteName + ' v' + CONFIG.version,
'color: #3B82F6; font-size: 16px; font-weight: bold');
// Le préfixe %c indique à la console d'appliquer les styles CSS du 2e argument.
// Résultat : "PearTech v1.0.0" affiché en bleu gras 16px dans la console DevTools.

Couplage zéro. product_grid.js ne sait pas que main.js existe. Il émet juste un signal sur document. main.js est le seul à écouter — mais n'importe quel autre module pourrait s'y abonner sans toucher à product_grid.js.

Les modules du projet communiquent entre eux via des CustomEvents sur le document. main.js les écoute tous pour les logger. C'est le pattern "publish/subscribe" appliqué au DOM.

javascript — 3 événements écoutés
// Émis par filtre.js quand un filtre change
document.addEventListener('filterChanged', function(e) {
if (CONFIG.debug) console.log('Filtre changé:', e.detail.filter);
});
// Émis par panier.js quand un produit est ajouté au panier
document.addEventListener('productAddedToCart', function(e) {
if (CONFIG.debug) console.log('Produit ajouté:', e.detail.name);
});
// Émis par product_grid.js ou catalogue.js quand un produit est cliqué
document.addEventListener('productSelected', function(e) {
if (CONFIG.debug) console.log('Produit sélectionné:', e.detail.name);
});
Pattern Publish / Subscribe — pourquoi c'est élégant
Sans CustomEvent (couplage fort)
panier.js importerait main.js
dépendance circulaire
modification de l'un casse l'autre
impossible de les tester séparément
Avec CustomEvent (couplage nul)
panier.js dispatch un événement
ne connaît pas main.js
main.js écoute sans connaître panier.js
chaque module est indépendant
03 — API globale

window.PearTech — accessible depuis tous les scripts

window.PearTech — l'API publique de main.js
PearTech.notify(msg, type)
Dispatche showNotification sur document. type = 'success' | 'error' | 'info'. Aucun écouteur actuellement infrastructure pour une future UI toast globale.
PearTech.getTheme()
Lit body.classList.contains('dark-mode') retourne 'dark' ou 'light'. Utilisable depuis la console dev pour connaître le thème actuel.
PearTech.setTheme(theme)
Toggle body.classList + localStorage.setItem('peartech-theme', theme). Ne met pas à jour l'icône ni aria-pressed (contrairement à darkmode.js) pour usage programmatique uniquement.

En assignant un objet à window.PearTech, n'importe quel autre script de la page peut appeler window.PearTech.notify() ou PearTech.setTheme('dark') sans importer quoi que ce soit.

javascript
window.PearTech = {
version: CONFIG.version,
// Publie un événement de notification (pour une future toast UI) notify: function(message, type) {
const event = new CustomEvent('showNotification', {
detail: { message: message, type: type || 'info' }
});
document.dispatchEvent(event);
},
// Retourne 'dark' ou 'light' selon la classe CSS sur body getTheme: function() {
return document.body.classList.contains('dark-mode') ? 'dark' : 'light';
},
// Change le thème ET le persiste en localStorage setTheme: function(theme) {
if (theme === 'dark') {
document.body.classList.add('dark-mode');
localStorage.setItem('peartech-theme', 'dark');
} else {
document.body.classList.remove('dark-mode');
localStorage.setItem('peartech-theme', 'light');
}
}
};
Pourquoi window.PearTech plutôt qu'une simple fonction ?
Namespace
Toutes les fonctions groupées sous un seul nom évitent les collisions avec d'autres librairies
Accessibilité globale
N'importe quel script peut appeler PearTech.notify() sans import ni dépendance
Extensibilité
Facile d'ajouter de nouvelles méthodes sans casser l'existant
window.PearTech.notify() dispatche un showNotification CustomEvent mais aucun listener ne l'écoute encore dans le projet — c'est une infrastructure prête à l'emploi pour une future implémentation de toasts (messages de notification visuels). Le pattern est déjà en place, il suffit d'ajouter l'écouteur.
04 — Mesure de performance

performance.now() — mesurer le chargement réel

DOMContentLoaded vs window.load — chronologie
t = 0ms
Navigation premier octet HTML reçu
~200-400ms
DOMContentLoaded — HTML parsé, scripts init()
...
Images, CSS externes, polices chargées...
~800-2000ms
window.load performance.now() loadTime loggé
CONFIG.debug = false en prod ces logs n'apparaissent pas. Mettre à true en développement pour mesurer les régressions de perf.
javascript
if (CONFIG.debug) {
window.addEventListener('load', function() {
const loadTime = performance.now();
console.log('Page chargée en', Math.round(loadTime), 'ms');
});
}
DOMContentLoaded vs window.load — la différence
DOMContentLoaded
HTML parsé et DOM construit. Les images et CSS ne sont pas encore chargés. C'est là que la plupart des scripts s'initialisent.
window.load
Tout est chargé : HTML, CSS, images, scripts. performance.now() ici donne le temps total de chargement de la page.
performance.now() retourne le nombre de millisecondes écoulées depuis que la page a commencé à se charger (navigation timing). C'est plus précis que Date.now() car il utilise une horloge haute résolution (sous-milliseconde).

Glossaire

Concepts et termes techniques utilisés dans ce fichier

Définitions des termes importants rencontrés dans cette documentation.

Document Object Model
Représentation en mémoire de la page HTML sous forme d'arbre. Chaque balise devient un nœud. JavaScript modifie cet arbre via getElementById(), querySelector(), innerHTML. Modifier le DOM = modifier ce que l'utilisateur voit en temps réel.
Immediately Invoked Function Expression
Fonction qui s'exécute immédiatement à sa déclaration : (function() { ... })(). Crée un scope isolé — tout ce qui est déclaré dedans reste privé et ne pollue pas window. Tous les fichiers JS du projet utilisent ce pattern.
DOMContentLoaded
Événement déclenché quand le HTML est parsé et le DOM prêt — avant images et CSS. Moment idéal pour initialiser les modules JS. Distinct de window.load qui attend tous les assets réseau.
CustomEvent
Événement personnalisé émis par le code : document.dispatchEvent(new CustomEvent('nom', {detail: données})). Permet à des modules de communiquer sans se connaître — couplage zéro. L'abonné écoute via document.addEventListener('nom', fn).
localStorage
Stockage clé-valeur persistant dans le navigateur — survit à la fermeture. Capacité ~5 Mo. Clé/valeur uniquement en string : les objets passent par JSON.stringify() avant stockage et JSON.parse() à la lecture.
Scope (portée)
Contexte d'accessibilité d'une variable. let/const → block scope (limité au bloc {}). var → function scope (remonte à la fonction). Les IIFE créent un scope fermé pour éviter les collisions de noms entre modules.
Callback
Fonction passée comme argument pour être appelée plus tard. addEventListener('click', callback) — le callback est invoqué lors du clic. Mécanisme de base de l'asynchronisme JS — le code continue, le callback est exécuté quand l'événement se produit.
