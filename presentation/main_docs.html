01 — Objet de configuration

CONFIG — un seul endroit pour tout paramétrer ¶

L'objet CONFIG est déclaré à l'extérieur de DOMContentLoaded mais à l'intérieur de l'IIFE. Il contient les paramètres globaux du site et contrôle le comportement de debug.

La structure de CONFIG

javascript
const CONFIG = {
    siteName: 'PearTech', // Nom du site — utilisé dans les logs console
    version:  '1.0.0',   // Version sémantique — visible dans la console DevTools
    debug:    true        // true = logs actifs | false = silencieux (production)
};

Pourquoi CONFIG est hors de DOMContentLoaded

CONFIG ne dépend d'aucun élément DOM — c'est de la configuration pure. Le placer dans le scope IIFE (hors de DOMContentLoaded) signifie qu'il est disponible immédiatement à l'exécution du script, avant même que le DOM soit prêt. Si d'autres fonctions définies dans l'IIFE voulaient l'utiliser dans des traitements hors-DOM, elles y auraient accès.

Le flag debug — Couper les logs en production

javascript
if (CONFIG.debug) {
    console.log('%c' + CONFIG.siteName + ' v' + CONFIG.version,
        'color: #3B82F6; font-size: 16px; font-weight: bold');
    console.log('Modules chargés: ...');
}

Tous les console.log() sont conditionnés par if (CONFIG.debug). En production, mettre debug: false suffit à silencer tous les logs sans toucher au reste du code. C'est le pattern de feature flag le plus simple : un booléen central contrôle une famille de comportements.

console.log avec style CSS

La syntaxe console.log('%c' + texte, 'color: #3B82F6; ...') permet d'appliquer du CSS à la sortie console. %c est un spécificateur de format : le second argument est la chaîne CSS appliquée au texte qui suit. Résultat : le nom du site s'affiche en bleu gras dans DevTools — utile pour identifier rapidement les logs de PearTech parmi d'autres.

💡
Le pattern CONFIG.debug est plus propre que de commenter/décommenter des console.log manuellement. En équipe, passer debug: false avant un déploiement est une opération unique et traçable dans le code source.

Couplage zéro. product_grid.js ne sait pas que main.js existe. Il émet juste un signal sur document. main.js est le seul à écouter — mais n'importe quel autre module pourrait s'y abonner sans toucher à product_grid.js.

Les modules du projet communiquent entre eux via des CustomEvents sur le document. main.js les écoute tous pour les logger. C'est le pattern "publish/subscribe" appliqué au DOM.

03 — API globale

window.PearTech — accessible depuis tous les scripts ¶

window.PearTech — l'API publique de main.js
PearTech.notify(msg, type)
Dispatche showNotification sur document. type = 'success' | 'error' | 'info'. Aucun écouteur actuellement infrastructure pour une future UI toast globale.
PearTech.getTheme()
Lit body.classList.contains('dark-mode') retourne 'dark' ou 'light'. Utilisable depuis la console dev pour connaître le thème actuel.
PearTech.setTheme(theme)
Toggle body.classList + localStorage.setItem('peartech-theme', theme). Ne met pas à jour l'icône ni aria-pressed (contrairement à darkmode.js) pour usage programmatique uniquement.

En assignant un objet à window.PearTech, n'importe quel autre script de la page peut appeler window.PearTech.notify() ou PearTech.setTheme('dark') sans importer quoi que ce soit.

javascript
window.PearTech = {
version: CONFIG.version,
// Publie un événement de notification (pour une future toast UI) notify: function(message, type) {
const event = new CustomEvent('showNotification', {
detail: { message: message, type: type || 'info' }
});
document.dispatchEvent(event);
},
// Retourne 'dark' ou 'light' selon la classe CSS sur body getTheme: function() {
return document.body.classList.contains('dark-mode') ? 'dark' : 'light';
},
// Change le thème ET le persiste en localStorage setTheme: function(theme) {
if (theme === 'dark') {
document.body.classList.add('dark-mode');
localStorage.setItem('peartech-theme', 'dark');
} else {
document.body.classList.remove('dark-mode');
localStorage.setItem('peartech-theme', 'light');
}
}
};
Pourquoi window.PearTech plutôt qu'une simple fonction ?
Namespace
Toutes les fonctions groupées sous un seul nom évitent les collisions avec d'autres librairies
Accessibilité globale
N'importe quel script peut appeler PearTech.notify() sans import ni dépendance
Extensibilité
Facile d'ajouter de nouvelles méthodes sans casser l'existant
window.PearTech.notify() dispatche un showNotification CustomEvent mais aucun listener ne l'écoute encore dans le projet — c'est une infrastructure prête à l'emploi pour une future implémentation de toasts (messages de notification visuels). Le pattern est déjà en place, il suffit d'ajouter l'écouteur.

04 — Mesure de performance

performance.now() — mesurer le chargement réel ¶

DOMContentLoaded vs window.load — chronologie
t = 0ms
Navigation premier octet HTML reçu
~200-400ms
DOMContentLoaded — HTML parsé, scripts init()
...
Images, CSS externes, polices chargées...
~800-2000ms
window.load performance.now() loadTime loggé
CONFIG.debug = false en prod ces logs n'apparaissent pas. Mettre à true en développement pour mesurer les régressions de perf.
javascript
if (CONFIG.debug) {
window.addEventListener('load', function() {
const loadTime = performance.now();
console.log('Page chargée en', Math.round(loadTime), 'ms');
});
}
DOMContentLoaded vs window.load — la différence
DOMContentLoaded
HTML parsé et DOM construit. Les images et CSS ne sont pas encore chargés. C'est là que la plupart des scripts s'initialisent.
window.load
Tout est chargé : HTML, CSS, images, scripts. performance.now() ici donne le temps total de chargement de la page.
performance.now() retourne le nombre de millisecondes écoulées depuis que la page a commencé à se charger (navigation timing). C'est plus précis que Date.now() car il utilise une horloge haute résolution (sous-milliseconde).

performance.now() — Temps réel de chargement complet

javascript
if (CONFIG.debug) {
    window.addEventListener('load', function() {
        const loadTime = performance.now(); // ms depuis le début de la navigation
        console.log('Page chargée en', Math.round(loadTime), 'ms');
    });
}

performance.now() vs Date.now()

performance.now() retourne le temps depuis le début de la navigation (le moment où l'utilisateur a demandé la page) en millisecondes avec précision sub-milliseconde. Date.now() retourne le timestamp Unix courant — inutilisable pour mesurer une durée de chargement car il faudrait connaître le timestamp de début de navigation.

window.load vs DOMContentLoaded pour la mesure

La mesure est dans window.addEventListener('load') et non DOMContentLoaded. load se déclenche quand tout est chargé : HTML, CSS, images, polices, scripts. C'est le temps perçu par l'utilisateur — il ne peut pas utiliser la page tant que les images ne sont pas affichées. DOMContentLoaded mesurerait uniquement le parsing HTML, pas le chargement des ressources.

💡
En production, performance.now() dans window.load donne typiquement 500ms-3s selon la connexion et le nombre d'images. C'est une mesure de Performance Web (Core Web Vitals — LCP, TTI). Les outils comme Lighthouse font exactement cette mesure mais avec des conditions réseau simulées. Le log de main.js est la version développeur sans tooling externe.

Interprétation des métriques performance.now()

< 500ms
Excellent — cache chaud ou page légère
500ms-1s

1s-2s

05 — Récapitulatif

Vue d'ensemble — CONFIG, CustomEvents et window.PearTech ¶

main.js est le chef d'orchestre passif du projet : il écoute sans imposer, expose sans forcer. Voici l'intégralité de son architecture.

Flux d'exécution complet

javascript
// ─ IIFE (scope isolé immédiat) ──────────────
const CONFIG = { siteName, version, debug };

document.addEventListener('DOMContentLoaded', function() {

    // 1. Log de démarrage (si debug)
    console.log('%cPearTech v1.0.0', 'color:#3B82F6...');

    // 2. 3 listeners d'événements custom globaux
    document.addEventListener('filterChanged',       e => log(e.detail.filter));
    document.addEventListener('productAddedToCart',  e => log(e.detail.name));
    document.addEventListener('productSelected',     e => log(e.detail.name));

    // 3. Exposition de l'API globale
    window.PearTech = {
        version,
        notify(message, type) { ... },   // Émetteur de notifications
        getTheme() { ... },              // Lecteur du thème actuel
        setTheme(theme) { ... }          // Setter du thème + localStorage
    };

    // 4. Mesure du temps de chargement (si debug)
    window.addEventListener('load', () =>
        console.log('Page chargée en', Math.round(performance.now()), 'ms'));
});

window.PearTech — L'API publique du site

window.PearTech expose 3 utilitaires accessibles depuis n'importe quel autre script ou depuis la console : notify() — publie un CustomEvent showNotification (pour une future modale toast). getTheme() — lit la présence de la classe dark-mode sur body et retourne 'dark' ou 'light'. setTheme(theme) — applique la classe et sauvegarde dans localStorage (duplique la logique de darkmode.js intentionnellement pour que les autres scripts aient un accès unifié).

performance.now() — Mesure du temps de chargement

performance.now() retourne le nombre de millisecondes écoulées depuis le début de la navigation (le moment où l'utilisateur a demandé la page). Appelé dans window.addEventListener('load', ...) — qui se déclenche quand tout est chargé (images, scripts, CSS, polices) — il donne le temps de chargement total perçu par l'utilisateur.

✓
main.js illustre le principe de couplage faible via les CustomEvents. Au lieu d'importer filtre.js dans main.js (couplage fort), main.js écoute l'événement filterChanged. filtre.js et main.js ne se connaissent pas — ils communiquent par messages. Ajouter un nouveau module qui émet filterChanged ne nécessite aucune modification de main.js.

05 — Ordre du code

Pourquoi chaque bloc est placé là où il est ¶

main.js sépare clairement la configuration statique (hors DOM) des comportements dynamiques (dans DOMContentLoaded). À l'intérieur, l'ordre est : diagnostic → communication interne → interface publique → monitoring.

1
Config statique — HORS DOMconst CONFIG = { siteName, version, debug }
Déclarée hors de DOMContentLoaded car elle ne dépend pas du DOM. CONFIG contient des valeurs fixes connues à la compilation (nom du site, version, mode debug). La placer dans DOMContentLoaded la rendrait disponible quelques dizaines de millisecondes plus tard — inutilement. Elle est accessible dès le chargement du script, avant même que le HTML soit parsé.
Utilisée dans : log de démarrage (étape 2), window.PearTech.version (étape 4), CONFIG.debug && console.log dans les listeners (étape 3).
↓ tout le reste attend DOMContentLoaded
2
Diagnostic démarrageconsole.log('[PearTech] DOM prêt — main.js opérationnel')
Premier signal dans la console — confirme que DOMContentLoaded s'est déclenché. En cas de problème (script bloquant, erreur dans un module précédent), ce log confirme que main.js a bien atteint DOMContentLoaded. Le placer en premier garantit qu'il apparaît avant les logs des listeners — chronologie de lecture cohérente avec la chronologie d'exécution.
↓ écouter les événements des autres modules avant d'exposer l'API
3
Listeners customaddEventListener('filterChanged') + ('productAddedToCart') + ('productSelected')
Avant window.PearTech — écouter d'abord, exposer ensuite. Ces listeners s'abonnent aux événements émis par filtre.js, panier.js et catalogue.js. Ils doivent être attachés avant que d'autres scripts puissent émettre ces événements — ce qui se produit juste après DOMContentLoaded. Les placer avant l'exposition de l'API reflète l'ordre de priorité : communication interne avant interface publique.
Ces événements sont émis par : filtre.js (filterChanged), panier.js (productAddedToCart), catalogue.js (productSelected).
↓ exposer l'API après que tout est configuré en interne
4
API publiquewindow.PearTech = { version, notify(), getTheme(), setTheme() }
Après les listeners internes — l'API publique est publiée une fois tout configuré. window.PearTech est utilisable par d'autres scripts et depuis la console navigateur. setTheme() appelle document.body.classList — le body doit exister (garanti par DOMContentLoaded). Exposer après les listeners évite une fenêtre d'incohérence où l'API serait accessible mais les listeners internes pas encore branchés.
setTheme() utilise document.body. notify() émet sur document. getTheme() lit document.body.classList. Les trois nécessitent le DOM.
↓ monitoring passif — après tout le reste
5
Mesure performancewindow.addEventListener('load', …) — performance.now()
En tout dernier — surveillance passive, pas un comportement. window.load se déclenche après le chargement complet (HTML + CSS + images + scripts — tout). Il est distinct de DOMContentLoaded (HTML seulement). Ce listener est un outil de monitoring optionnel — il ne fait que lire et logger une durée. Son absence ne bloquerait rien d'autre.
La mesure de performance est toujours en dernier : elle ne doit pas influencer ce qu'elle observe. Placer un gros calcul avant ce log fausserait le temps mesuré.
Règle générale de ce fichier

La seule variable hors DOMContentLoaded est CONFIG car elle est statique. Tout le reste (listeners, API, mesure) attend DOMContentLoaded. L'API window.PearTech est publiée après les listeners internes pour garantir la cohérence de l'état.

Concepts et termes techniques utilisés dans ce fichier ¶

Définitions des termes importants rencontrés dans cette documentation.

Document Object Model
Représentation en mémoire de la page HTML sous forme d'arbre. Chaque balise devient un nœud. JavaScript modifie cet arbre via getElementById(), querySelector(), innerHTML. Modifier le DOM = modifier ce que l'utilisateur voit en temps réel.
Immediately Invoked Function Expression
Fonction qui s'exécute immédiatement à sa déclaration : (function() { ... })(). Crée un scope isolé — tout ce qui est déclaré dedans reste privé et ne pollue pas window. Tous les fichiers JS du projet utilisent ce pattern.
DOMContentLoaded
Événement déclenché quand le HTML est parsé et le DOM prêt — avant images et CSS. Moment idéal pour initialiser les modules JS. Distinct de window.load qui attend tous les assets réseau.
CustomEvent
Événement personnalisé émis par le code : document.dispatchEvent(new CustomEvent('nom', {detail: données})). Permet à des modules de communiquer sans se connaître — couplage zéro. L'abonné écoute via document.addEventListener('nom', fn).
localStorage
Stockage clé-valeur persistant dans le navigateur — survit à la fermeture. Capacité ~5 Mo. Clé/valeur uniquement en string : les objets passent par JSON.stringify() avant stockage et JSON.parse() à la lecture.
Scope (portée)
Contexte d'accessibilité d'une variable. let/const → block scope (limité au bloc {}). var → function scope (remonte à la fonction). Les IIFE créent un scope fermé pour éviter les collisions de noms entre modules.
Callback
Fonction passée comme argument pour être appelée plus tard. addEventListener('click', callback) — le callback est invoqué lors du clic. Mécanisme de base de l'asynchronisme JS — le code continue, le callback est exécuté quand l'événement se produit.

main.js : orchestrateur et point d'entrée de l'application ¶

💡
Analogie — le chef de cuisine : main.js est le chef de cuisine qui ne cuisine pas lui-même mais s'assure que tout est prêt : les ingrédients (données), les cuisiniers (modules), le four chaud (DOM prêt). Il lance tout dans le bon ordre et gère les urgences si quelque chose manque.
💡
Analogie — DOMContentLoaded : Attendre DOMContentLoaded avant d'initialiser, c'est attendre que le décor soit en place avant que les acteurs entrent sur scène. Sans ça, les acteurs cherchent des éléments qui n'existent pas encore.
🚨 PIÈGE — Initialiser les modules avant DOMContentLoaded
Si un module appelle document.getElementById() avant que le HTML soit parsé, il retourne null. Toute l'initialisation doit se faire dans le callback de DOMContentLoaded (ou en bas du body, juste avant </body>).
🚨 PIÈGE — Ordre d'initialisation des modules interdépendants
Si cart.js dépend de data.js, data.js doit être chargé (balise script) et initialisé avant cart.js. Dans main.js, initialise dans l'ordre des dépendances : données → utilitaires → modules UI → interactions.
PRO TIP
Centralise la gestion d'erreurs globale dans main.js avec window.addEventListener('error', handler) et window.addEventListener('unhandledrejection', handler). Un seul endroit pour logguer ou afficher les erreurs inattendues.
PRO TIP
Utilise console.group('PearTech init') / console.groupEnd() dans main.js pour regrouper tous les logs d'initialisation. En production, remplace par un flag DEBUG = false qui désactive tous les logs.

// ============================================
// main.js - Point d'entrée principal
// Initialise les événements globaux, expose des
// utilitaires communs et log les performances
// ============================================

(function() {
    'use strict'; // Mode strict activé

    // ── Configuration globale ─────────────────────────────────────

    const CONFIG = {
        siteName: 'PearTech', // Nom du site (utilisé dans les logs)
        version: '1.0.0',     // Version de l'application
        debug: true           // Active les logs de debug (mettre false en production)
    };

    document.addEventListener('DOMContentLoaded', function() {

        // ── Log de démarrage ──────────────────────────────────────

        if (CONFIG.debug) {
            // Affiche le nom du site en bleu gras dans la console
            console.log('%c' + CONFIG.siteName + ' v' + CONFIG.version,
                'color: #3B82F6; font-size: 16px; font-weight: bold');
            // Liste tous les modules qui doivent être chargés
            console.log('Modules chargés:');
            console.log('✓ theme.js - Dark/Light mode');
            console.log('✓ search.js - Recherche produits');
            console.log('✓ animations.js - Scroll reveal');
            console.log('✓ filters.js - Filtres et tabs');
            console.log('✓ cart.js - Panier');
        }

        // ── Écoute des événements custom globaux ──────────────────
        // Ces événements sont émis par les autres modules via dispatchEvent

        // Écoute les changements de filtre (émis par filtre.js)
        document.addEventListener('filterChanged', function(e) {
            if (CONFIG.debug) {
                console.log('Filtre changé:', e.detail.filter); // Log le nouveau filtre appliqué
            }
        });

        // Écoute les ajouts au panier (émis par panier.js)
        document.addEventListener('productAddedToCart', function(e) {
            if (CONFIG.debug) {
                console.log('Produit ajouté:', e.detail.name); // Log le nom du produit ajouté
            }
        });

        // Écoute les sélections de produit (émis par product_grid.js ou catalogue.js)
        document.addEventListener('productSelected', function(e) {
            if (CONFIG.debug) {
                console.log('Produit sélectionné:', e.detail.name);
            }
        });

        // ── Utilitaires globaux accessibles partout ───────────────
        // window.PearTech permet à tous les scripts d'accéder à ces fonctions

        window.PearTech = {
            version: CONFIG.version, // Expose la version de l'app

            // Déclenche un événement de notification (ex: pour une future modale toast)
            notify: function(message, type) {
                const event = new CustomEvent('showNotification', {
                    detail: { message: message, type: type || 'info' } // Type par défaut : 'info'
                });
                document.dispatchEvent(event); // Publie l'événement sur le document
            },

            // Retourne le thème actuel ('dark' ou 'light')
            getTheme: function() {
                return document.body.classList.contains('dark-mode') ? 'dark' : 'light';
            },

            // Change le thème et le sauvegarde dans localStorage
            setTheme: function(theme) {
                if (theme === 'dark') {
                    document.body.classList.add('dark-mode');           // Active le dark mode
                    localStorage.setItem('peartech-theme', 'dark');    // Sauvegarde la préférence
                } else {
                    document.body.classList.remove('dark-mode');        // Désactive le dark mode
                    localStorage.setItem('peartech-theme', 'light');   // Sauvegarde la préférence
                }
            }
        };

        // ── Mesure du temps de chargement ─────────────────────────

        if (CONFIG.debug) {
            window.addEventListener('load', function() { // Après le chargement complet (images, scripts...)
                const loadTime = performance.now(); // Temps écoulé depuis la navigation en ms
                console.log('Page chargée en', Math.round(loadTime), 'ms');
            });
        }
    });

})(); // Fin de l'IIFE

function load(key) {
  try {
    const raw = localStorage.getItem(key);
    return raw ? JSON.parse(raw) : null;
  } catch { return null; } // corrompu → null
