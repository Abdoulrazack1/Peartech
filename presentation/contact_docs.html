Comprendre
contact.js

Ce fichier gère le formulaire de contact de page_contact.html. Il intercepte la soumission, valide tous les champs (y compris la checkbox RGPD et le format email), simule un envoi avec un délai d'une seconde, puis affiche un message de retour coloré selon le résultat.

contact.js · ~70 lignes · 3 fonctions · page_contact.html · aucune dépendance
00 — Avant de commencer

Validation, simulation réseau et feedback visuel

📬
Analogie : contact.js est comme un employé de guichet : il vérifie d'abord que tous les documents sont présents et correctement remplis, affiche un panneau "traitement en cours", puis confirme que le message est bien parti (en réalité, rien n'est envoyé côté serveur — c'est une simulation).
Structure du fichier — 3 blocs
1 — Écoute
addEventListener('submit') intercepte le formulaire avant son envoi natif
2 — Validation
Vérifie champs vides + checkbox + format email. Arrête si erreur.
3 — Feedback
Affiche "envoi…" puis "envoyé !" après 1s. Reset du formulaire.
01 — Validation des champs

Deux niveaux de vérification

javascript
contactForm.addEventListener('submit', function(e) {
    e.preventDefault(); // Bloque le rechargement de page

    const name    = document.getElementById('name').value.trim();
    const email   = document.getElementById('email').value.trim();
    const subject = document.getElementById('subject').value.trim();
    const message = document.getElementById('message').value.trim();
    const consent = document.getElementById('consent').checked; // Boolean

    // Niveau 1 : tous les champs présents + checkbox cochée
    if (!name || !email || !subject || !message || !consent) {
        showMessage('Veuillez remplir tous les champs...', 'error');
        return; // Arrêt immédiat
    }

    // Niveau 2 : format email valide
    if (!validateEmail(email)) {
        showMessage('Veuillez saisir une adresse email valide.', 'error');
        return;
    }

    // ▶ Si on arrive ici, tout est valide → envoi simulé
});

validateEmail() — la regex expliquée caractère par caractère

javascript
function validateEmail(email) {
    const re = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
    return re.test(email); // true si valide
}
// ^           → début de la chaîne
// [^\s@]+     → 1+ caractères qui ne sont ni espace ni @   (la partie avant @)
// @           → le caractère @ obligatoire
// [^\s@]+     → 1+ caractères (le domaine)
// \.          → un point littéral (échappé avec \ car . seul = "n'importe quel char")
// [^\s@]+     → 1+ caractères (l'extension : fr, com, org…)
// $           → fin de la chaîne
/ Tests de la regex
[email protected]✓Format correct
[email protected]✓Sous-domaine autorisé
jean [email protected]✗Espace interdit avant @
jeanATdupont.fr✗Pas de @
jean@dupont✗Pas de point dans le domaine
💡Cette regex est une validation de format, pas d'existence. [email protected] passerait la validation. En production, une vérification côté serveur (ou un envoi de confirmation) est nécessaire pour valider l'existence réelle.

02 — Envoi simulé

setTimeout — simuler un délai réseau

Simulation vs production — ce qui change
Version simulée (actuelle)
showMessage('Envoi...', 'info')
↓ setTimeout 1000ms
showMessage('Envoyé', 'success')
contactForm.reset()
Version production
showMessage('Envoi...', 'info')
↓ fetch('/api/contact', {method:'POST',...})
.then(r => r.ok ?
showMessage('Envoyé', 'success')
: showMessage('Erreur', 'error')
La structure showMessage → action asynchrone → showMessage(résultat) est identique dans les deux cas. Seul le mécanisme d'attente change : setTimeout remplacé par fetch().then().
javascript
showMessage('Envoi en cours...', 'info'); // Feedback immédiat

setTimeout(() => {
    showMessage(
        'Votre message a bien été envoyé. Nous vous répondrons dans les plus brefs délais.',
        'success'
    );
    contactForm.reset(); // Vide tous les champs d'un coup
}, 1000); // 1000ms = 1 seconde
💡
En production : Le setTimeout serait remplacé par un fetch() vers une API backend. Le showMessage "Envoi en cours..." resterait, et le message "succès" ou "erreur" serait affiché selon la réponse du serveur. La structure resterait identique — seule la partie "simulation" change.

Les 3 états visuels de formMessage

formMessage.className = "form-message " + type → une seule ligne écrase la classe précédente. Pas de classList.add/remove à gérer.
javascript
function showMessage(text, type) {
    if (!formMessage) return; // Sécurité si élément absent
    formMessage.textContent = text;
    formMessage.className   = `form-message ${type}`; // 'success', 'error', ou 'info'
    formMessage.scrollIntoView({ behavior: 'smooth', block: 'nearest' });
}
Les 3 types de messages et leurs classes CSS
'error'
Champ manquant ou email invalide. Fond rouge atténué.
'info'
"Envoi en cours..." Fond bleu atténué. Transitoire.
'success'
Message envoyé avec succès. Fond vert atténué.
💡formMessage.className = `form-message ${type}` remplace toute la classe à chaque appel. Cela évite d'accumuler les classes error info success après plusieurs soumissions. À chaque call, l'état visuel est proprement réinitialisé.
