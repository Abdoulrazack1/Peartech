Validation, simulation réseau et feedback visuel ¶

Analogie : contact.js est comme un employé de guichet : il vérifie d'abord que tous les documents sont présents et correctement remplis, affiche un panneau "traitement en cours", puis confirme que le message est bien parti (en réalité, rien n'est envoyé côté serveur — c'est une simulation).
Structure du fichier — 3 blocs
1 — Écoute
addEventListener('submit') intercepte le formulaire avant son envoi natif
2 — Validation
Vérifie champs vides + checkbox + format email. Arrête si erreur.
3 — Feedback
Affiche "envoi…" puis "envoyé !" après 1s. Reset du formulaire.

validateEmail() — la regex expliquée caractère par caractère

javascript
function validateEmail(email) {
const re = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
return re.test(email); // true si valide
}
// ^            début de la chaîne
// [^\s@]+      1+ caractères qui ne sont ni espace ni @   (la partie avant @)
// @            le caractère @ obligatoire
// [^\s@]+      1+ caractères (le domaine)
// \.           un point littéral (échappé avec \ car . seul = "n'importe quel char")
// [^\s@]+      1+ caractères (l'extension : fr, com, org…)
// $            fin de la chaîne

formMessage.className = `form-message ${type}` remplace toute la classe à chaque appel. Cela évite d'accumuler les classes error info success après plusieurs soumissions. À chaque call, l'état visuel est proprement réinitialisé.

Les 3 états de showMessage()

La fonction showMessage(text, type) gère les 3 états du feedback formulaire : 'error' (rouge — champ manquant ou email invalide), 'success' (vert — message envoyé), 'info' (neutre — envoi en cours). Le type contrôle la classe CSS appliquée à formMessage.

javascript
function showMessage(text, type) {
    formMessage.textContent = text;          // texte brut (pas innerHTML — protection XSS)
    formMessage.className   = 'form-message ' + type; // ex: 'form-message error'
    formMessage.style.display = 'block';     // rend visible

    if (type === 'success') {
        setTimeout(() => {
            formMessage.style.display = 'none'; // auto-masque après 5s
        }, 5000);
    }
}

textContent vs innerHTML pour le message

formMessage.textContent = text écrit le texte comme du contenu brut — pas d'interprétation HTML. Si text contenait <script> ou , ce serait affiché comme du texte littéral, pas exécuté ni rendu. C'est la protection minimale contre le XSS : ne jamais écrire de données utilisateur via innerHTML.

Auto-masquage uniquement pour 'success'

Le setTimeout de 5s ne s'applique qu'au type 'success'. Les erreurs restent affichées jusqu'à la prochaine soumission — si on les masquait automatiquement, l'utilisateur pourrait rater le message d'erreur sur un formulaire long. Le succès en revanche peut disparaître seul : la confirmation a été vue.

💡
formMessage.className = 'form-message ' + type réécrit la classe entière plutôt qu'ajouter/retirer. Si on avait fait classList.add(type) sans retirer l'ancien type, un message 'error' suivi d'un 'success' aurait les deux classes — le style CSS serait imprévisible. Réécrire la classe entière garantit un état propre.

04 — Récapitulatif

Vue d'ensemble — tout le fichier en un schéma ¶

contact.js est un module de formulaire complet : collecte, double validation, envoi simulé et feedback. Voici l'intégralité de son architecture.

Flux d'exécution complet

javascript
// 1. Récupération + guard
const contactForm = document.getElementById('contact-form');
if (!contactForm) return;

// 2. addEventListener('submit') — point d'entrée unique
contactForm.addEventListener('submit', function(e) {
    e.preventDefault();

    // 3. Collecte des valeurs des 5 champs
    const { name, email, subject, message, consent } = getFieldValues();

    // 4. Validation niveau 1 : tous les champs remplis
    if (!name || !email || !subject || !message || !consent) {
        showMessage('Remplissez tous les champs...', 'error');
        return;
    }

    // 5. Validation niveau 2 : format email
    if (!validateEmail(email)) {
        showMessage('Email invalide', 'error');
        return;
    }

    // 6. Envoi simulé avec délai
    showMessage('Envoi en cours...', 'info');
    setTimeout(() => {
        showMessage('Message envoyé !', 'success');
        contactForm.reset();
    }, 1000);
});

// Fonctions auxiliaires (hoistées)
function showMessage(text, type) { ... }
function validateEmail(email) { ... }

e.preventDefault() sur submit — Interception du formulaire

Le comportement natif d'un formulaire HTML à la soumission est d'envoyer une requête HTTP (GET ou POST) vers l'URL définie dans action, ce qui recharge la page. e.preventDefault() annule ce comportement entièrement pour prendre le contrôle en JS.

La cascade de validations

Les deux niveaux de validation sont intentionnellement séparés et ordonnés : simple avant complexe. La vérification !name || !email || ... (niveau 1) est en O(1) — simple comparaison. La validation regex de l'email (niveau 2) est plus coûteuse. Valider d'abord que le champ n'est pas vide, puis valider son format, évite d'exécuter la regex sur une chaîne vide.

contactForm.reset() — Vider le formulaire proprement

contactForm.reset() est une méthode native HTML qui réinitialise tous les champs du formulaire à leurs valeurs initiales (vides pour les inputs text, false pour les checkboxes). C'est plus fiable que de vider manuellement chaque champ avec input.value = '' — on n'a pas à connaître les noms de tous les champs.

💡
Le setTimeout(() => {...}, 1000) simule un délai réseau de 1 seconde. En production réelle, ce bloc serait remplacé par un fetch('https://api.example.com/contact', { method: 'POST', body: ... }) suivi d'un .then() pour gérer la réponse. La structure du code ne changerait pas — seul le contenu du setTimeout serait remplacé par la promesse fetch.

05 — Ordre du code

Pourquoi chaque bloc est placé là où il est ¶

contact.js exploite le hoisting JS pour placer les helpers en bas et le comportement principal en haut. La validation suit une cascade d'early returns : simple → complexe.

1
Récupération DOMgetElementById('contact-form') + getElementById('form-message')
Un seul accès DOM par élément, partagé dans tout le scope. contactForm est utilisé dans le guard clause, dans le listener submit, et indirectement dans showMessage(). formMessage est utilisé dans showMessage(). Les récupérer une fois en tête de scope économise N × 2 getElementById (où N = nombre de soumissions de formulaire). Plus lisible, plus efficace.
contactForm → guard clause (étape 2) + listener submit (étape 3). formMessage → showMessage() (étape 7).
↓ vérifier immédiatement après récupération
2
Sortie rapideif (!contactForm) { return; }
Pattern fail-fast — stoppe tout si le formulaire est absent. contact.js est chargé sur toutes les pages du projet. Sur les pages sans formulaire (accueil, catalogue, profil…), contactForm vaut null. Sans ce return, contactForm.addEventListener('submit', …) lèverait un TypeError qui pourrait faire crasher tout le JS de la page selon la configuration du gestionnaire d'erreurs global.
Sans guard : sur page_accueil.html (sans formulaire), contactForm.addEventListener → TypeError: Cannot read properties of null. Impact : tous les scripts qui s'exécutent après dans la même page peuvent être affectés.
↓ le comportement principal — hoisting permet de l'écrire avant les helpers
3
Écoute soumissioncontactForm.addEventListener('submit', function(e) { … })
Écrit avant showMessage() et validateEmail() grâce au hoisting. Ces deux fonctions sont des function declarations — le moteur JS les remonte (hoist) au sommet du scope avant toute exécution. Le listener peut donc les appeler même si elles sont physiquement plus bas dans le fichier. Ce choix de style place le 'quoi' (comportement principal) avant le 'comment' (détails des helpers).
Appelle showMessage() et validateEmail() — hoistées depuis les étapes 6 et 7.
↓ à l'intérieur du callback submit : lire avant valider
4
Collecte des valeursconst name = …value.trim() × 5 champs + checkbox
Toujours collecter avant de valider. On ne peut pas valider des données qu'on n'a pas encore lues. Les 5 valeurs sont lues et stockées dans des variables locales — évite de relire .value plusieurs fois pendant les validations. La méthode .trim() est appliquée immédiatement pour normaliser : les espaces de début/fin ne doivent pas passer la validation comme du contenu valide.
↓ validation en cascade : la moins coûteuse d'abord
5
Double validationif (!name || !email || …) → if (!validateEmail(email))
Deux niveaux dans l'ordre du moins coûteux au plus coûteux. (1) Vérification 'champs vides' : simple comparaison booléenne, quasi-instantanée, couvre le cas le plus fréquent d'erreur. (2) Validation email avec RegExp : légèrement plus coûteuse. Le premier if court-circuite le second dans 80% des cas d'erreur — on n'exécute la regex que si tous les champs sont remplis.
Validation en cascade (early return) : chaque vérification échoue vite et remonte un message précis. Le code de succès n'est atteint que si toutes les vérifications ont passé.
↓ envoi simulé — uniquement si tout est valide
6
Envoi simuléshowMessage('Envoi en cours…') + setTimeout(1000) → succès
Après toutes les validations — jamais avant. Ce bloc ne s'exécute que si les deux guard clauses de l'étape 5 ont été passés avec succès. Le message 'Envoi en cours…' donne un feedback immédiat à l'utilisateur. Le setTimeout(1000) simule un appel réseau — en production il serait remplacé par fetch('/api/contact', {method:'POST', body:formData}).
↓ helpers en bas — accessibles dès le début grâce au hoisting
7
Helpers hoistésfunction showMessage(text, type) + function validateEmail(email)
Écrits en dernier — accessibles en premier grâce au hoisting JS. Le moteur JS remonte toutes les function declarations au sommet de leur scope avant d'exécuter la première ligne. Écrire ces helpers en bas est un choix de style délibéré : le lecteur voit d'abord le comportement principal (le listener et les validations), puis les détails d'implémentation. C'est le pattern 'top-down' : du haut niveau vers le bas niveau.
showMessage() utilise formMessage récupéré à l'étape 1. validateEmail() est une fonction pure (pas de dépendance externe).
Règle générale de ce fichier

Le hoisting permet d'écrire le listener submit avant showMessage() et validateEmail(). C'est un choix de lisibilité : on voit d'abord le flux principal (écouter → collecter → valider → envoyer), puis les détails d'implémentation des helpers.

Concepts et termes techniques utilisés dans ce fichier ¶

Définitions des termes importants rencontrés dans cette documentation.

Document Object Model
Représentation en mémoire de la page HTML sous forme d'arbre. Chaque balise devient un nœud. JavaScript modifie cet arbre via getElementById(), querySelector(), innerHTML. Modifier le DOM = modifier ce que l'utilisateur voit en temps réel.
Immediately Invoked Function Expression
Fonction qui s'exécute immédiatement à sa déclaration : (function() { ... })(). Crée un scope isolé — tout ce qui est déclaré dedans reste privé et ne pollue pas window. Tous les fichiers JS du projet utilisent ce pattern.
DOMContentLoaded
Événement déclenché quand le HTML est parsé et le DOM prêt — avant images et CSS. Moment idéal pour initialiser les modules JS. Distinct de window.load qui attend tous les assets réseau.
Expression régulière (regex)
Pattern de correspondance de texte. Syntaxe : /pattern/. /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(str) valide un email. replace(/pattern/g, '') remplace toutes les occurrences. Utilisées ici pour validation email et nettoyage de chaînes.
preventDefault()
Annule le comportement natif du navigateur. e.preventDefault() sur un submit empêche le rechargement de page. Sur un lien # empêche le saut brutal. N'arrête pas la remontée de l'événement aux parents.
Callback
Fonction passée comme argument pour être appelée plus tard. addEventListener('click', callback) — le callback est invoqué lors du clic. Mécanisme de base de l'asynchronisme JS — le code continue, le callback est exécuté quand l'événement se produit.

Formulaire de contact : validation, feedback et sécurité ¶

💡
Analogie — la validation : Valider un formulaire avant envoi, c'est comme un contrôleur de billets à l'entrée du train. Il vérifie avant que le train parte — pas après. Si le billet est invalide, l'utilisateur reste sur le quai (la page) avec un message clair.
💡
Analogie — preventDefault : Sans e.preventDefault(), soumettre un formulaire, c'est comme appuyer sur un bouton éjectable dans un avion. La page se recharge immédiatement, effaçant tout. preventDefault() débranche le comportement par défaut et te laisse le contrôle.
🚨 PIÈGE — Valider uniquement côté client
La validation JS peut être contournée en désactivant JavaScript ou en envoyant une requête directement. La validation côté client est une UX amélioration — jamais une sécurité. Dans un vrai projet, le serveur doit valider à nouveau toutes les données reçues.
🚨 PIÈGE — Afficher le message de succès sans vider le formulaire
Après une soumission réussie, vider les champs du formulaire avec form.reset() ou en vidant chaque .value. Sinon l'utilisateur peut soumettre deux fois accidentellement en croyant que la première tentative avait échoué.
PRO TIP
Utilise l'attribut HTML novalidate sur le formulaire pour désactiver la validation native du navigateur et garder le contrôle total du feedback visuel (styles, messages d'erreur personnalisés).
PRO TIP
Valide en temps réel sur l'événement blur (perte de focus) plutôt que sur input. L'utilisateur voit l'erreur quand il quitte le champ, pas pendant qu'il tape — moins frustrant.

// contact.js - Formulaire de contact
// Gère la validation des champs et l'envoi simulé
// du message de contact
// ============================================

(function() {
    'use strict'; // Mode strict activé

    document.addEventListener('DOMContentLoaded', function() {

        // ── Récupération des éléments du formulaire ───────────────

        const contactForm = document.getElementById('contact-form'); // Le formulaire HTML
        const formMessage = document.getElementById('form-message'); // Zone d'affichage des messages de retour

        if (!contactForm) return; // Si le formulaire n'existe pas sur cette page, on arrête

        // ── Gestion de la soumission du formulaire ─────────────────

        contactForm.addEventListener('submit', function(e) {
            e.preventDefault(); // Empêche le rechargement de page lors de la soumission

            // ── Récupération des valeurs des champs ────────────────

            const name    = document.getElementById('name').value.trim();    // Nom complet (espaces retirés)
            const email   = document.getElementById('email').value.trim();   // Adresse email
            const subject = document.getElementById('subject').value.trim(); // Sujet du message
            const message = document.getElementById('message').value.trim(); // Corps du message
            const consent = document.getElementById('consent').checked;      // Checkbox RGPD cochée ou non

            // ── Validation : tous les champs doivent être remplis ──

            if (!name || !email || !subject || !message || !consent) {
                // Au moins un champ est vide ou la checkbox n'est pas cochée
                showMessage(
                    'Veuillez remplir tous les champs obligatoires et accepter le traitement des données.',
                    'error' // Type 'error' pour afficher en rouge
                );
                return; // Arrête la soumission
            }

            // ── Validation du format de l'email ───────────────────

            if (!validateEmail(email)) { // Vérifie que l'email a un format valide
                showMessage('Veuillez saisir une adresse email valide.', 'error');
                return;
            }

            // ── Envoi simulé avec délai ────────────────────────────
            // En production, remplacer par un vrai fetch() vers une API backend

            showMessage('Envoi en cours...', 'info'); // Message pendant le "chargement"

            setTimeout(() => { // Simule un délai réseau de 1 seconde
                showMessage(
                    'Votre message a bien été envoyé. Nous vous répondrons dans les plus brefs délais.',
                    'success' // Type 'success' pour afficher en vert
                );
                contactForm.reset(); // Vide tous les champs du formulaire après succès
            }, 1000); // 1000ms = 1 seconde de délai simulé
        });

        // ── Fonction d'affichage des messages de retour ───────────

        function showMessage(text, type) {
            if (!formMessage) return; // Sécurité : ne fait rien si l'élément n'existe pas
            formMessage.textContent = text; // Définit le texte du message
            formMessage.className = `form-message ${type}`; // Ajoute la classe CSS (success, error, info)
            // Fait défiler la page jusqu'au message pour que l'utilisateur le voie
            formMessage.scrollIntoView({ behavior: 'smooth', block: 'nearest' });
        }

        // ── Fonction de validation de l'email ─────────────────────

        function validateEmail(email) {
            // Expression régulière : vérifie la présence de @ et d'un domaine valide
            const re = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
            return re.test(email); // Retourne true si l'email est valide, false sinon
        }
    });
})(); // Fin de l'IIFE

① Validation champ par champ — retour d'erreur structuré

javascript
const validators = {
  name: (v) => {
    if (!v.trim()) return 'Le nom est requis';
    if (v.trim().length < 2) return 'Minimum 2 caractères';
    return null; // null = pas d'erreur
  },
  email: (v) => {
    if (!v.trim()) return 'L\'email est requis';
    if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(v)) return 'Email invalide';
    // Regex email minimaliste. Vérifie : texte@texte.texte
    // Pas parfaite (RFC 5322 est complexe) mais couvre 99% des cas.
    return null;
  },
  message: (v) => {
    if (!v.trim()) return 'Le message est requis';
    if (v.trim().length < 10) return 'Minimum 10 caractères';
    return null;
  }
};
// Objet de fonctions : chaque clé = nom du champ, valeur = fonction de validation.
// validators['email']('test') → 'Email invalide'
// validators['email']('a@b.com') → null (valide)

function validateAll(data) {
  const errors = {};
  let isValid  = true;

  Object.keys(validators).forEach(field => {
    const error = validators[field](data[field] || '');
    if (error) {
      errors[field] = error;
      isValid = false;
    }
  });

  return { isValid, errors };
  // Retourne { isValid: false, errors: { email: 'Email invalide' } }
  // ou { isValid: true, errors: {} } si tout va bien.
}
