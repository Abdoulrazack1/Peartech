Comprendre
confirmation.js

Ce fichier gère la page de confirmation de commande. Il lit les données de commande écrites par paiement.js dans sessionStorage, les injecte dans le DOM, nettoie le panier, puis supprime les données de session pour éviter qu'elles s'affichent à nouveau lors d'une prochaine visite.

04 — Nettoyage du panier

Vider le panier après confirmation ¶

Le nettoyage du panier est la dernière action de confirmation.js et la seule action irréversible : une fois le panier vidé, les données de commande ne peuvent pas être récupérées depuis le panier (elles sont déjà dans sessionStorage pour la confirmation).

Le code complet

javascript
// Utilise l'API centralisée PearTechCart si disponible
if (window.PearTechCart) {
    window.PearTechCart.clear(); // Vide le panier et met à jour localStorage
} else {
    // Fallback : remise à zéro directe du compteur
    localStorage.setItem('peartech-cart-count', '0');
}

// Met à jour le badge du header
const badge = document.getElementById('cart-count');
if (badge) {
    badge.textContent = '0';
    badge.style.display = 'none'; // Cache le badge quand le compteur est à 0
}

PearTechCart.clear() vs fallback localStorage

window.PearTechCart est défini dans panier.js. Il expose une API propre dont clear() qui vide le tableau cartItems ET met à jour localStorage ET met à jour le badge. Si panier.js n'est pas chargé sur cette page, le fallback localStorage.setItem('peartech-cart-count', '0') effectue uniquement la mise à zéro du compteur — suffisant pour que le badge disparaisse.

Pourquoi le badge est géré en plus de PearTechCart.clear()

En théorie, PearTechCart.clear() devrait mettre à jour le badge. En pratique, confirmation.js met à jour le badge explicitement en plus comme sécurité : si panier.js n'a pas correctement mis à jour le badge (ex: timing ou bug), la page de confirmation garantit que le header affiche bien 0 après la commande.

⚠️
Placer le nettoyage du panier après l'affichage des données (et non avant) est critique. Si PearTechCart.clear() était appelé avant les injections DOM et qu'une erreur survenait, l'utilisateur verrait sa commande confirmée mais le panier non vidé. L'ordre actuel garantit qu'on affiche d'abord, puis on nettoie.

05 — Récapitulatif

Vue d'ensemble — séquence complète ¶

confirmation.js est le script le plus simple de la chaîne d'achat : aucune interaction utilisateur, aucun event listener, aucun appel réseau. C'est un script de lecture et affichage pur.

La séquence complète du fichier

javascript
// 1. DOMContentLoaded
//    └── let orderData; (déclaration dans le bon scope)

// 2. sessionStorage.getItem('peartech-last-order')
//    ├── si trouvé → JSON.parse() + removeItem() (consommer les données)
//    └── si absent/corrompu → catch(e) → orderData reste undefined

// 3. if (!orderData) → fallback avec données fictives
//    └── garantit que la page ne s'affiche jamais vide

// 4. Injections DOM (x8 éléments) + forEach items → innerHTML
//    └── textContent pour les données simples, innerHTML pour la liste

// 5. Nettoyage : PearTechCart.clear() + badge → 0

Pourquoi removeItem() immédiatement après getItem()

La donnée peartech-last-order est consommée en une fois : lire puis supprimer immédiatement. Si l'utilisateur rafraîchit la page (F5), getItem() retourne null — le fallback s'affiche. C'est intentionnel : on ne veut pas qu'un rechargement de page re-affiche une ancienne commande comme si elle venait d'être passée.

textContent vs innerHTML — Sécurité XSS

Les injections simples (orderData.email, orderData.orderNumber…) utilisent .textContent et non .innerHTML. textContent traite la valeur comme du texte brut — si la valeur contient <script data-cfasync="false" src="/cdn-cgi/scripts/5c5dd728/cloudflare-static/email-decode.min.js"></script><script>alert(1)</script>, ce texte s'affiche littéralement sans exécution. C'est la protection de base contre les attaques XSS (Cross-Site Scripting).

confirmation.js est un exemple de script de rendu final d'une chaîne de traitement : paiement.js écrit, confirmation.js lit et consomme. Le pattern write → read → delete garantit qu'une commande ne peut être affichée qu'une seule fois, même en cas de rafraîchissement.

06 — Ordre du code

Pourquoi chaque bloc est placé là où il est ¶

confirmation.js suit la séquence lire → supprimer immédiatement → fallback → afficher → nettoyer en dernier. Les deux suppressions (removeItem sessionStorage à l'étape 3, clear() panier à l'étape 7) illustrent la même règle : supprimer au plus tard possible pour les données irremplaçables, au plus tôt possible pour les données temporaires.

1
Déclaration de portéelet orderData (sans valeur initiale)
Déclarée sans valeur pour être accessible dans tout le bloc DOMContentLoaded. orderData est lue dans le bloc if (stored), dans le bloc if (!orderData) (fallback), et dans toutes les injections DOM. En JS, let déclarée dans un if est hors scope pour les blocs suivants — la déclarer en tête de scope résout ce problème.
Si déclarée à l'intérieur du if (stored) { let orderData = … } : le fallback et les injections DOM ne pourraient pas y accéder → ReferenceError: orderData is not defined.
↓ lire la source de vérité avant d'afficher quoi que ce soit
2
Lecture sessionStoragesessionStorage.getItem('peartech-last-order') + JSON.parse dans try/catch
Source de vérité principale — doit être lue avant tout affichage. Le try/catch autour du JSON.parse est indispensable : une chaîne corrompue dans sessionStorage (données tronquées, quota dépassé) lèverait une SyntaxError non attrapée qui stopperait toute la page. La page de confirmation resterait blanche avec un panier non vidé.
Alimente orderData (étape 1). Si null/invalide → fallback (étape 4) prend le relais.
↓ supprimer immédiatement après lecture — action défensive
3
Suppression immédiatesessionStorage.removeItem('peartech-last-order') — juste après parse réussi
Supprimée au plus tôt, pas en fin de script. Si l'utilisateur appuie sur F5, la page se recharge. Sans cette suppression immédiate, la même commande serait réaffichée — l'utilisateur verrait une 'fausse' confirmation pour une commande déjà enregistrée. De plus, si une erreur JS survenait dans les injections DOM (étape 5–6), un removeItem placé en fin de script ne s'exécuterait jamais.
Si removeItem était en étape 7 (après les injections DOM) : une erreur dans l'injection (élément manquant) → exception → le script s'arrête avant removeItem → la donnée reste en sessionStorage → réaffichée au prochain chargement.
Principe fondamental : supprimer/invalider une donnée dès qu'elle est lue avec succès, jamais à la fin d'une longue séquence qui peut échouer.
↓ fallback uniquement si les données réelles sont absentes
4
Données de secoursif (!orderData) { orderData = { … données factices … } }
Placé après la tentative de lecture — on ne sait pas si le fallback est nécessaire avant d'avoir essayé. Ce bloc s'exécute seulement si orderData est encore undefined (sessionStorage vide, JSON corrompu, ou page accédée directement via URL). Il garantit que la page ne reste jamais vide. À partir de cette étape, orderData est garantie non-null.
À partir d'ici, toutes les injections (étapes 5–6) peuvent utiliser orderData sans vérification supplémentaire.
↓ données garanties → on peut afficher (simple avant complexe)
5
Injections simplesgetElementById('order-number').textContent = … (×5 éléments)
Avant la construction HTML complexe — du plus simple au plus complexe. Les injections textContent directes (numéro de commande, email, nom, adresse, date) sont triviales : un élément → une valeur. Les placer avant le forEach suit la convention de lisibilité : voir les cas simples avant les cas complexes. Fonctionnellement, l'ordre entre les injections simples et la boucle est indifférent.
↓ liste des articles — construction HTML par boucle
6
Construction HTMLorderData.items.forEach(item => …) + innerHTML
Plus complexe que les injections simples — génère du HTML dynamique. Cette boucle construit la liste complète des articles commandés avec leurs images, prix et quantités. La placer après les injections simples (étape 5) est une convention : on affiche d'abord les informations de synthèse (numéro, email), puis le détail. En pratique, l'ordre est inversable sans impact fonctionnel.
↓ vider le panier en tout dernier — action irréversible
7
Nettoyage irréversiblePearTechCart.clear() + badge mis à 0
En tout dernier — vider le panier après que tout l'affichage est confirmé sans erreur. Si le panier était vidé à l'étape 3 (juste après la lecture) et qu'une erreur JS survenait dans les injections DOM, l'utilisateur verrait une page de confirmation vide avec son panier déjà perdu. Placer le nettoyage en dernier garantit que l'affichage complet a réussi avant toute action irréversible.
Si PearTechCart.clear() était à l'étape 3 : erreur dans l'injection DOM (ex: getElementById('order-number') → null) → exception → page vide + panier vidé. L'utilisateur perd sa commande et ses articles.
Règle d'or : les actions irréversibles (vider, supprimer) se placent en dernier, après que leur prérequis (l'affichage) est confirmé.
Règle générale de ce fichier

L'ordre de suppression est délibéré : sessionStorage.removeItem() tôt (protège contre F5), cart.clear() tard (protège contre les erreurs JS). Ces deux décisions opposées reflètent la même logique : minimiser les risques.

Concepts et termes techniques utilisés dans ce fichier ¶

Définitions des termes importants rencontrés dans cette documentation.

Document Object Model
Représentation en mémoire de la page HTML sous forme d'arbre. Chaque balise devient un nœud. JavaScript modifie cet arbre via getElementById(), querySelector(), innerHTML. Modifier le DOM = modifier ce que l'utilisateur voit en temps réel.
Immediately Invoked Function Expression
Fonction qui s'exécute immédiatement à sa déclaration : (function() { ... })(). Crée un scope isolé — tout ce qui est déclaré dedans reste privé et ne pollue pas window. Tous les fichiers JS du projet utilisent ce pattern.
DOMContentLoaded
Événement déclenché quand le HTML est parsé et le DOM prêt — avant images et CSS. Moment idéal pour initialiser les modules JS. Distinct de window.load qui attend tous les assets réseau.
sessionStorage
Identique à localStorage mais limité à l'onglet courant. Effacé à la fermeture de l'onglet. Utilisé ici pour transmettre la commande de paiement.js vers confirmation.js sans la persister.
JSON — JavaScript Object Notation
Format texte pour sérialiser des objets JS en chaîne. JSON.parse() = string → object. JSON.stringify() = object → string. localStorage ne stocke que des strings — tout objet doit être sérialisé.

💡
Analogie — URLSearchParams : Une URL avec paramètres est comme une lettre avec une adresse et des post-its collés dessus. URLSearchParams te permet de lire les post-its (?orderId=42&total=99) sans décoller la lettre ou te souvenir de leur position.
💡
Analogie — Nettoyage après commande : Vider le panier après confirmation, c'est comme le serveur qui débarrasse la table après que tu aies mangé. Tu gardes le ticket (numéro de commande affiché) mais les plats (localStorage) disparaissent.
🚨 PIÈGE — Afficher une page de confirmation sans vérifier qu'une commande existe
Si l'utilisateur arrive directement sur /confirmation.html sans paramètre d'URL, la page doit gérer ce cas. Toujours vérifier que les paramètres requis existent (urlParams.get('orderId') peut retourner null) et afficher un message d'erreur ou rediriger.
🚨 PIÈGE — Vider le panier avant de confirmer l'affichage
Le nettoyage de localStorage doit se faire après que les données ont été lues et affichées. Si tu vides d'abord et que le rendu échoue ensuite, l'utilisateur perd ses données sans voir sa confirmation.
PRO TIP
Génère un numéro de commande unique côté client avec Date.now().toString(36).toUpperCase(). Pas besoin de serveur pour un numéro de référence visuellement convaincant.
PRO TIP
Après une commande confirmée, désactive le bouton retour avec history.pushState(null, '', location.href) suivi d'un listener sur popstate. Évite que l'utilisateur revienne sur un panier vide et repasse commande accidentellement.

URLSearchParams

Lit les paramètres de l'URL proprement. Pas de split manuel sur ? et &.

Lecture avant nettoyage

Afficher les données d'abord, vider localStorage ensuite. L'ordre est critique.

// ============================================
// confirmation.js - Page de confirmation de commande
// Lit les données de commande stockées en sessionStorage
// (renseignées par paiement.js) et les affiche dynamiquement
// ============================================

// IIFE pour isoler le code et éviter les conflits
(function() {
    'use strict'; // Mode strict : meilleure sécurité et détection d'erreurs

    // Attend que le DOM soit complètement chargé avant d'exécuter le code
    document.addEventListener('DOMContentLoaded', function() {

        // Variable qui contiendra les données de la commande
        let orderData;

        // Récupère les données de la dernière commande stockées dans sessionStorage
        // sessionStorage est utilisé car les données doivent survivre à un rechargement de page
        // mais être effacées à la fermeture de l'onglet (contrairement à localStorage)
        const stored = sessionStorage.getItem('peartech-last-order');

        if (stored) {
            try {
                // Tente de parser la chaîne JSON en objet JavaScript
                orderData = JSON.parse(stored);
                // Une fois lues, on supprime les données pour éviter de les réafficher plus tard
                // (par exemple si l'utilisateur rafraîchit la page, on ne veut pas garder l'ancienne commande)
                sessionStorage.removeItem('peartech-last-order');
            } catch (e) {
                // En cas d'erreur de parsing (ex: JSON invalide), on log une alerte
                console.warn('Impossible de parser peartech-last-order', e);
            }
        }

        // Si aucune donnée n'a été trouvée (pas de commande précédente ou erreur),
        // on utilise des données factices pour que la page ait un affichage cohérent
        // (utile en développement ou si l'utilisateur arrive directement sur la page)
        if (!orderData) {
            orderData = {
                orderNumber: 'CMD-' + Math.floor(Math.random() * 1000000), // Génère un numéro aléatoire
                email: 'jean.dupont@email.com',
                delivery: {
                    name: 'Jean Dupont',
                    address: '15 rue de la Paix, 75001 Paris, France',
                    mode: 'Livraison standard (3-5 jours)',
                    payment: 'Carte bancaire **** 4242'
                },
                items: [
                    { name: 'Ordinateur Portable Gamer X1', quantity: 1, price: 1200.00 },
                    { name: 'Souris sans fil Pro',          quantity: 1, price: 50.00 },
                    { name: 'Sacoche de transport',         quantity: 1, price: 30.00 }
                ],
                shipping: 0, // Frais de port (gratuit ici)
                total: 1280.00 // Total calculé
            };
        }

        // ── Remplissage des éléments du DOM avec les données de la commande ──

        // Numéro de commande
        document.getElementById('order-number').textContent = orderData.orderNumber;

        // Email du client (texte simple, pas de lien)
        document.getElementById('customer-email').textContent = orderData.email; // CORRECTION : texte simple, pas de lien

        // Informations de livraison
        document.getElementById('delivery-name').textContent   = orderData.delivery.name;
        document.getElementById('delivery-address').textContent = orderData.delivery.address;
        document.getElementById('delivery-mode').textContent   = orderData.delivery.mode;
        document.getElementById('payment-method').textContent  = orderData.delivery.payment;

        // ── Construction de la liste des articles commandés ──
        const itemsContainer = document.getElementById('order-items');
        let itemsHtml = '';
        // Parcourt chaque article et génère le HTML correspondant
        orderData.items.forEach(item => {
            itemsHtml += `
                <div class="order-item">
                    <span class="item-name">${item.quantity}x ${item.name}</span>
                    <span class="item-price">${item.price.toFixed(2).replace('.',',')} €</span>
                </div>
            `;
        });
        // Injecte le HTML dans le conteneur
        itemsContainer.innerHTML = itemsHtml;

        // ── Affichage des frais de port ──
        // Si shipping est 0, on affiche "Gratuit", sinon on formate le montant
        const shippingText = orderData.shipping === 0
            ? 'Gratuit'
            : orderData.shipping.toFixed(2).replace('.',',') + ' €';
        document.getElementById('shipping-cost').textContent = shippingText;

        // ── Affichage du total payé ──
        document.getElementById('total-paid').textContent = orderData.total.toFixed(2).replace('.',',') + ' €';

        // ── Nettoyage du panier après confirmation de commande ──

        // Utilise l'API centralisée PearTechCart si elle existe (pour vider le panier)
        if (window.PearTechCart) {
            window.PearTechCart.clear(); // vide le panier
        } else {
            // Fallback : on remet simplement le compteur à zéro dans localStorage
            localStorage.setItem('peartech-cart-count', '0');
        }

        // Met à jour le badge du panier dans le header
        const badge = document.getElementById('cart-count');
        if (badge) {
            badge.textContent = '0';      // Affiche 0
            badge.style.display = 'none'; // Cache le badge (car zéro)
        }
    });
})();

function load(key) {
  try {
    const raw = localStorage.getItem(key);
    return raw ? JSON.parse(raw) : null;
  } catch { return null; } // corrompu → null
}

  // JSON.stringify : tableau/objet JS → string stockable

function initConfirmation() {
  const params = new URLSearchParams(location.search);
  // location.search = '?orderId=ABC123&total=249.98&items=3'
  // URLSearchParams parse automatiquement tous les paramètres.

  const orderId = params.get('orderId');
  const total   = params.get('total');
  const items   = params.get('items');
  // get retourne la string ou null si le paramètre est absent.

  // ── Guard : paramètres requis absents ─────────────────────────────
  if (!orderId || !total) {
    showError('Commande introuvable. Retournez à la boutique.');
    return; // Stoppe l'initialisation proprement
  }

  // ── Affichage ─────────────────────────────────────────────────────
  renderConfirmation({ orderId: orderId, total: total, items: items });

  // ── Nettoyage APRÈS l'affichage ───────────────────────────────────
  // Ordre critique : afficher d'abord, nettoyer ensuite.
  // Si renderConfirmation plante, on ne perd pas les données localStorage.
  PearTechCart.clearCart();
  // Vide le panier côté localStorage. L'utilisateur a commandé, c'est logique.

  cleanURL();
  // Remplace l'URL ?orderId=… par /confirmation.html (sans paramètres sensibles)
  // dans l'historique. history.replaceState, pas pushState (on ne veut pas
  // que l'utilisateur puisse revenir sur l'URL avec paramètres).
}

function cleanURL() {
  history.replaceState({}, '', location.pathname);
  // replaceState : modifie l'URL courante sans créer une nouvelle entrée d'historique.
  // location.pathname = '/confirmation.html' (sans le ?query).
}

function generateOrderId() {
  return Date.now().toString(36).toUpperCase()
    + '-'

  // ⚠ En production : le vrai ID vient du serveur après validation paiement.

}
