Principe : une classe CSS fait tout le travail visuel ¶

darkmode.js ne change aucune couleur directement. Il se contente d'ajouter ou retirer la classe dark-mode sur le <body>. C'est le CSS qui fait tout le reste grâce aux variables CSS conditionnelles.

01 — Persistance

Mémoriser le choix de l'utilisateur ¶

La persistance du thème repose sur localStorage : le choix de l'utilisateur est sauvegardé entre les sessions. L'initialisation applique le thème sauvegardé avant d'attacher le listener de clic — c'est l'ordre qui prévient le flash visuel.

Lecture et valeur par défaut

javascript
const savedTheme = localStorage.getItem('peartech-theme') || 'dark';
// Si 'peartech-theme' n'existe pas → getItem retourne null
// null || 'dark' → 'dark' (valeur par défaut)
applyTheme(savedTheme); // Applique IMMÉDIATEMENT avant tout listener

localStorage.getItem() — Lire depuis le stockage

localStorage.getItem('clé') retourne la valeur stockée (toujours une chaîne) ou null si la clé n'existe pas. L'opérateur || fournit une valeur par défaut : null || 'dark' → 'dark'. C'est plus court et idiomatique que const t = localStorage.getItem('peartech-theme'); const savedTheme = t ? t : 'dark';.

Pourquoi applyTheme() avant le listener de clic

Si on attachait le listener d'abord et appelait applyTheme() après, il y aurait une fraction de seconde où la page s'affiche avec le thème par défaut du CSS avant d'appliquer le thème sauvegardé — c'est le FOUC (Flash Of Unstyled Content). En appelant applyTheme(savedTheme) dès le DOMContentLoaded, le thème est appliqué avant que l'utilisateur voie quoi que ce soit.

💡
localStorage persiste entre les sessions (fermeture/réouverture du navigateur). sessionStorage ne persiste que pour l'onglet courant. Pour un choix de thème, localStorage est le bon choix : l'utilisateur s'attend à retrouver son thème préféré à la prochaine visite.

02 — La fonction centrale

applyTheme() — 4 choses en même temps ¶

applyTheme() — les 2 appels et ce qu'ils font
applyTheme('dark')
① body.classList.add('dark-mode')
② icon 'light_mode'
③ aria-pressed 'true'
④ CSS --bg: #0a0b0e actif
applyTheme('light')
① body.classList.remove('dark-mode')
② icon 'dark_mode'
③ aria-pressed 'false'
④ CSS --bg: #ffffff actif
Pas de localStorage.setItem() dans applyTheme() — intentionnel. Au chargement, on appelle applyTheme() pour lire sans écrire. L'écriture est faite par l'appelant (le listener de clic) uniquement quand l'utilisateur change d'état.
applyTheme() — les 4 effets simultanés pour chaque état
applyTheme('dark')
① body.classList.add('dark-mode')
② themeIcon.textContent = 'light_mode'
③ aria-pressed = 'true'
④ CSS active --bg: #0a0b0e via body.dark-mode
applyTheme('light')
① body.classList.remove('dark-mode')
② themeIcon.textContent = 'dark_mode'
③ aria-pressed = 'false'
④ CSS active --bg: #ffffff via body:not(.dark-mode)
La fonction est volontairement sans localStorage.setItem() — appelée à la fois au chargement (lecture) et au clic (après écriture). Séparer les deux évite une écriture inutile à chaque chargement.
javascript
function applyTheme(theme) {
if (theme === 'dark') {
document.body.classList.add('dark-mode');          // 1. Active le thème sombre via CSS if (themeIcon) themeIcon.textContent = 'light_mode'; // 2. Icône = soleil (pour passer au clair) themeToggle.setAttribute('aria-pressed', 'true');   // 3. ARIA : bouton "pressé" = dark actif } else {
document.body.classList.remove('dark-mode');         // 1. Désactive le thème sombre if (themeIcon) themeIcon.textContent = 'dark_mode';  // 2. Icône = lune (pour passer au sombre) themeToggle.setAttribute('aria-pressed', 'false');  // 3. ARIA : bouton "non pressé" }
}
// Note : la sauvegarde localStorage est volontairement HORS de cette fonction.
// applyTheme est appelée au chargement (lecture) ET au clic (après écriture).
// Si elle écrivait elle-même, le chargement initial ferait une écriture inutile.
aria-pressed : cet attribut indique à un lecteur d'écran si le bouton est dans un état "activé" ou non — comme un interrupteur. "true" = dark mode ON, "false" = dark mode OFF. Sans ça, les utilisateurs qui naviguent au clavier ou avec un lecteur d'écran ne sauraient pas dans quel état est le thème.

03 — La bascule

Le clic — déterminer le thème opposé ¶

DOM comme source de vérité — pourquoi pas une variable booléenne
Variable JS — risque de désync
let isDark = true;
// Un autre script fait :
body.classList.remove('dark-mode');
// isDark = true, DOM = light
// désynchronisé
DOM — toujours synchronisé
// Lecture directe au clic :
body.classList.contains('dark-mode')
// true si dark, false si light
// Toujours exact, jamais périmé
Principe : lire depuis le DOM au moment où l'info est nécessaire, plutôt que stocker une copie qui peut diverger.
javascript
themeToggle.addEventListener('click', function() {
// Lit l'état ACTUEL depuis le DOM (source de vérité fiable) const currentTheme = document.body.classList.contains('dark-mode') ? 'dark' : 'light';
const newTheme = currentTheme === 'dark' ? 'light' : 'dark'; // Inverse applyTheme(newTheme);                                    // Applique visuellement localStorage.setItem('peartech-theme', newTheme);    // Sauvegarde
});
Le thème courant est lu sur document.body.classList plutôt que depuis une variable JS. C'est la source de vérité
Pourquoi lire body.classList et pas une variable JS ?
Variable JS comme état
let isDark = true; // variable JS
Si le CSS ou un autre script change la classe sans passer par la variable désynchronisation
La variable dit "dark" mais le DOM dit "light" bug silencieux
DOM comme source de vérité
body.classList.contains('dark-mode')
L'état lu est toujours l'état réel visible dans le navigateur
Impossible d'être désynchronisé avec ce que l'utilisateur voit
— même si localStorage et le DOM sont désynchronisés (ex: bug), on part toujours de ce qu'on voit réellement.

Lire l'état actuel depuis le DOM, pas depuis une variable

javascript
themeToggle.addEventListener('click', function() {
    // Lit l'état RÉEL du DOM (pas une variable potentiellement désynchronisée)
    const currentTheme = document.body.classList.contains('dark-mode') ? 'dark' : 'light';
    const newTheme     = currentTheme === 'dark' ? 'light' : 'dark'; // Bascule
    applyTheme(newTheme);
    localStorage.setItem('peartech-theme', newTheme); // Persiste
});

Pourquoi lire classList plutôt qu'une variable ?

Une variable let currentTheme = 'dark' pourrait se désynchroniser si un autre script modifie la classe dark-mode directement (ex: window.PearTech.setTheme('light') dans main.js). En lisant document.body.classList.contains('dark-mode') à chaque clic, on lit l'état réel et actuel du DOM — source de vérité unique.

Le double ternaire — logique de bascule

La ligne const newTheme = currentTheme === 'dark' ? 'light' : 'dark' inverse le thème courant. C'est une bascule binaire : dark → light, light → dark. Plus lisible qu'un if/else sur deux lignes, et moins risqué qu'un !isDark qui suppose un état booléen en mémoire.

applyTheme() vs appliquer manuellement

Plutôt que d'écrire directement document.body.classList.add('dark-mode') dans le listener, on appelle applyTheme(newTheme) qui centralise les 4 effets (classe body, icône, aria-pressed, autres éléments). Si on ajoutait un 5ème effet plus tard (ex: changer les meta-colors), il serait dans applyTheme() — le listener n'a pas besoin de changer.

💡
La sauvegarde localStorage.setItem('peartech-theme', newTheme) est dans le listener de clic, pas dans applyTheme(). Pourquoi ? applyTheme() est aussi appelée à l'initialisation (pour appliquer le thème sauvegardé). Si elle sauvegardait elle-même, l'initialisation réécrirait inutilement le localStorage — opération superflue à chaque chargement de page.
04 — Récapitulatif

Tout le fichier en un schéma ¶

darkmode.js est l'un des modules les plus courts mais illustre parfaitement plusieurs patterns JS fondamentaux : guard clause, valeur par défaut, initialisation avant listener.

Flux d'exécution complet

javascript
// 1. Récupération et guard
const themeToggle = document.getElementById('theme-toggle');
if (!themeToggle) return;

// 2. Éléments secondaires
const themeIcon = themeToggle.querySelector('.material-symbols-outlined');

// 3. Lecture du localStorage avec fallback
const savedTheme = localStorage.getItem('peartech-theme') || 'dark';

// 4. Application AVANT le listener (prévient le FOUC)
applyTheme(savedTheme);

// 5. Listener de clic pour les changements futurs
themeToggle.addEventListener('click', function() {
    const currentTheme = document.body.classList.contains('dark-mode') ? 'dark' : 'light';
    const newTheme = currentTheme === 'dark' ? 'light' : 'dark';
    applyTheme(newTheme);
    localStorage.setItem('peartech-theme', newTheme);
});

// 6. applyTheme() — définie après utilisation (hoisting fonctionne ici)
function applyTheme(theme) { ... }

La fonction applyTheme() — 4 choses simultanément

Quand applyTheme('dark') est appelée : 1. Ajoute la classe dark-mode sur document.body (active tous les styles CSS dark). 2. Change le contenu de l'icône en 'light_mode' (pour indiquer qu'on peut passer au mode clair). 3. Met à jour aria-pressed="true" pour l'accessibilité. 4. Ne fait rien d'autre — tout le visuel est géré par le CSS via la classe dark-mode.

✓
La ligne document.body.classList.contains('dark-mode') dans le listener est plus fiable que mémoriser le thème dans une variable JS : elle lit l'état réel du DOM. Si quelque chose modifiait la classe dark-mode en dehors de darkmode.js (ex: un autre script), le bouton resterait synchronisé avec l'état réel.

05 — Ordre du code

Pourquoi chaque bloc est placé là où il est ¶

darkmode.js illustre la règle initialiser avant d'écouter. L'appel à applyTheme(savedTheme) avant le listener de clic est la décision architecturale la plus importante : elle élimine le flash de couleur visible au chargement.

1
Ressource centraledocument.getElementById('theme-toggle')
Première ligne car tout le module en dépend. themeToggle est utilisé dans le guard clause, pour récupérer l'icône, dans applyTheme(), et dans le listener de clic. Récupérer une seule fois en tête de scope : 1 requête DOM au lieu de 4+. Une variable de scope est aussi plus rapide qu'un getElementById répété (l'accès variable est O(1), la requête DOM implique un parcours du tree).
↓ guard clause immédiatement après — sans délai
2
Protection nullif (!themeToggle) { console.warn(…); return; }
Avant toute utilisation de themeToggle. Toutes les lignes suivantes supposent que themeToggle est non-null : themeToggle.querySelector(), themeToggle.addEventListener(), themeToggle.setAttribute(). Un seul return ici protège l'intégralité du code restant. Sans lui, 4 TypeError en cascade si le bouton est absent.
Sans guard : themeToggle.querySelector('.material-symbols-outlined') → TypeError: Cannot read properties of null (reading 'querySelector') → tout darkmode.js crashe → le thème n'est jamais appliqué → la page reste en mode clair quel que soit le réglage sauvegardé.
↓ l'icône est dans le bouton — garanti d'exister après le guard
3
Élément secondairethemeToggle.querySelector('.material-symbols-outlined')
Après le guard clause — on est certain que themeToggle existe. themeIcon est récupérée avant savedTheme car applyTheme() (appelée à l'étape 5) la modifie pour changer le texte de l'icône ('dark_mode' ↔ 'light_mode'). Si themeIcon était récupérée après l'appel à applyTheme(), l'icône ne serait pas mise à jour lors de l'initialisation — elle resterait figée sur la valeur par défaut du HTML.
Utilisée dans applyTheme() : themeIcon.textContent = theme === 'dark' ? 'light_mode' : 'dark_mode'.
↓ lire la préférence avant de l'appliquer
4
Lecture préférencelocalStorage.getItem('peartech-theme') || 'dark'
Avant applyTheme(savedTheme) — on ne peut pas appliquer ce qu'on n'a pas encore lu. La valeur par défaut 'dark' (opérateur ||) s'applique à la première visite (clé absente) ou si le stockage est indisponible. Cette lecture doit se faire avant l'appel à applyTheme() (étape suivante) pour que le bon thème soit appliqué dès le premier pixel affiché.
Si lu après applyTheme(savedTheme) : on appellerait applyTheme(undefined) → comportement imprévisible selon le code de la fonction (peut appliquer le thème par défaut ou rien du tout).
↓ CRITIQUE : appliquer le thème AVANT le listener de clic → évite le FOUC
5
Initialisation visuelleapplyTheme(savedTheme) — appel initial
L'étape la plus critique de tout le fichier. Le thème doit être appliqué dès le premier pixel affiché. Si cet appel était après le listener de clic (étape 6), la page s'afficherait d'abord avec les couleurs par défaut du CSS (fond clair, texte foncé), puis le thème sombre serait appliqué une fraction de seconde plus tard — visible par l'utilisateur comme un flash blanc (FOUC : Flash Of Unstyled Content). Sur les connexions lentes ou les appareils bas de gamme, ce flash peut durer plusieurs secondes.
Si après le listener : 1. Page charge. 2. HTML s'affiche avec couleurs par défaut (blanc). 3. JS s'exécute, applique dark-mode. 4. Flash visible. Sur mobile 3G : flash de 2 à 5 secondes. L'utilisateur pense que le site est cassé.
Règle critique : l'état visuel initial doit être appliqué avant tout listener d'interaction. Initialiser, puis écouter — jamais l'inverse.
↓ le listener ne gère que les changements futurs
6
InteractivitéthemeToggle.addEventListener('click', …)
Après l'initialisation — ne gère que les changements futurs. Ce listener détecte la classe dark-mode actuelle sur document.body, calcule son opposé, appelle applyTheme() et sauvegarde dans localStorage. Il peut être placé après applyTheme(savedTheme) car il ne s'exécute qu'au clic — jamais au chargement. L'ordre de lecture correspond à l'ordre d'exécution réel.
↓ implémentation des détails en dernier grâce au hoisting
7
Implémentationfunction applyTheme(theme) { … }
Définie en dernier, appelée à l'étape 5 — possible grâce au hoisting. applyTheme est une function declaration — hoistée au sommet du scope par le moteur JS. Elle peut donc être appelée avant d'être physiquement écrite. Ce choix de style (appel avant définition) est le pattern 'top-down' : on voit d'abord ce que fait le code (initialiser, écouter les clics), puis comment ça fonctionne en détail.
Appelle : document.body.classList.add/remove('dark-mode'), themeIcon.textContent, themeToggle.setAttribute('aria-pressed'), localStorage.setItem.
Règle générale de ce fichier

L'ordre des étapes 4 → 5 → 6 est une contrainte UX, pas seulement technique : lire la préférence, l'appliquer immédiatement, puis seulement attacher le listener. Inverser 5 et 6 ne causerait pas d'erreur JS mais créerait un flash visible. function applyTheme() est écrite en dernier — hoisting permet d'y faire référence dès l'étape 5.

Glossaire

Concepts et termes techniques utilisés dans ce fichier ¶

Définitions des termes importants rencontrés dans cette documentation.

Document Object Model
Représentation en mémoire de la page HTML sous forme d'arbre. Chaque balise devient un nœud. JavaScript modifie cet arbre via getElementById(), querySelector(), innerHTML. Modifier le DOM = modifier ce que l'utilisateur voit en temps réel.
Immediately Invoked Function Expression
Fonction qui s'exécute immédiatement à sa déclaration : (function() { ... })(). Crée un scope isolé — tout ce qui est déclaré dedans reste privé et ne pollue pas window. Tous les fichiers JS du projet utilisent ce pattern.
DOMContentLoaded
Événement déclenché quand le HTML est parsé et le DOM prêt — avant images et CSS. Moment idéal pour initialiser les modules JS. Distinct de window.load qui attend tous les assets réseau.
localStorage
Stockage clé-valeur persistant dans le navigateur — survit à la fermeture. Capacité ~5 Mo. Clé/valeur uniquement en string : les objets passent par JSON.stringify() avant stockage et JSON.parse() à la lecture.
classList
Interface de manipulation des classes CSS. add(), remove(), toggle(), contains(). Plus fiable que modifier className comme une string. toggle('active', condition) ajoute ou retire selon un booléen.
ARIA — Accessible Rich Internet Applications
Attributs HTML pour les technologies d'assistance (lecteurs d'écran). aria-label décrit une action, aria-expanded indique si un menu est ouvert, aria-pressed l'état d'un bouton bascule. Sans ARIA, un lecteur d'écran ne sait pas à quoi sert un bouton sans texte visible.

Maîtriser le dark mode sans piège ¶

💡
Analogie — le chef d'orchestre : darkmode.js est comme un chef d'orchestre qui lève la baguette : il ne joue aucun instrument lui-même. Il dit juste "maintenant c'est sombre" en ajoutant dark-mode sur le body. Chaque musicien (variable CSS) sait quoi faire quand le chef lève la baguette.
💡
Analogie — le FOUC : Imaginez un peintre qui peint votre salon en blanc, puis en noir deux secondes plus tard. L'utilisateur voit le flash blanc. C'est exactement le FOUC. Appeler applyTheme() avant le listener de clic, c'est peindre en noir dès le début — pas de flash.
🚨 PIÈGE — Oublier la valeur par défaut dans localStorage
localStorage.getItem('peartech-theme') retourne null si la clé n'existe pas encore (première visite). Sans || 'dark', applyTheme(null) est appelé et le comportement devient imprévisible. Toujours fournir un fallback : getItem('cle') || 'valeur-defaut'.
🚨 PIÈGE — Appeler applyTheme() après addEventListener
Si l'initialisation visuelle est placée après le listener de clic, la page s'affiche d'abord en mode clair, puis bascule en sombre une fraction de seconde plus tard. Sur mobile ou connexion lente, ce flash dure 2 à 5 secondes. L'ordre correct : lire localStorage → appliquer le thème → puis attacher le listener.
PRO TIP
Ajoute prefers-color-scheme: dark en CSS comme valeur initiale avant que JS ne charge. Ainsi même si JS est désactivé ou lent, l'utilisateur voit le bon thème selon son OS.
PRO TIP
Pour tester le FOUC : dans DevTools → onglet Performance → coche "Enable CPU throttling" × 6. Tu verras exactement ce que voit un utilisateur sur mobile bas de gamme.

localStorage

// ============================================
// darkmode.js - Gestion du thème sombre/clair
// Bascule entre dark et light mode, avec
// sauvegarde de la préférence dans localStorage
// ============================================

// IIFE (Immediately Invoked Function Expression) pour isoler le code
// et éviter les conflits avec d'autres variables globales
(function() {
    'use strict'; // Active le mode strict pour une meilleure sécurité et optimisation

    // Attend que le DOM soit complètement chargé avant d'exécuter le code
    // Cela garantit que les éléments HTML (comme le bouton) sont accessibles
    document.addEventListener('DOMContentLoaded', function() {

        // Récupère l'élément bouton qui permet de basculer le thème (par son ID)
        const themeToggle = document.getElementById('theme-toggle');

        // Si le bouton n'existe pas dans la page, on affiche un avertissement
        // et on arrête l'exécution du script (return)
        if (!themeToggle) {
            console.warn('Bouton theme-toggle introuvable');
            return;
        }

        // Récupère l'icône Material Symbols à l'intérieur du bouton
        // pour pouvoir changer son texte (ex: "dark_mode" / "light_mode")
        const themeIcon = themeToggle.querySelector('.material-symbols-outlined');

        // Récupère le thème sauvegardé dans localStorage, ou 'dark' par défaut
        // L'opérateur || signifie : si localStorage.getItem retourne null/undefined, on prend 'dark'
        const savedTheme = localStorage.getItem('peartech-theme') || 'dark';

        // Applique le thème récupéré (ajoute/enlève la classe CSS et met à jour l'icône)
        applyTheme(savedTheme);

        // Ajoute un écouteur d'événement au clic sur le bouton
        themeToggle.addEventListener('click', function() {
            // Détermine le thème actuel en vérifiant si la classe 'dark-mode' est présente sur le body
            const currentTheme = document.body.classList.contains('dark-mode') ? 'dark' : 'light';
            // Calcule le nouveau thème : si c'était dark, on passe à light, et vice-versa
            const newTheme = currentTheme === 'dark' ? 'light' : 'dark';

            // Applique le nouveau thème
            applyTheme(newTheme);

            // Sauvegarde la préférence dans localStorage pour qu'elle persiste entre les visites
            localStorage.setItem('peartech-theme', newTheme);

            // Log facultatif pour le débogage
            console.log('Thème changé:', newTheme);
        });

        /**
         * Applique le thème (dark ou light) en modifiant le DOM
         * @param {string} theme - 'dark' ou 'light'
         */
        function applyTheme(theme) {
            if (theme === 'dark') {
                // Ajoute la classe 'dark-mode' au body pour activer les styles CSS du mode sombre
                document.body.classList.add('dark-mode');
                // Si l'icône existe, on change son texte en 'light_mode' (lune)
                if (themeIcon) themeIcon.textContent = 'light_mode';
                // Met à jour l'attribut aria-pressed pour l'accessibilité (état pressé = true)
                themeToggle.setAttribute('aria-pressed', 'true'); // CORRECTION
            } else {
                // Enlève la classe 'dark-mode' du body pour revenir au mode clair
                document.body.classList.remove('dark-mode');
                // Change l'icône en 'dark_mode' (soleil)
                if (themeIcon) themeIcon.textContent = 'dark_mode';
                // Met à jour aria-pressed à false
                themeToggle.setAttribute('aria-pressed', 'false'); // CORRECTION
            }
        }

        // Log de fin d'initialisation
        console.log('Theme module initialisé - Mode:', savedTheme);
    });

})();

applyTheme() — les 4 actions simultanées

C'est la fonction centrale. Elle est déclarée avec function (pas const) pour bénéficier du hoisting — elle peut être appelée avant sa définition dans le fichier.

javascript
function applyTheme(theme) {

  // ① Classe CSS — déclenche TOUTES les variables CSS du site
  // body.dark-mode → CSS applique : --bg:#0a0b0e, --text:#e4e8f0…
  document.body.classList[theme === 'dark' ? 'add' : 'remove']('dark-mode');

  // ② Icône — l'icône montre l'ACTION possible, pas l'état actuel
  // En mode sombre : affiche 'light_mode' (pour passer au clair)
  // En mode clair  : affiche 'dark_mode'  (pour passer au sombre)
  themeIcon.textContent = theme === 'dark' ? 'light_mode' : 'dark_mode';

  // ③ Accessibilité — indique aux lecteurs d'écran si le mode sombre EST actif
  themeToggle.setAttribute('aria-pressed', theme === 'dark');

  // ④ Persistance — sauvegarde pour la prochaine visite
  // localStorage.setItem(clé, valeur) — toujours des strings
  localStorage.setItem('peartech-theme', theme);
}

④ La fonction applyTheme — 4 effets en un appel

javascript
function applyTheme(theme) {
  // ── 1. Classe CSS sur le body ──────────────────────────────────────
  // classList[méthode] avec bracket notation = conditionnel élégant.
  // Équivalent à : if (theme==='dark') body.add('dark-mode') else body.remove(…)
  document.body.classList[theme === 'dark' ? 'add' : 'remove']('dark-mode');

  // ── 2. Mise à jour de l'icône ──────────────────────────────────────
  // Logique inversée intentionnelle : en mode sombre, l'icône = soleil
  // (pour signifier "cliquez pour passer au clair"). Contre-intuitif
  // mais c'est la convention Material Design.
  if (themeIcon) themeIcon.textContent = theme === 'dark' ? 'light_mode' : 'dark_mode';

  // ── 3. Attribut ARIA ───────────────────────────────────────────────
  // aria-pressed indique si un bouton toggle est "enfoncé".
  // true = dark actif, false = light actif.
  // Les lecteurs d'écran annoncent "bouton thème sombre, activé/désactivé".
  themeToggle.setAttribute('aria-pressed', theme === 'dark');
  // Note : setAttribute convertit le booléen en string "true"/"false" automatiquement.

  // ── 4. Persistance ────────────────────────────────────────────────
  // setItem crée la clé si inexistante, la met à jour si existante.
  // Pas besoin de vérifier si elle existe d'abord.
  localStorage.setItem('peartech-theme', theme);
}

⑤ Le listener de clic — lire le DOM, pas une variable

javascript
themeToggle.addEventListener('click', function() {
  // On lit l'état DEPUIS LE DOM, pas depuis une variable JS.
  // Pourquoi ? Une variable peut se désynchroniser (autre script, extension).
  // Le DOM reflète toujours ce que l'utilisateur voit réellement.
  const isDark = document.body.classList.contains('dark-mode');

  // Calcul du thème opposé. Ternaire plus lisible qu'un if/else ici.
  applyTheme(isDark ? 'light' : 'dark');

  // Pas besoin de localStorage.setItem ici : applyTheme le fait déjà.
});
