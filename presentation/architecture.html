data.js
Expose window.PearTechDB avec ~34 produits, 4 catégories, 7 fonctions de requête (search, getById, getBestSellers…). C'est la base de données du site.
Tout le site affiche du contenu vide. Les grilles, le dropdown, la recherche, la fiche produit — tout retourne undefined ou plante silencieusement.
product_grid.js
Génère les cartes HTML des produits (meilleures ventes + nouveautés) et les injecte dans la grille de l'accueil. Émet categorySelected.
La grille de l'accueil reste vide. Les boutons de filtrage (filtre.js) ne trouvent aucune carte sur laquelle opérer.
filtre.js
Lit les attributs data-category sur les cartes existantes et les affiche/cache selon la catégorie sélectionnée. Écoute categorySelected.
Cliquer sur une catégorie (dropdown ou boutons) n'a aucun effet. Toutes les cartes restent visibles en permanence.
recherche.js
Connecte le champ de recherche du header à PearTechDB.search(). Affiche une modale de résultats. Debounce 300ms.
La barre de recherche ne réagit pas. L'utilisateur ne peut pas trouver de produit par mot-clé depuis le header.
dropdown.js
Génère dynamiquement le menu de navigation avec les 4 catégories issues de PearTechDB. Émet categorySelected au clic.
Le menu de catégories dans la nav est vide. L'utilisateur ne peut pas naviguer par catégorie depuis le header.
product_detail.js
Lit l'ID depuis l'URL, charge le produit, génère tout le HTML de la fiche (tabs, galerie, options, prix, produits similaires, notes).
La page produit reste blanche. Impossible d'afficher une fiche, de voir les specs ou d'ajouter à partir de la fiche.
panier.js
Gère localStorage peartech-cart. Expose window.PearTechCart (add, remove, clear, get). Délégation globale sur tous les boutons "Ajouter".
Aucun bouton "Ajouter au panier" ne fonctionne. Le badge du panier ne se met pas à jour. cart.js, paiement.js et confirmation.js n'ont plus d'API pour accéder au panier.
cart.js
Affiche le récapitulatif du panier, gère les quantités, valide les codes promo, calcule les totaux, transmet la promo via sessionStorage.
La page panier est vide. L'utilisateur ne peut pas modifier son panier, appliquer un code promo, ni passer à la caisse.
catalogue.js
Affiche tous les produits d'une catégorie (via URL param), avec filtres de prix, tri, pagination. Gère son propre ajout au panier.
La page catalogue affiche une grille vide. Impossible de parcourir les produits d'une catégorie.
paiement.js
Affiche le résumé de commande en temps réel, valide le formulaire, génère un numéro de commande, écrit dans sessionStorage, redirige.
La page paiement ne s'initialise pas. L'utilisateur ne peut pas finaliser sa commande — le tunnel d'achat est cassé à son avant-dernière étape.
confirmation.js
Lit sessionStorage['peartech-last-order'], l'affiche dans 9 zones DOM, le supprime immédiatement (usage unique), vide le panier.
La page de confirmation reste vide. Le panier n'est pas vidé. L'utilisateur ne voit pas le récapitulatif de sa commande.
profil.js
Gère l'auth simulée (localStorage), affiche un SPA avec 7 sections (infos, commandes, adresses, favoris…), sessions avec expiration 30min.
La page profil reste vide. L'utilisateur ne peut pas se connecter, créer un compte, ou accéder à son historique.
contact.js
Valide le formulaire de contact (regex email, longueurs), simule l'envoi avec setTimeout, affiche feedback visuel.
Le formulaire ne valide rien et ne répond pas à la soumission. Aucun feedback utilisateur.
favoris.js
Stocke les IDs favoris en localStorage. Délégation globale sur tous les cœurs [data-fav-btn]. Expose window.Favoris. Page dédiée avec grille animée.
Aucun bouton cœur ne réagit. Les favoris ne sont ni sauvegardés ni affichés. La page favoris reste vide.
darkmode.js
Applique immédiatement le thème depuis localStorage au chargement. Bascule dark↔light au clic. Met à jour l'icône et aria-pressed.
Le site s'affiche toujours en mode clair par défaut. Le bouton de thème est silencieux. La préférence n'est jamais sauvegardée.
menu_burger.js
Crée dynamiquement le bouton burger et le panneau mobile (DOM + styles). Swipe, focus trap, fermeture Escape, sync avec la recherche desktop.
Sur mobile, il n'y a aucun menu de navigation. Le site est inutilisable sur petits écrans.
main.js
Écoute les 4 CustomEvents globaux pour le debug. Expose window.PearTech (notify, getTheme, setTheme). Mesure le temps de chargement.
Les logs de debug disparaissent. window.PearTech est undefined — tout code tiers qui l'utiliserait plante. Aucune mesure de perf.
animation.js
IntersectionObserver pour révéler les éléments au scroll. MutationObserver pour les éléments injectés dynamiquement. Ombre header + focus search.
Toutes les animations de scroll disparaissent. Le site fonctionne mais est visuellement statique et moins engageant.
carousel.js
Lance le défilement automatique du carousel (setInterval 5s). Pause au survol. Contrôle via les radio inputs CSS.
Le carousel de l'accueil reste figé sur la première slide. Il ne défile jamais automatiquement.
faq.js
Accordéon animé (height 0→auto), recherche avec debounce 200ms, filtrage par catégorie. Aucune dépendance.
Les questions de la FAQ sont toutes ouvertes ou toutes fermées sans animation. La recherche et les filtres par catégorie ne fonctionnent pas.
03 — Ordre de chargement annoté

Chaque étape : pourquoi ici, ce qu'elle rend disponible, ce qui casse si on la déplace

Les scripts HTML sont exécutés dans l'ordre de leurs balises <script>. Ce n'est pas anecdotique — chaque position dans la liste est le résultat d'une contrainte de dépendance précise.

Timeline d'exécution — page_accueil.html (cas le plus complet)
1
data.js
— OBLIGATOIREMENT EN PREMIER
S'exécute immédiatement, de manière synchrone. Crée et remplit window.PearTechDB avant même que le DOM soit prêt. Les 9 fichiers qui suivent vont tenter d'accéder à cet objet — il doit déjà exister.
✅ window.PearTechDB disponible
✅ 34 produits chargés en mémoire
✅ 7 fonctions de requête accessibles
☠ Si déplacé après : product_grid, dropdown, recherche, catalogue, cart → window.PearTechDB undefined → erreurs silencieuses → grilles vides.

— après data.js, avant tout ce qui utilise le panier
Installe immédiatement (hors DOMContentLoaded) l'écouteur de délégation sur document pour les clics .btn-add-cart. S'il est chargé après product_grid ou catalogue, des clics précoces seraient perdus. Il expose aussi window.PearTechCart que product_detail, cart et paiement vont utiliser.
✅ window.PearTechCart disponible
✅ Délégation panier active dès le départ
☠ Si déplacé après product_grid : un utilisateur rapide pourrait cliquer "Ajouter" avant que la délégation soit installée → clic ignoré silencieusement.

✅ window.Favoris disponible
✅ Délégation cœurs active dès le départ
✅ Badges favoris synchronisés

4
main.js
— installe les écouteurs d'événements globaux
Écoute les CustomEvents émis par filtre.js, panier.js et product_detail.js. Doit être installé avant qu'ils commencent à émettre (DOMContentLoaded). Expose window.PearTech.
✅ window.PearTech disponible
✅ 4 écouteurs CustomEvent installés
☠ Si après filtre.js/product_grid : les premiers événements seraient émis avant que main.js écoute → perdus (CustomEvents ne se mettent pas en queue).

05 — CustomEvents — le bus de communication

Pourquoi les CustomEvents ? Permettre à des modules de se parler sans se connaître

Sans CustomEvents, un module qui veut en informer un autre devrait appeler directement ses fonctions — créant un couplage fort. Avec CustomEvents, chaque module diffuse un signal sur document : les intéressés s'abonnent, les autres ignorent. Les modules peuvent être retirés ou ajoutés sans toucher aux autres.

06 — Carte de couplage

Qui est fortement couplé ? Qui est totalement isolé ?

Le couplage fort signifie qu'un module appelle directement les fonctions d'un autre — les lier ensemble. Le couplage faible passe par des APIs publiques ou des événements. L'isolation totale signifie aucune interaction avec les autres modules.

PearTechCart.clear() après confirmation

8 clés de stockage — qui écrit, qui lit, pourquoi ce type

localStorage — persiste entre les sessions
'peartech-cart'
[{id, name, price, quantity, options, image}]

R/W cart.js — affichage + quantités
W catalogue.js — ajout local
R paiement.js — résumé live
del confirmation.js — vide après commande

W→0 confirmation.js — reset après commande

'peartech-theme'
'dark' | 'light'
R/W darkmode.js — lecture au boot, écriture au clic
W main.js — via PearTech.setTheme()
'peartech-session' · 'peartech-users' · 'peartech-user-{b64}'
session email · [{email,password,name,…}] · profil complet
R/W exclusif profil.js — auth simulée
'peartech-notes-{id}'
texte libre par produit
R/W exclusif product_detail.js
sessionStorage — effacé à la fermeture de l'onglet
'peartech-promo'
{"code":"PROMO10","discount":0.1}
W cart.js — quand un code promo est validé
R paiement.js — applique la réduction
sessionStorage plutôt que localStorage : le code promo ne doit pas persister indéfiniment. Il est lié à une session d'achat précise.
'peartech-last-order'
{orderNumber, email, delivery{name,address,mode,payment}, items[], shipping, promoCode, promoDiscount, total}
W paiement.js — après validateForm() + generateOrderNumber()
R puis supprimé confirmation.js — usage UNIQUE
sessionStorage : le récapitulatif de commande ne doit s'afficher qu'une fois. Un rafraîchissement de la page ne doit pas re-afficher une ancienne commande. sessionStorage.removeItem() garantit l'usage unique.
🤔 Pourquoi localStorage et pas sessionStorage pour le panier ?
Un utilisateur peut ajouter des produits le lundi, fermer l'onglet, revenir le jeudi et retrouver son panier. sessionStorage disparaîtrait à la fermeture — le panier serait perdu. localStorage persiste jusqu'à vidage explicite → meilleure UX pour l'achat multi-session.
🤔 Pourquoi des IDs et pas des objets complets pour les favoris ?
Si on stockait {id:7, name:"iPhone 15", price:1099, image:"..."}, le prix pourrait être obsolète à la prochaine visite. En stockant seulement [7] et en relisant PearTechDB.getProductById(7) à chaque affichage, les données sont toujours fraîches.

10 — Flux par scénario

Ce qui se passe concrètement — étape par étape

3 scénarios en détail — séquences annotées
Tunnel d'achat — du clic "Ajouter" à la page de confirmation
①
panier.js
Clic .btn-add-cart intercepté par délégation → lit data-product-id → doublon ? qty++ : push() → saveCart() → LS peartech-cart
②
cart.js
Lit LS peartech-cart → renderCart() → quantités ± → code promo → calculateTotals() → clic Commander → SS peartech-promo → redirect page_paiement.html
③
paiement.js
Lit LS peartech-cart + SS peartech-promo → totaux live → validateForm() → generateOrderNumber() → SS peartech-last-order → PearTechCart.clear() → redirect page_confirmation.html
④
confirmation.js
Lit SS peartech-last-order → removeItem() immédiat (usage unique) → injecte numéro, articles, adresse dans 9 zones DOM → badge panier → 0
LS peartech-cart : écrit par panier.js, lu par cart.js + paiement.js, supprimé par confirmation.js
SS peartech-promo : cart.js → paiement.js (1 passage de page)
SS peartech-last-order : paiement.js → confirmation.js (usage unique)
️ Cycle de vie d'un favori — ajout, badge, page dédiée
①
favoris.js
Clic [data-fav-btn] → e.stopPropagation() → toggle(id) → indexOf : déjà favori ? splice() : push() → save() → LS peartech-favoris = [ids]
②
favoris.js
updateHeartBtn() → SVG fill rouge ↔ none · updateAllBadges() → tous les #fav-count · showToast() → div créé → visible 2.5s → fade 300ms → remove()
③
favoris.js
Sur page_favoris.html uniquement : initFavorisPage() → polling PearTechDB (50×100ms) → getProductsByIds(ids) → buildCard(p) × N → grille animée · retrait : fade + scale 300ms → re-render
Changement de thème — chargement + bascule
①
darkmode.js
Chargement : LS.getItem('peartech-theme') || 'dark' → applyTheme() → body.classList.add('dark-mode') + icône 🌙/☀️ + aria-pressed
②
darkmode.js
Clic bouton : lit body.classList.contains('dark-mode') (DOM = source de vérité, pas une variable) → newTheme inversé → applyTheme(newTheme) + LS.setItem('peartech-theme', newTheme)
③
main.js
window.PearTech.setTheme(theme) — alternative console-dev uniquement, sans mise à jour icône ni aria → darkmode.js reste la source canonique
Tunnel d'achat complet — du clic "Ajouter" à la confirmation
panier.js
→
Délégation sur document intercepte le clic sur .btn-add-cart. Lit data-product-id, .product-name, .product-price depuis la carte DOM. Ajoute à l'objet cart. saveCart() → localStorage['peartech-cart'].
cart.js
→
Lit localStorage['peartech-cart']. Affiche chaque article. Utilisateur saisit code promo → validPromoCodes → calculateTotals(). Clic "Commander" → sessionStorage['peartech-promo'] → location = 'page_paiement.html'.
paiement.js
→
Lit cart + sessionStorage['peartech-promo']. renderSummary() en temps réel à chaque changement d'option. Clic "Confirmer" → validateForm() → generateOrderNumber() → assemble orderData → sessionStorage['peartech-last-order'] → PearTechCart.clear() → redirect confirmation.
confirmation.js
→
Lit sessionStorage['peartech-last-order']. Supprime immédiatement la clé (usage unique). Injecte orderNumber, email, adresse, liste articles dans 9 zones DOM. Vide le badge panier.
️ Cycle de vie d'un favori
favoris.js
→
Délégation document : e.target.closest('[data-fav-btn]'). e.stopPropagation() évite la navigation vers la fiche. toggle(id) → indexOf → push ou splice → save() → localStorage['peartech-favoris'].
favoris.js
→
updateHeartBtn(btn, added) → SVG remplacé (fill rouge ↔ none). updateAllBadges() → tous les #fav-count, .fav-badge. showToast() → div créé, visible 2.5s, fade 300ms, remove.
favoris.js
→
Sur page_favoris.html : initFavorisPage() → polling PearTechDB → render() → getAll() → PearTechDB.getProductById() → buildCard() → grille. Retirer un favori depuis la grille → removeCard() → fade + scale animation 300ms → render().
Changement de thème
darkmode.js
→
Au chargement : localStorage.getItem('peartech-theme') || 'dark' → applyTheme() → body.classList.add('dark-mode') + icône + aria-pressed.
darkmode.js
→
Clic bouton : lit body.classList.contains('dark-mode') (source de vérité DOM, pas une variable). Calcule newTheme. applyTheme(newTheme) + localStorage.setItem('peartech-theme', newTheme).
main.js
→
window.PearTech.setTheme(theme) fait la même chose mais sans mettre à jour l'icône ni aria-pressed → accessible depuis la console dev uniquement.
