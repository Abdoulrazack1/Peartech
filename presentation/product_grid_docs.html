Remplissage dynamique de la grille — IIFE et polling

La page d'accueil contient une grille vide #products-grid. Ce fichier JavaScript la remplit automatiquement avec les fiches produits en allant chercher les données dans la base de données PearTechDB.

🏗️
Analogie : Imaginez un mur d'affichage vide dans un magasin. Ce fichier est l'employé qui arrive chaque matin avec une pile de fiches produits et qui les accroche une à une sur le mur, en suivant toujours le même protocole : photo, prix, badge promotion, stock, étoiles.

Vocabulaire essentiel — lisez ça en premier

Ces termes reviennent partout dans le fichier. Si vous les connaissez, le reste devient logique.

📚 Glossaire de base
Variable
Une boîte avec un nom pour stocker une valeur. const prix = 999 crée une boîte nommée "prix" contenant 999. const = boîte scellée (valeur ne changera plus). let = boîte ouverte (valeur peut changer).
Fonction
Un groupe d'instructions avec un nom. On l'écrit une fois, on l'appelle autant de fois qu'on veut. Comme une recette nommée : "pour faire une crêpe, fais X, Y, Z".
Tableau (Array)
Une liste ordonnée de valeurs entre crochets : ["iPhone", "Samsung", "Pixel"]. On accède à un élément par son numéro (qui commence à 0) : tableau[0] = "iPhone".
Objet (Object)
Un ensemble de paires clé/valeur entre accolades : { nom: "iPhone", prix: 999 }. On accède à une valeur par sa clé : produit.nom = "iPhone".
DOM
La représentation de la page HTML en mémoire que JavaScript peut lire et modifier. Changer le DOM = changer ce qu'on voit à l'écran en temps réel, sans recharger la page.
innerHTML
Le contenu HTML à l'intérieur d'un élément. Écrire zone.innerHTML = "<p>Bonjour</p>" remplace instantanément tout le contenu de la zone par un paragraphe "Bonjour".
Événement
Une action qui se produit : un clic, une frappe clavier, le chargement d'une image, la fin du chargement de la page. On peut "écouter" ces événements et déclencher une fonction quand ils arrivent.
window
L'objet racine du navigateur. Tout ce qu'on y met est accessible depuis n'importe quel script de la page. C'est la zone partagée entre tous les fichiers JavaScript.
console.log/warn
Affiche un message dans l'onglet "Console" des outils développeurs (F12). Invisible pour l'utilisateur normal, très utile pour déboguer.
01 — La coquille protectrice

L'IIFE : une bulle isolée

La toute première et toute dernière ligne du fichier forment une structure spéciale qui entoure tout le reste.

🏠
Analogie : C'est comme un appartement dans un immeuble. Ce qui se passe chez vous (vos meubles, vos variables) ne déborde pas chez le voisin. Deux appartements peuvent avoir une variable prix sans que l'un écrase l'autre. Sans IIFE, ce serait comme vivre dans une maison sans murs séparant les pièces.
🔤
IIFE = Immediately Invoked Function Expression = une fonction qui s'exécute immédiatement à sa définition.

Syntaxe : (function(){ /* code */ })()
— Les premières parenthèses définissent la fonction.
— Les dernières parenthèses () la déclenchent instantanément.
— Résultat : le code s'exécute, les variables restent enfermées à l'intérieur.
🔤
'use strict' est une déclaration placée en tête de fichier. Elle active le "mode strict" de JavaScript qui :
— Interdit d'utiliser une variable sans l'avoir déclarée avec const ou let
— Interdit certaines syntaxes ambiguës
— Transforme des erreurs silencieuses en erreurs visibles

Sans lui, JavaScript laisserait passer des erreurs discrètement. Avec lui, les erreurs sont signalées immédiatement dans la console.
javascript
(function () {           // ← ouverture de la bulle isolée
    'use strict';        // ← mode strict : zéro tolérance pour les erreurs

    // Tout le code du fichier est ici, à l'abri
    const PLACEHOLDER = '/asset/image/no-image.png';
    // ... fonctions, événements ...

})();                    // ← fermeture + déclenchement immédiat

Pourquoi DOMContentLoaded ?

À l'intérieur de l'IIFE, tout le code principal est dans un écouteur d'événement DOMContentLoaded. Voici pourquoi c'est obligatoire :

⚠️
Le problème du timing : Le navigateur lit le fichier HTML de haut en bas. Si JavaScript est chargé avant d'avoir lu la balise <div id="products-grid">, il ne trouvera pas cet élément — il n'existe pas encore en mémoire. Le code planterait silencieusement.

La solution : DOMContentLoaded est l'événement qui se déclenche quand le navigateur a fini de lire tout le HTML. En l'attendant, on est certain que tous les éléments de la page existent déjà.

02 — Constantes et images

Le système de fallback d'images

Chaque produit devrait avoir une photo. Mais que se passe-t-il si la photo manque ? Ou si elle est là mais le serveur est en panne et elle ne se charge pas ? Le code prévoit 3 niveaux de secours.

3 niveaux de secours — cascade de l'image
1
vraie photo
product.images[0] — ex: "iphone15.jpg"
↓ si null ou erreur de chargement (onerror)
2
placeholder
PLACEHOLDER — SVG générique intégré en base64
↓ si PLACEHOLDER lui-même échoue
3
img.src = ''
vide — le navigateur affiche son icône image cassée
img.onerror = () => { img.onerror = null; img.src = PLACEHOLDER; } — le second onerror = null évite une boucle infinie si PLACEHOLDER échoue aussi.
🔤
const = constante = une variable dont la valeur ne peut plus changer après sa définition. On les écrit en MAJUSCULES par convention pour les distinguer des variables ordinaires. const PLACEHOLDER = '...' crée une boîte scellée nommée PLACEHOLDER.
Les 3 niveaux de secours pour les images
Niveau 1 — La vraie photo du produit : product.images[0]
↓
si la photo n'existe pas dans la base de données, ou si le serveur échoue à la charger
Niveau 2 — Photo générique de la catégorie : FALLBACKS["cat_apple"] → URL Unsplash
↓
si cette URL Unsplash est aussi introuvable
Niveau 3 — Image "pas de photo" locale : PLACEHOLDER → /asset/image/no-image.png
↑
si une image commence à se charger mais échoue en cours de route → onerror remonte ici
onerror HTML — se déclenche si l'image plante pendant le chargement
🔤
FALLBACKS est un objet JavaScript qui fait correspondre une catégorie à une URL. C'est un dictionnaire :

{ "cat_apple": "https://unsplash.com/...?w=400&q=80", "cat_android": "https://..." }

On l'utilise comme ça : FALLBACKS["cat_apple"] → on obtient l'URL Unsplash pour Apple.

?w=400&q=80 à la fin de l'URL n'est pas du JS : c'est une instruction envoyée au serveur Unsplash. Cela lui dit "renvoie l'image en 400px de large et à 80% de qualité" → image 5 fois plus légère qu'en haute résolution.

Comment fonctionne onerror ?

HTML généré dans le JS
<img
  src="https://exemple.com/photo-iphone.jpg"   ← essaie la vraie photo
  data-fallback="https://unsplash.com/apple..."   ← plan B stocké en attribut caché
  onerror="this.onerror=null; this.src=this.dataset.fallback||'/asset/image/no-image.png'"
>

Explication ligne par ligne de onerror :
  this           → "je" = l'élément img lui-même
  this.onerror = null  → j'éteins mon propre gestionnaire d'erreur
                         POURQUOI ? Si le plan B aussi échoue, onerror se
                         déclencherait à nouveau → boucle infinie. null l'arrête.
  this.src = ...       → je change ma source vers le plan B
  || '/asset/...'      → si même le plan B est vide, utilise le placeholder local
💡
Pourquoi stocker le plan B dans data-fallback ? Parce que onerror n'a pas accès aux variables JavaScript comme FALLBACKS. Mettre le plan B dans un attribut HTML (data-fallback) permet à l'attribut onerror de le lire directement via this.dataset.fallback.
03 — La fonction principale

renderProducts( productsToRender )

C'est la fonction la plus importante du fichier. Elle reçoit une liste de produits et transforme chaque produit en une carte HTML visible sur la page.

🏭
Analogie : Imaginez une usine à cartes postales. On lui envoie une liste d'adresses (les produits). Pour chaque adresse, elle suit toujours 7 étapes dans l'ordre : calculer le prix, choisir le badge, vérifier le stock, écrire les caractéristiques, dessiner les étoiles, vérifier les favoris, imprimer la carte. Toutes les cartes finies sont empilées, agrafées ensemble, puis collées d'un coup sur le mur (injection dans le DOM).
Flux complet de renderProducts
Entrée : tableau de produits — ex: [{id:1, name:"iPhone 15"...}, {id:2...}]
↓
Cherche l'élément #products-grid dans la page avec getElementById
↓
Vérifications : la grille existe ? Le tableau est non-vide ?
NON → problème
↓
Affiche un message d'erreur dans la grille, puis return (arrêt immédiat)
OUI → on fabrique les cartes
↓
.map() : traite chaque produit un par un
↓
7 calculs pour chaque produit :
① Prix et réduction ② Badge ③ Stock ④ Specs ⑤ Étoiles ⑥ Favoris ⑦ Balise <article>
↓
tous les produits traités
.join('') — colle tous les blocs HTML en une seule grande chaîne
↓
grid.innerHTML = résultat — injecte tout d'un coup dans la page
↓
forEach sur les boutons panier — attache les événements clic
🔤
.map() est une méthode de tableau qui transforme chaque élément. tableau.map(fn) crée un nouveau tableau où chaque élément a été transformé par la fonction fn.

Exemple simple : [1, 2, 3].map(n => n * 2) → [2, 4, 6]

Dans notre code : products.map(product => "...HTML...") transforme chaque objet produit en une chaîne de texte HTML. Le tableau original products n'est pas modifié.
🔤
.join('') colle les éléments d'un tableau en une seule chaîne. ["a","b","c"].join('') → "abc". ["a","b","c"].join('-') → "a-b-c". Ici on utilise '' (rien entre les éléments) pour coller les blocs HTML côte à côte.

Pourquoi injecter d'un seul coup avec innerHTML ?

💡
On aurait pu créer chaque carte et l'ajouter au DOM une par une dans une boucle. Mais chaque modification du DOM force le navigateur à recalculer toute la mise en page. 8 modifications = 8 recalculs. En construisant tout le HTML en mémoire d'abord, puis en l'injectant une seule fois, on ne déclenche qu'un seul recalcul. La page s'affiche instantanément.

Calcul du prix et de la réduction — ligne par ligne

javascript
// On récupère le prix actuel du produit
const price = product.basePrice;    // ex: 750

// On cherche un ancien prix valide.
// product.oldPrice && product.oldPrice > price
// ↑ "oldPrice existe" ET "oldPrice est plus grand que le prix actuel"
// Sans la 2e condition, un oldPrice de 100 sur un produit à 999€
// afficherait une "réduction" de -899% → n'importe quoi
const oldPrice = (product.oldPrice && product.oldPrice > price)
    ? product.oldPrice   // si les deux conditions sont vraies : on garde l'ancien prix
    : null;              // sinon : pas d'ancien prix valide

// Calcul du pourcentage de réduction.
// Formule mathématique : (1 - prix_actuel / ancien_prix) × 100
// Exemple : 1 - (750/999) = 1 - 0.750 = 0.250 → × 100 = 25%
// Math.round() arrondit au nombre entier le plus proche
// sans ça on obtiendrait "24.9249%" — pas très élégant
const discount = oldPrice
    ? Math.round((1 - price / oldPrice) * 100)
    : 0;   // pas d'ancien prix → 0% de réduction

// Formater le prix en style français : 1200 → "1 200,00 €"
// toLocaleString('fr-FR') gère automatiquement :
//   - l'espace comme séparateur de milliers (1 200 au lieu de 1,200)
//   - la virgule comme séparateur décimal (1 200,00 au lieu de 1,200.00)
// minimumFractionDigits: 2 force toujours 2 chiffres après la virgule
// "50" devient "50,00" pas "50"
const priceFmt = price.toLocaleString('fr-FR', { minimumFractionDigits: 2 }) + ' €';

05 — Notation par étoiles

De 4.7 à ½

La note est un nombre décimal comme 4.7. Il faut le décomposer en étoiles pleines + demi-étoile éventuelle + étoiles vides pour toujours obtenir 5 au total.

Décomposition de la note 4.7 en étoiles HTML
Math.floor(4.7)
= 4 → étoiles pleines : ★★★★
4.7 % 1 >= 0.5
= 0.7 ≥ 0.5 → 1 demi-étoile : ½
5 - 4 - 1
= 0 étoiles vides : (aucune)
Résultat : ★★★★½ (toujours 5 symboles au total)
Pour rating = 3.2 : 3 pleines + 0 demi (0.2 < 0.5) + 2 vides. Pour rating = 5.0 : 5 pleines + 0 demi + 0 vides.
🔤
Math.floor() = arrondir vers le bas, toujours. Math.floor(4.9) = 4. Math.floor(4.1) = 4. Jamais 5, même à 4.99.

L'opérateur % (modulo) = le reste d'une division. 7 % 2 = 1 (7 = 3×2 + 1). 4.7 % 1 = 0.7 (la partie après la virgule). C'est la façon de dire "donne-moi uniquement la partie décimale".

'.repeat(n)' = répète une chaîne n fois. '★'.repeat(4) = '★★★★'.
Calcul étape par étape pour rating = 4.7
Départ : rating = 4.7
↓
Math.floor(4.7) = on garde uniquement la partie entière
fullStars = 4 → on affichera 4 étoiles pleines : ★★★★
↓
4.7 % 1 = 0.7 → est-ce que 0.7 est ≥ à 0.5 ?
hasHalf = true → on affichera une demi-étoile : ½
↓
5 (total) - 4 (pleines) - 1 (demi vaut 1) = 0
emptyStars = 0 → aucune étoile vide nécessaire
↓
'★'.repeat(4) + '½' + '☆'.repeat(0)
Résultat affiché : ★★★★½

Qu'est-ce que localStorage ?

🔤
localStorage est un mini-espace de stockage dans le navigateur. Il persiste même quand on ferme l'onglet ou le navigateur. C'est comme une clé USB intégrée à Chrome/Firefox.

— localStorage.setItem('cle', 'valeur') → enregistre
— localStorage.getItem('cle') → récupère (ou null si absent)

Contrainte : il ne stocke que du texte. Pas d'objets, pas de tableaux. C'est pour ça qu'on utilise JSON.stringify(objet) pour convertir en texte avant de stocker, et JSON.parse(texte) pour reconvertir en objet au moment de récupérer.

⚠️Le texte stocké dans localStorage est exactement la même chaîne que ce que JSON.stringify a produit. Ce n'est pas "magiquement" un objet — c'est du texte. Sans JSON.parse() au moment de le relire, on obtiendrait une chaîne de caractères brute, et cart.find(...) échouerait immédiatement.
— chaque ligne expliquée
javascript
// Étape 1 : Lire le panier existant dans localStorage
// localStorage.getItem('peartech-cart') retourne :
//   - null si rien n'a encore été stocké (premier achat)
//   - une chaîne JSON comme '[{"id":1,"name":"iPhone","quantity":1}]'
// JSON.parse() convertit cette chaîne en vrai tableau JavaScript
// || [] = "si le résultat est null/undefined, utilise un tableau vide"
const cart = JSON.parse(localStorage.getItem('peartech-cart')) || [];

// Étape 2 : Vérifier si ce produit est déjà dans le panier
// .find() parcourt le tableau et retourne le premier élément
// pour lequel la condition (i.id === productId) est vraie
// Retourne undefined si aucun élément ne correspond
const existing = cart.find(i => i.id === productId);

if (existing) {
    existing.quantity++;    // Produit déjà là → +1 à la quantité (1 → 2 → 3...)
} else {
    cart.push(cartItem);    // Nouveau produit → on l'ajoute à la fin du tableau
}

// Étape 3 : Sauvegarder le panier mis à jour
// JSON.stringify() reconvertit le tableau en texte pour localStorage
localStorage.setItem('peartech-cart', JSON.stringify(cart));

// Étape 4 : Calculer le total d'articles pour le badge du panier
// .reduce() parcourt le tableau en accumulant une valeur
// Démarre à 0, ajoute la quantité de chaque article
// Exemple : [{qty:2}, {qty:3}, {qty:1}] → 0 + 2 + 3 + 1 = 6
const total = cart.reduce((acc, i) => acc + i.quantity, 0);

// Étape 5 : Mettre à jour le badge visuel "3" sur l'icône panier
const badge = document.getElementById('cart-count');
if (badge) badge.textContent = total || '';
// total || '' = si total = 0, affiche rien (on cache le badge vide)

07 — Initialisation

init( attempts ) — Attendre sans bloquer

product_grid.js se charge vite, mais data.js (qui contient PearTechDB) peut prendre du temps à cause du réseau. On ne peut pas juste "attendre" — le navigateur se figerait. La solution : vérifier régulièrement, jusqu'à 50 fois.

📞
Analogie : C'est comme appeler un collègue qui n'est peut-être pas encore arrivé au bureau. Plutôt que de rester scotché au téléphone (ce qui bloquerait tout), vous laissez un message qui dit "rappelle dans 100ms". S'il n'est toujours pas là, vous relaissez le même message. Après 50 tentatives sans réponse (5 secondes), vous laissez un message d'erreur sur sa boîte vocale.
🔤
Récursion = une fonction qui s'appelle elle-même. Ici init() appelle init(attempts + 1). Sans le compteur attempts et la condition < 50, ça tournerait indéfiniment — comme deux miroirs se faisant face.

setTimeout(fn, 100) = "dans 100 millisecondes, exécute fn". 100 ms = 0.1 seconde. 50 × 100 ms = 5 000 ms = 5 secondes. Si PearTechDB n'est toujours pas là en 5s, il y a un vrai problème réseau.

Sélection intelligente des 8 produits à afficher

javascript
// On compose une liste en deux étapes :
// 1. Tous les bestsellers (isBestSeller = true)
// 2. Les nouveautés qui ne sont PAS déjà dans les bestsellers
//    (pour éviter les doublons si un produit est à la fois new et bestseller)
// Ensuite .slice(0, 8) prend au maximum les 8 premiers

const featured = [
    ...db.products.filter(p => p.isBestSeller),          // bestsellers
    ...db.products.filter(p => p.isNew && !p.isBestSeller) // nouveautés non-bestseller
].slice(0, 8);

// Si aucun produit n'est ni bestseller ni nouveau,
// on affiche simplement les 8 premiers de la base
// featured.length ? featured : ... = "si featured est non-vide, utilise-le, sinon utilise..."
renderProducts(featured.length ? featured : db.products.slice(0, 8));
🔤
Spread operator (...) : les trois points "déversent" un tableau dans un autre. [...A, ...B] crée un nouveau tableau avec les éléments de A, puis ceux de B. Ni A ni B ne sont modifiés.

Exemple : [...[1,2], ...[3,4]] → [1, 2, 3, 4]

.filter(condition) : crée un nouveau tableau ne gardant que les éléments pour lesquels la condition est vraie.

.slice(0, 8) : crée un nouveau tableau avec les éléments de l'index 0 à 7 (8 non inclus). Les index commencent à 0 en JavaScript.
