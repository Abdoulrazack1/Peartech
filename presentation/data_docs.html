L'architecture en couches de data.js

data.js est organisé en 3 couches imbriquées : les données brutes (categories[] et products[]), les fonctions de transformation (helpers qui filtrent, cherchent, trient), et l'exposition publique (window.PearTechDB et dispatchEvent('PearTechDBReady')).

javascript
// Couche 1 — Données brutes
const categories = [{ id, name, icon, slug }];
const products   = [{ id, name, specs, price, images, category, tags }];

// Couche 2 — Helpers internes
function getBestSellers() { return products.filter(p => p.bestSeller); }
function search(query) { /* cherche dans name + specs + tags */ }
function getProductById(id) { return products.find(p => p.id === id); }

// Couche 3 — Exposition publique
window.PearTechDB = { products, categories, search, getProductById, ... };
document.dispatchEvent(new CustomEvent('PearTechDBReady'));

Pourquoi tout est dans une IIFE

Sans IIFE, const products = [...] serait une variable globale accessible et modifiable depuis n'importe quel script. Un products.push(fakeProduct) depuis la console corromprait silencieusement le catalogue. L'IIFE encapsule les données brutes — seul window.PearTechDB est exposé, et uniquement les méthodes souhaitées.

💡
La propriété products est exposée sur window.PearTechDB mais c'est une référence au tableau original (pas une copie). Les scripts qui lisent PearTechDB.products peuvent donc le modifier via .push() ou .splice(). En production rigoureuse, on exposerait [...products] (copie) ou on utiliserait Object.freeze().

L'IIFE — pourquoi encapsuler dans une fonction anonyme

Comme tous les scripts du projet, data.js s'enveloppe dans une IIFE (Immediately Invoked Function Expression). Cela protège les variables internes de l'espace global.

Sans IIFE vs avec IIFE — protection du scope
Sans IIFE — dangereux
const categories = […];
const products = […];

// window.categories = [… // accessible !]
// window.products = [… // accessible !]
// N'importe quel autre script peut
// modifier ou écraser les tableaux
Avec IIFE — sécurisé
(function() {
  const categories = […];
  const products = […];
  // privés — invisibles depuis window
  window.PearTechDB = { …helpers };
  // seul cet objet est accessible
})();
Les variables categories et products restent privées. L'extérieur ne peut accéder aux données qu'à travers les méthodes contrôlées de window.PearTechDB — comme un objet avec des getters.

Flux d'exécution de data.js — Pas de DOMContentLoaded

data.js s'exécute entièrement de façon synchrone dès son chargement. Pas de DOMContentLoaded car il n'a pas besoin du DOM. La séquence : déclaration des tableaux → définition des helpers → window.PearTechDB = {...} → dispatchEvent('PearTechDBReady'). Tout se passe en une fraction de milliseconde.

✓
data.js doit être le premier script chargé dans le HTML (avant tous les autres). Si un autre script s'exécute avant data.js et cherche window.PearTechDB, il trouvera undefined. L'ordre des balises <script> dans le HTML est critique.

search(query) — 3 champs simultanément ¶

search(query) — 3 champs vérifiés en même temps, toLowerCase partout
query = "iphone pro" q = "iphone pro" (toLowerCase)
pour chaque produit dans products[]
name.toLowerCase()
"iphone 15 pro max" contient "iphone pro" inclus
category.toLowerCase()
"smartphones" ne contient pas "iphone pro" on continue
tags.join(' ')
"iphone apple 5g pro camera" contient "iphone pro"
Logique OR : suffit qu'UN des 3 champs match pour inclure le produit · toLowerCase des deux côtés = recherche insensible à la casse
javascript
search: function(query) {
const q = query.toLowerCase();  // normalise : "MAC"  "mac" return products.filter(p => p.name.toLowerCase().includes(q) ||        // ex: "macbook air m2" inclut "mac"  p.description.toLowerCase().includes(q) || // si le name ne matche pas, tente ici p.tags.some(tag => tag.includes(q))        // ["macbook","m2"]  "macbook" inclut "mac"  );
}
search("mac") — analyse sur chaque produit

PearTechDBReady — Signal de disponibilité asynchrone

javascript
// Pattern d'attente robuste utilisé dans dropdown.js et product_grid.js
function initWhenReady(fn) {
    if (window.PearTechDB) {
        fn(); // data.js déjà chargé de façon synchrone
    } else {
        document.addEventListener('PearTechDBReady', fn, { once: true });
        // { once: true } : le listener se supprime automatiquement après 1 déclenchement
    }
}

L'option { once: true } dans addEventListener est un raccourci pour removeEventListener après le premier déclenchement. Sans elle, le listener resterait actif en mémoire — si PearTechDBReady était émis plusieurs fois (par exemple en cas de rechargement partiel), fn() serait appelée plusieurs fois.

💡
Le signal PearTechDBReady est émis une seule fois à la fin de data.js. Les modules qui l'écoutent avec { once: true } sont garantis de n'initialiser qu'une fois. C'est une forme légère d'injection de dépendance : data.js annonce sa disponibilité plutôt que d'être importé directement.

Pourquoi chaque bloc est placé là où il est ¶

data.js suit la logique données sources → fonctions de transformation → exposition. Pas de DOM, pas d'événements. Chaque bloc dépend de ceux qui le précèdent via la closure du scope.

1
Catégoriesconst categories = [ … ]
En tout premier — les produits y font référence via categoryId. Chaque produit a un categoryId qui pointe vers une catégorie. Les fonctions helpers (getProductsByCategory, getCategoryBySlug) cherchent dans categories. L'ordre reflète aussi la hiérarchie métier : une catégorie contient des produits — définir d'abord le conteneur, puis le contenu.
Référencée par : chaque produit (via categoryId), getCategoryBySlug(), getProductsByCategory(), getBrand().
↓ les produits référencent les catégories
2
Produitsconst products = [ … ]
Après categories — chaque produit possède un categoryId défini dans categories. Placer les produits après les catégories signale cette dépendance à quiconque lit le fichier. Les helpers (getBestSellers, getNewProducts, search…) filtrent ce tableau — il doit donc être déclaré avant eux.
Référencé par tous les helpers. Référence categories via categoryId.
↓ helpers après les données qu'ils transforment
3
Helpersconst helpers = { getBestSellers, getNewProducts, search, getProductById, … }
Après products et categories — ils les utilisent par closure. Les helpers capturent products et categories via la closure du scope englobant. Si helpers était déclaré avant products, les fonctions comme getProductById refermeraient sur une variable déclarée mais non assignée (undefined) — les appels retourneraient toujours undefined.
Si avant const products : helpers.getBestSellers() → products.filter(p => p.isBestSeller) → undefined.filter() → TypeError. Les helpers doivent voir les données finales, pas les variables vides.
↓ exposer seulement quand tout est prêt
4
Exposition globalewindow.PearTechDB = { categories, products, …helpers }
Après tout le reste — assemble les pièces définies avant. On ne peut exposer que ce qui est déjà défini. Placer ce bloc avant helpers ferait que window.PearTechDB.search serait undefined. C'est le 'sceau final' : tout est prêt, on publie l'API. L'événement PearTechDBReady est émis juste après pour signaler aux modules en attente (comme dropdown.js).
Assemble : categories, products, tous les helpers. Émet ensuite : document.dispatchEvent(new CustomEvent('PearTechDBReady')).

5
Alias directwindow.products = products
Après l'exposition principale — raccourci de compatibilité secondaire. Certains scripts anciens (favoris.js, product_grid.js) accèdent directement à window.products sans passer par window.PearTechDB. Cet alias maintient la compatibilité sans changer ces scripts. Il est placé après window.PearTechDB car c'est un raccourci, pas la voie principale d'accès.
↓ compatibilité Node.js — environnement secondaire
6
Compatibilité Node.jsif (typeof module !== 'undefined') { module.exports = … }
En tout dernier — compatibilité pour les tests en Node.js. Ce bloc permet d'importer data.js depuis un script Node.js (tests unitaires, scripts de génération). Règle générale de ce fichier

L'ordre dans data.js est une contrainte de closure : les helpers capturent products et categories — ils doivent les voir assignés. window.PearTechDB est exposé en dernier pour que toutes ses propriétés soient non-undefined. L'émission de PearTechDBReady permet aux modules chargés après (dropdown.js) de savoir quand démarrer.

🚨 PIÈGE — Muter le tableau de produits directement

🚨 PIÈGE — Recherche naïve sensible à la casse et aux accents

Le module pattern IIFE avec API publique

javascript
// IIFE qui retourne un objet public
// Les données et fonctions internes restent privées dans la closure
window.PearTechDB = (function() {

  // ── DONNÉES PRIVÉES (inaccessibles depuis l'extérieur) ──
  const products = [
    { id: 1, name: 'MacBook Air M2', category: 'laptops', price: 1299,
      specs: { processor: 'M2', ram: '8 Go', storage: '256 Go' } },
    // ... 49 autres produits
  ];

  // ── INDEX POUR RECHERCHE O(1) ──────────────────────────
  // Construit UNE FOIS au démarrage. Chaque getById() sera instantané.
  const indexById = {};
  products.forEach(function(p) {
    indexById[p.id] = p; // { 1: produit1, 2: produit2, … }
  });

  // ── FONCTION PRIVÉE — normalisation des chaînes ────────
  function normalize(str) {
    return str
      .toLowerCase()             // 'MacBook' → 'macbook'
      .normalize('NFD')           // décompose les accents
      .replace(/[̀-ͯ]/g, ''); // supprime les diacritiques → 'été' = 'ete'
  }

  // ── API PUBLIQUE ────────────────────────────────────────
  return {

    products: products, // lecture seule (ne PAS modifier)

    getById: function(id) {
      return indexById[id]; // O(1) — accès direct par clé objet
    },

    search: function(query) {
      const q = normalize(query);
      return products.filter(function(p) {
        return normalize(p.name).includes(q)
            || normalize(p.category).includes(q);
      });
    },

    getByCategory: function(cat) {
      return products.filter(p => p.category === cat);
    }
  };

})(); // exécution immédiate
